# ADR-0015: Interaction-Matrix Diagram Required for Volume Approval

**Status:** Final
**Date:** 2026-07-13
**Supersedes:** —
**Related RFCs:** —
**Related Volumes:** 1 (Foundation, Ch. 3 Dependency Table), Constitution Principle 5 (Event Driven)

---

## Context

Volume 1, Chapter 3 defines a dependency table that specifies which Volumes depend on which other Volumes. This table answers the question "which Volume must be Approved before this Volume can be Approved?" — a dependency ordering question. However, it does NOT answer a fundamentally different question: "at runtime, which module is allowed to call which other module, and through what mechanism?"

Constitution Principle 5 (Event Driven) mandates that cross-module communication occurs through the Event Bus, not through direct synchronous calls. This principle exists to decouple modules, enable independent testing, and prevent the spaghetti-dependency patterns that plague monolithic architectures. But the current dependency table conflates two distinct relationships:

1. **Architectural dependency** (Volume A depends on Volume B's interfaces): This is a design-time relationship about API contracts and approval ordering. It is what Volume 1, Ch. 3 currently documents.

2. **Runtime call permission** (Module X may synchronously call Module Y, or must communicate via Event Bus): This is a runtime relationship about which function calls are legal and through what transport. It is NOT currently documented anywhere.

The conflation is dangerous. For example, the dependency table shows that Core Runtime (Volume 2) depends on Provider Platform (Volume 4). Does this mean Core Runtime can import and call Provider Platform functions directly? Or must Core Runtime request LLM invocations through the Event Bus? The answer has significant implications for testability (direct calls require mocking; Event Bus calls require event simulation), deployability (direct calls create tight coupling), and security (direct calls create a larger attack surface).

Without a documented runtime interaction pattern, each Volume's author will make an implicit, unrecorded choice. Some will use direct calls (because they're simpler), others will use the Event Bus (because Principle 5 says to). Over time, the codebase will accumulate an inconsistent mix of synchronous and asynchronous cross-module calls, and no document will exist to identify which is which. This is exactly the kind of silent architectural violation that Constitution Principle 1 prohibits.

## Decision

An interaction-matrix diagram becomes a required artifact for Volume Approval. The diagram is maintained as a single Mermaid table in Volume 1, Chapter 3 (alongside the existing dependency table), and is updated as part of each Volume's Approval PR.

### Diagram Content

The interaction-matrix must show, for every pair of modules that interact at runtime:

1. **Source module:** The module initiating the interaction.
2. **Target module:** The module being called or receiving the event.
3. **Call mechanism:** One of:
   - `SYNC` — Direct synchronous function call / import (permitted only for same-Volume module interactions or explicitly approved cross-Volume exceptions).
   - `ASYNC` — Asynchronous communication via the Event Bus (the default for all cross-Volume interactions per Constitution Principle 5).
   - `FORK` — Fire-and-forget event (publish with no expectation of reply).
4. **Event Bus topics** (if ASYNC or FORK): The specific topic(s) published to and/or subscribed from.
5. **Interface:** The specific function name or event schema that mediates the interaction.

### Format

The matrix is rendered as a Mermaid `block-beta` diagram (or, if Mermaid support is insufficient, a markdown table with equivalent information). The source of truth is the markdown; the rendered diagram is a convenience.

Example entry:
```
| Core Runtime → Provider Platform | ASYNC | topic: llm.invocation.request | LLMInvocationRequest event schema |
```

### Approval Gate Extension

This extends the Constitution's "Definition of Approved": a Volume cannot be marked Approved without a current interaction-matrix row (or rows) covering each new interface it introduces or consumes. Specifically:

- If Volume N introduces a new interface that another module calls, the interaction-matrix must include an entry for that call.
- If Volume N begins calling another module's interface (either synchronously or asynchronously), the interaction-matrix must include an entry for that call.
- The Volume's Approval PR must include a diff of the interaction-matrix showing the added/modified rows.

### Exceptions

Synchronous cross-Volume calls are permitted only when explicitly justified and approved. The justification must be documented in the interaction-matrix entry. Accepted justifications include:
- **Same-Volume internal calls:** Modules within the same Volume may call each other synchronously without restriction (the Volume boundary IS the decoupling boundary).
- **Performance-critical hot paths:** Where Event Bus latency is measurably unacceptable (requires benchmark data in the justification).
- **Bootstrap/initialization:** Where the Event Bus is not yet available (e.g., system startup).

Any other synchronous cross-Volume call requires an RFC documenting the justification and the planned migration path to asynchronous communication.

## Alternatives Considered

### Option A: No Matrix — Rely on Dependency Table Alone
**Description:** Continue using only the Volume 1, Ch. 3 dependency table to govern cross-module relationships. Trust that Constitution Principle 5 and code review will prevent unauthorized synchronous cross-Volume calls.
**Pros:**
- No additional documentation overhead.
- The dependency table already exists and is familiar to all contributors.
- Code review can catch violations on a case-by-case basis.
- Avoids the risk of the matrix becoming stale and misleading (a common problem with architecture documentation).
**Cons:**
- Already shown to be ambiguous — the Assessment identified this conflation as a contributor to the 3.0/5.0 score.
- Code review is a late-stage, probabilistic enforcement mechanism. A synchronous import added in a PR is easy to miss, especially in a large codebase with many modules.
- No single document exists that a new contributor can consult to understand "how do modules talk to each other?"
- Violations of Principle 5 accumulate silently because there is no baseline to compare against.
- The dependency table cannot express Event Bus topics, which are essential for understanding the asynchronous communication pattern.
**Why not chosen:** The dependency table answers a different question than the interaction matrix. Relying on it for runtime call governance is using the wrong tool for the job. The ambiguity has already caused confusion (per the Assessment), and will only worsen as the codebase grows.

### Option B: Automated Call-Graph Extraction from Code
**Description:** Instead of a manually maintained diagram, use static analysis tooling (e.g., TypeScript's compiler API, Madge, or a custom AST parser) to extract the actual call graph from the codebase and compare it against the permitted interaction patterns.
**Pros:**
- Always reflects the actual code — no risk of documentation drift.
- Can be run in CI as an automated check (e.g., "no synchronous imports across Volume boundaries except where explicitly allowlisted").
- Scales better than manual maintenance as the codebase grows.
- Provides objective, verifiable evidence of compliance with Principle 5.
**Cons:**
- The codebase does not exist yet. The agentx project is in the specification phase (14 Volumes being authored). A call-graph tool has nothing to analyze.
- Spec-first approach: the architecture is defined in the handbook before code is written. The interaction matrix is a SPECIFICATION, not a reflection of code. It defines what SHOULD be, not what IS.
- Even once code exists, call-graph extraction for TypeScript cross-module calls requires module-boundary conventions (e.g., directory-per-Volume, barrel exports) that don't yet exist.
- Automated tools produce false positives and false negatives. A function call that looks cross-Volume in the AST might be an internal utility; a cross-Volume call via a shared interface might be missed.
- The tooling itself becomes a maintenance burden.
**Why not chosen:** Automated call-graph extraction is a valuable FUTURE enforcement mechanism, but it cannot replace the specification. The interaction matrix is needed NOW, during the specification phase, to define the target architecture. Once code exists, the matrix can be validated against the code using automated tools (this is a natural Volume 14 / CI integration). But the matrix must exist first.

### Option C: Manual Interaction-Matrix in Volume 1, Maintained Per-Volume (Chosen)
**Description:** A single Mermaid-based table in Volume 1, Chapter 3, updated incrementally as part of each Volume's Approval PR. The matrix specifies source module, target module, call mechanism (SYNC/ASYNC/FORK), Event Bus topics, and interface name.
**Pros:**
- Directly addresses the ambiguity by documenting the runtime call pattern for every cross-module interaction.
- Maintained incrementally — each Volume's Approval PR updates only its own rows, keeping the change set small and reviewable.
- Single source of truth in Volume 1, consistent with the handbook's "architecture lives in the handbook" philosophy.
- Enables future automated validation (Volume 14) by providing the specification to compare code against.
- Mermaid rendering provides a visual representation alongside the machine-readable markdown table.
- Extends the "Definition of Approved" quality gate in a way that is checkable in a PR (did the interaction-matrix diff cover all new interfaces?).
**Cons:**
- Manual maintenance is subject to drift. If a developer adds a synchronous cross-Volume call without updating the matrix, the matrix becomes inaccurate.
- Increases the Approval PR checklist (one more artifact to review).
- The Mermaid format may not scale well to a large number of interactions (50+ entries). A markdown table with filtering/searching may be more practical for larger codebases.
- Requires discipline to keep updated. If the matrix is treated as a one-time artifact, it loses value quickly.
- The "source of truth" claim is only as strong as the enforcement mechanism. Without CI validation (future Volume 14 work), the matrix is aspirational, not enforceable.

## Consequences

- **Positive:** Every cross-module runtime interaction is explicitly documented with its mechanism (synchronous vs. asynchronous) and Event Bus topics.
- **Positive:** Constitution Principle 5 (Event Driven) becomes verifiable, not just aspirational. A reviewer can check the matrix to confirm that all cross-Volume calls are asynchronous.
- **Positive:** New contributors have a single document to consult for understanding the platform's communication architecture.
- **Positive:** The matrix provides the specification that future automated call-graph validation (Volume 14) will compare code against.
- **Negative:** Volume Approval PRs gain an additional checklist item (interaction-matrix update). Estimated +30-60 minutes per Approval PR.
- **Negative:** The matrix may become stale if not rigorously maintained. The "Definition of Approved" gate mitigates this but does not eliminate the risk (post-Approval code changes could introduce undocumented interactions).
- **Negative:** The Mermaid format may need to be replaced with a markdown table or a dedicated tool if the number of interactions exceeds ~50 entries (projected for v1.0).
- **Impact on existing contracts:** Volume 1, Chapter 3 gains a new section (Interaction Matrix) alongside the existing Dependency Table. The Constitution's "Definition of Approved" gains a new criterion: interaction-matrix coverage for all new interfaces.
- **When to revisit:** After the first 5 Volumes have been Approved and their interaction-matrix entries have been written. Assess whether the Mermaid format is scaling adequately or needs to be replaced. Also revisit when Volume 14 (Testing & QA) is authored — the interaction matrix should be formalized as a contract-test input specification at that point.
