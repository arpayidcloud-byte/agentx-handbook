# ADR-0011: Lightweight Threat Model Required Before Tool SDK Code Generation

**Status:** Final
**Date:** 2026-07-13
**Supersedes:** —
**Related RFCs:** RFC-0021

---

## Context

Constitution Principle 1 (Architecture First) states that no architectural decision may be made by silent omission — if the corpus is deferring something, that deferral must be recorded on the record. The Tool SDK (Volume 7) is the single most security-sensitive component in the agentx platform: it executes shell commands, writes files, and reads filesystem contents on the operator's behalf, all driven by agent instructions that originate from an external LLM provider. This constitutes a direct attack surface.

Despite this, no threat model exists anywhere in the 16-Volume handbook, the RFC corpus, or the ADR corpus. The Architecture Assessment (which scored the project 3.0/5.0) identified four distinct trust boundaries — operator-trusted, agent-controlled, tool-executed, and external network — but no document systematically enumerates the threats that cross these boundaries.

By omitting a threat model, the project is implicitly making the decision "no threat model is needed before v0.1." This is a silent violation of Principle 1. Even if the decision were deliberate (i.e., "we accept the risk of shipping without a threat model"), it must be recorded. This ADR records the opposite decision: a threat model IS required, but at a scope appropriate for v0.1.

RFC-0021 (Tool SDK Security Model) is the natural home for this work, and its authoring has not yet begun. This ADR gates RFC-0021's progression to "Implementation Ready" on the existence of at least a lightweight threat model covering the four identified trust boundaries.

## Decision

A lightweight STRIDE-per-trust-boundary threat model IS required before Tool SDK code generation begins (RFC-0021). The threat model need not be comprehensive — it must cover the four trust boundaries identified in the Architecture Assessment and enumerate the highest-likelihood threats for each. Specifically:

1. **Operator-trusted boundary:** Threats where an agent tricks the operator into approving a malicious action (e.g., prompt injection causing a misleading approval prompt).
2. **Agent-controlled boundary:** Threats where an agent exceeds its authorized tool access (e.g., a coding agent invoking security-agent-only tools).
3. **Tool-executed boundary:** Threats where tool execution escapes intended scope (e.g., shell command injection, path traversal in fs.write).
4. **External network boundary:** Threats where LLM provider responses are tampered with or where tool output leaks to unintended recipients.

The threat model document is filed under `04-Specifications/security/threat-model-v0.1.md` and is reviewed as part of RFC-0021's Specification Review quality gate (Engineering Execution Program §8). It does not need to cover every STRIDE category for every boundary — only those threats rated "High" or "Medium" likelihood with "High" or "Medium" impact.

A full, formal threat model covering all STRIDE categories, all data flows, and all assets is deferred to the post-v0.1 security hardening phase.

## Alternatives Considered

### Option A: Affirm the Deferral (No Threat Model Before v0.1)
**Description:** Explicitly record that no threat model is required before v0.1 CLI ships, accepting the risk of undiscovered vulnerabilities in the Tool SDK's attack surface.
**Pros:**
- No additional documentation work before code generation can begin.
- Faster path to v0.1 release.
- Threat modeling can be done retroactively during post-launch security review.
**Cons:**
- The Tool SDK IS the primary attack surface. Shipping it without any threat analysis means security mitigations are implemented ad-hoc rather than systematically.
- Violates the spirit of Constitution Principle 1 even if recorded — the decision to skip threat modeling for the most security-critical component undermines credibility.
- A retroactive threat model is significantly less useful because the code already embodies implicit security decisions that are harder to change.
- External security auditors (if engaged for enterprise customers per Volume 10) will flag the absence of a pre-implementation threat model.
**Why not chosen:** The Tool SDK executes arbitrary shell commands and filesystem writes on the operator's machine. Deferring threat modeling for this component specifically is an unacceptable risk, even for a v0.1 CLI. The cost of a lightweight model is small; the cost of a shipped vulnerability in this component is potentially catastrophic (remote code execution on operator machines).

### Option B: Full STRIDE Analysis Before Any Code Generation
**Description:** Require a complete STRIDE analysis covering all six threat categories (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege) across all trust boundaries, all data flows, all assets, and all threat actors before any code is written for any component.
**Pros:**
- Most thorough approach; leaves no gap in security analysis.
- Produces a reusable document that serves the project through v1.0 and beyond.
- Establishes the strongest possible security culture from day one.
**Cons:**
- Estimated 2-3 weeks of dedicated security analysis work before v0.1 code generation can begin — a significant schedule impact for a component that doesn't yet exist.
- Many STRIDE categories (e.g., Denial of Service) are not relevant to a single-operator CLI tool and would produce noise that dilutes focus on the real threats.
- The analysis would cover Volumes and components that don't exist yet (e.g., Volume 10 enterprise features), making much of it speculative.
- Creates a precedent that could slow future feature development if applied uniformly.
**Why not chosen:** Overkill for the v0.1 CLI scope. The platform is a single-operator local tool at this stage — a full STRIDE analysis of non-existent enterprise features is speculative work. The value of threat modeling is proportional to the concreteness of the system being analyzed.

### Option C: Lightweight Threat Model for Tool SDK Only, Full Analysis Deferred (Chosen)
**Description:** Require a focused threat model covering only the four trust boundaries identified in the Assessment, enumerating only the highest-likelihood threats per boundary. Scope is limited to the Tool SDK (Volume 7) and its immediate consumers. Full STRIDE analysis is deferred to post-v0.1.
**Pros:**
- Directly targets the highest-risk component with proportionate effort.
- Can be completed in 2-3 days by the RFC-0021 author with security review.
- Establishes the practice of threat modeling without creating an undue schedule burden.
- The four-trust-boundary framework from the Assessment provides clear structure without requiring the full STRIDE taxonomy.
- Results in concrete mitigations that are actionable for v0.1 implementation.
- Sets a precedent: future security-sensitive components get their own scoped threat model.
**Cons:**
- Not a complete security analysis — some threats may be missed.
- Requires judgment calls on what constitutes "highest-likelihood," which introduces subjectivity.
- The lightweight model may need significant expansion before enterprise deployment (Volume 10), meaning some rework.
- Creates a two-tier documentation standard (lightweight vs. full) that must be clearly communicated to avoid confusion.

## Consequences

- **Positive:** The Tool SDK will be designed with explicit awareness of its threat landscape, leading to better security decisions from the start (e.g., input sanitization, approval-gate design, sandbox boundaries).
- **Positive:** RFC-0021 gains a concrete pre-requisite that prevents it from reaching Implementation Ready without security analysis, enforcing Constitution Principle 1.
- **Positive:** The threat model document becomes a reference for contract-test security scenarios (Volume 14) and the Security Agent's (Volume 3) rule set.
- **Negative:** RFC-0021 authoring is blocked for an estimated 2-3 days while the threat model is produced and reviewed.
- **Negative:** The lightweight model may give a false sense of completeness; reviewers must understand it is explicitly scoped and not exhaustive.
- **Impact on existing contracts:** No existing Volume contracts change. RFC-0021's progression criteria gain one new quality gate.
- **When to revisit:** When Volume 10 (Enterprise & Multi-Tenant) enters Specification Review — a full STRIDE analysis will be required at that point, and the v0.1 lightweight model will be used as a foundation.
