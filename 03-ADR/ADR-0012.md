# ADR-0012: Secrets Storage Mechanism for v0.1 (Environment Variables with Abstraction Layer)

**Status:** Final
**Date:** 2026-07-13
**Supersedes:** —
**Related RFCs:** —
**Related Volumes:** 4 (Provider Platform), 16 (Secrets & Key Management, scoped but not authored)

---

## Context

The Provider Platform (Volume 4) defines a `CredentialResolver` interface responsible for obtaining provider API keys at runtime — every LLM invocation flows through this resolver. However, no decision exists in the corpus on WHERE credentials are actually stored or how they reach the resolver in v0.1. Volume 16 (Secrets & Key Management) is listed in the 14-Volume map but has not been authored; its scope includes enterprise-grade secret management (rotation, vault integration, multi-tenant isolation), which is clearly post-v0.1 work.

In the interim, the `CredentialResolver` needs a concrete backing implementation for v0.1. The most natural choice — environment variables — is so obvious that it risks being implemented silently without a recorded decision, violating Constitution Principle 1. More importantly, choosing env vars without an abstraction layer would bake the storage mechanism into Volume 4's public contract, making the future migration to Vault or AWS Secrets Manager (Volume 16) a breaking change rather than a configuration swap.

This ADR also addresses the logging risk: environment variables are trivially accessible to any code in the process. Without explicit safeguards, a debug log statement or error handler could inadvertently leak API keys into log files, CI output, or crash reports. This is a non-negotiable security requirement regardless of which storage mechanism is chosen.

## Decision

v0.1 uses environment variables as the credential storage mechanism, accessed through a dedicated `CredentialResolverConfig` abstraction interface. The specific decisions are:

1. **Storage mechanism:** Environment variables following the naming convention `AGENTX_SECRET_{PROVIDER_ID}_{KEY_TYPE}`. For example, `AGENTX_SECRET_OPENAI_API_KEY` or `AGENTX_SECRET_ANTHROPIC_API_KEY`. The `CredentialResolver` receives a `CredentialResolverConfig` implementation that reads from `process.env`.

2. **Abstraction layer:** A `CredentialResolverConfig` interface is introduced in Volume 4's public API:
   ```
   interface CredentialResolverConfig {
     resolve(providerId: string, keyType: string): Promise<string | undefined>;
   }
   ```
   The v0.1 implementation (`EnvVarCredentialResolverConfig`) reads from environment variables. Future implementations (`VaultCredentialResolverConfig`, `AWSSecretsManagerCredentialResolverConfig`) will be drop-in replacements with no changes to `CredentialResolver` or any downstream consumer.

3. **Logging safeguard:** A lint rule (implemented as an ESLint custom rule in the repository) flags any string literal, template literal, or log statement whose value could match the `AGENTX_SECRET_` prefix pattern. The rule is enforced in CI. Additionally, the `CredentialResolverConfig` implementation redacts resolved values in any toString() or serialization output.

4. **Migration path:** Volume 16 will define the `VaultCredentialResolverConfig` and `AWSSecretsManagerCredentialResolverConfig` implementations. The switchover is a configuration change (which `CredentialResolverConfig` implementation is instantiated at boot), not a code change. This is documented in Volume 4's "Future Extensions" section.

5. **Missing credential behavior:** If a required credential is not found, `resolve()` returns `undefined` and the caller (Provider Platform) produces a clear, user-facing error message: `"Provider '{providerId}' requires credential '{keyType}'. Set AGENTX_SECRET_{PROVIDER_ID}_{KEY_TYPE} in your environment."` The error message does NOT include the actual (missing) value, only the expected key name.

## Alternatives Considered

### Option A: Hardcoded Secrets in Configuration Files
**Description:** Store API keys directly in a configuration file (e.g., `~/.agentx/config.yaml` or `agentx.config.json`) alongside other settings.
**Pros:**
- Simple to implement — no abstraction needed beyond file I/O.
- Easy for users to manage multiple providers and keys in one place.
- Survives shell restarts (unlike env vars, which must be re-exported).
**Cons:**
- Security anti-pattern: config files are typically committed to version control accidentally. Even with `.gitignore`, the risk of credential exposure is high.
- File permissions on config files are often too permissive by default (0644 on many systems).
- No native support for secret rotation without file rewriting.
- Violates the industry-standard twelve-factor app principle (factor III: store config in the environment).
**Why not chosen:** This is a well-documented security anti-pattern. The agentx project explicitly positions itself as a security-conscious platform; using hardcoded secrets in files would undermine that positioning from day one. The convenience argument does not outweigh the risk.

### Option B: Direct Vault Integration from Day One
**Description:** Require HashiCorp Vault (or equivalent) as a runtime dependency for v0.1, with the `CredentialResolverConfig` implementation reading directly from Vault's KV store.
**Pros:**
- Enterprise-grade secret management from the start.
- Supports secret rotation, access auditing, and revocation natively.
- No credentials ever touch the local filesystem or environment.
- Positions the platform for enterprise deployment (Volume 10) immediately.
**Cons:**
- Adds a significant infrastructure dependency to what is intended to be a zero-infrastructure v0.1 CLI tool. Users would need to install, configure, and run Vault just to use the platform.
- Violates the v0.1 scope principle of "works on a single developer's laptop with no external services beyond the LLM provider."
- Increases onboarding friction dramatically — the primary v0.1 adoption path is a single developer trying the tool locally.
- Vault configuration (tokens, policies, KV paths) adds its own secret management bootstrap problem.
**Why not chosen:** The v0.1 target user is a single developer on a laptop. Requiring Vault for local development is disproportionate to the threat model at this stage. The abstraction layer (Option C) preserves the ability to add Vault later with zero code changes to consumers.

### Option C: Environment Variables with Abstraction Layer (Chosen)
**Description:** Use environment variables as the v0.1 storage mechanism, but access them exclusively through a `CredentialResolverConfig` interface that can be swapped for Vault/AWS SM implementations in future versions.
**Pros:**
- Zero infrastructure dependency — works immediately on any developer's machine.
- Follows twelve-factor app methodology (config in environment).
- The abstraction layer ensures Volume 4's public contract never depends on the storage mechanism.
- Lint-based logging safeguard prevents the most common credential leakage vector.
- Clear migration path to enterprise secret management (Volume 16).
- Convention-based naming (`AGENTX_SECRET_*`) makes it obvious which environment variables are secrets vs. regular config.
**Cons:**
- Environment variables are visible to any process running as the same user (via `/proc/<pid>/environ` on Linux). This is an inherent limitation of the mechanism.
- Environment variables do not survive shell restarts; users must re-export them or use `.env` files (which reintroduces the file-based risk, though the `.env` pattern is well-understood and supported by tooling like `dotenv`).
- No native rotation support — changing a key requires restarting the process (acceptable for v0.1 CLI usage).
- The lint rule for logging safeguard is a CI-time check, not a runtime guarantee. A developer could bypass it locally.

### Option D: Encrypted File-Based Storage
**Description:** Store credentials in a local encrypted file (e.g., `~/.agentx/secrets.enc`) using a key derived from a user passphrase or machine-specific key.
**Pros:**
- Credentials persist across shell restarts without requiring `.env` files.
- Encryption at rest provides better security than plaintext env vars or config files.
- No external infrastructure dependency.
**Cons:**
- Adds significant implementation complexity (encryption library, key derivation, file locking, passphrase management).
- Key management becomes its own bootstrap problem: where is the encryption key stored?
- Introduces a user-facing passphrase prompt on every session start (or a cached key, which undermines the encryption benefit).
- For a single-operator v0.1 CLI, the threat model does not justify this complexity — the primary threat is remote code execution via LLM, not local file access by other users on the same machine.
- Would still need the same abstraction layer for future Vault migration.
**Why not chosen:** The complexity-to-benefit ratio is poor for v0.1. The primary attack vector for v0.1 is not "another user reads my secrets file" but rather "an agent leaks a secret via logging or network exfiltration." The logging safeguard (lint rule) addresses the actual high-likelihood threat. Encrypted file storage can be reconsidered for Volume 16 if the threat model warrants it.

## Consequences

- **Positive:** v0.1 has a clear, documented credential storage mechanism with a concrete migration path to enterprise-grade solutions.
- **Positive:** Volume 4's `CredentialResolver` is decoupled from storage implementation from day one, preventing a future breaking change.
- **Positive:** The `AGENTX_SECRET_*` naming convention and lint rule provide a systematic defense against the most common credential leakage vector (logging).
- **Negative:** Users must set environment variables before using the platform. Documentation must include clear setup instructions for each supported provider.
- **Negative:** The lint rule for logging safeguard is not a runtime guarantee; a compromised dependency could bypass it.
- **Impact on existing contracts:** Volume 4 gains a new public interface (`CredentialResolverConfig`). Volume 16's scope is clarified: it provides alternative implementations of this interface, not a replacement for it.
- **When to revisit:** When Volume 16 (Secrets & Key Management) enters Specification Review. The migration from env vars to Vault/AWS SM must be validated as a non-breaking configuration change. Additionally, revisit if a v0.1 security incident involves credential leakage — the encrypted file option (Option D) would be reassessed at that point.
