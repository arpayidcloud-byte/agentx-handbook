# ADR-0014: Audit Log Append-Only Enforcement via Database Triggers

**Status:** Final
**Date:** 2026-07-13
**Supersedes:** —
**Related RFCs:** RFC-0024
**Related Volumes:** 6 (Audit & Compliance), 10 (Enterprise & Multi-Tenant)

---

## Context

Volume 6 (Audit & Compliance) defines the audit log's schema — the fields, their types, and the events that produce entries. However, it does not define the immutability guarantee that makes the audit log trustworthy. An audit log that can be silently modified after the fact is not an audit log; it is a suggestion log. The entire compliance and security posture of the platform rests on the assumption that audit entries, once written, cannot be altered or deleted by any component — including a compromised agent, a buggy service, or a malicious insider with application-level database access.

RFC-0024 proposes an append-only enforcement mechanism for the audit log. This ADR records the architectural choice of HOW that enforcement is implemented in v0.1, given the constraint that the v0.1 platform is a single-operator CLI tool backed by a local Postgres instance.

The key tension is between enforcement strength and implementation complexity. A perfectly immutable audit log requires infrastructure-level guarantees (e.g., S3 WORM, Amazon QLDB, or a blockchain-based log). These are disproportionate for v0.1 but essential for the enterprise deployment target (Volume 10). This ADR selects a v0.1 mechanism that provides strong guarantees within the database layer and documents the migration path to infrastructure-level immutability.

## Decision

Audit log immutability is enforced at the PostgreSQL level via database triggers and role-based permissions. The specific implementation:

### Database Triggers
A `BEFORE UPDATE` trigger and a `BEFORE DELETE` trigger are created on the `audit_log` table. Both triggers raise an exception (SQLSTATE `42501`, insufficient privilege), preventing any row modification or deletion. The triggers are created in a dedicated migration file and are included in every schema deployment.

Trigger pseudocode:
```sql
CREATE OR REPLACE FUNCTION audit_log_immutable()
RETURNS TRIGGER AS $$
BEGIN
    RAISE EXCEPTION 'audit_log table is append-only: % operation not permitted', TG_OP;
    RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER audit_log_no_update
    BEFORE UPDATE ON audit_log
    FOR EACH ROW EXECUTE FUNCTION audit_log_immutable();

CREATE TRIGGER audit_log_no_delete
    BEFORE DELETE ON audit_log
    FOR EACH ROW EXECUTE FUNCTION audit_log_immutable();
```

### Role-Based Permissions
The application's database role (`agentx_app`) is granted only `INSERT` and `SELECT` on the `audit_log` table. `UPDATE` and `DELETE` are explicitly revoked:

```sql
REVOKE UPDATE, DELETE ON audit_log FROM agentx_app;
```

A separate `audit_admin` role is created for legitimate maintenance operations (e.g., purging test data from a local development instance). This role is NOT granted to the application at runtime. Its credentials are stored separately and used only during explicit maintenance operations, never during normal platform operation.

### Transaction Isolation
All audit log inserts use `INSERT ... RETURNING id` within the same database transaction as the operation being audited. If the operation rolls back, the audit entry rolls back with it. If the operation commits, the audit entry commits atomically. This prevents "orphaned" audit entries (entries that reference operations that never completed) and "missing" audit entries (operations that completed without a corresponding audit record).

### Migration Path to Enterprise
Volume 10 (Enterprise & Multi-Tenant) will replace the Postgres-based audit log with a dedicated append-only storage system (e.g., S3 WORM, Amazon QLDB, or HashiCorp Vault audit device). The migration path is:
1. **Dual-write phase:** Audit entries are written to both Postgres (for backward compatibility with existing queries) and the append-only store.
2. **Read migration phase:** All audit queries are redirected to the append-only store.
3. **Postgres decommission phase:** The Postgres `audit_log` table is set to read-only and eventually dropped.

This migration path is documented in Volume 6, Chapter 4 ("Enterprise Audit Architecture") but is NOT implemented in v0.1.

## Alternatives Considered

### Option A: Application-Level Enforcement Only
**Description:** Enforce append-only semantics in application code by ensuring that no service, module, or agent ever calls `UPDATE` or `DELETE` on the `audit_log` table. This is enforced through code review, access patterns (only exposing an `appendAuditEntry()` function), and unit tests.
**Pros:**
- No database schema changes required.
- Simpler to implement — no triggers, no role management.
- Works with any database backend (not Postgres-specific).
- Easier to test in unit tests (mock the repository).
**Cons:**
- Fundamentally vulnerable to any application-level compromise. If an attacker gains the ability to execute arbitrary SQL through the application (e.g., via SQL injection, compromised dependency, or a bug in the query builder), they can modify or delete audit entries.
- A buggy ORM configuration or a developer's `console.log` debugging session could accidentally issue an UPDATE or DELETE.
- No defense-in-depth. A single application-level bug defeats the entire immutability guarantee.
- Does not protect against a malicious insider with database access who bypasses the application layer.
- Enterprise customers and security auditors will not accept application-level-only enforcement as sufficient.
**Why not chosen:** Application-level enforcement is the wrong layer for immutability. The entire point of an audit log is to be trustworthy EVEN WHEN the application is compromised. If the application is the only enforcer, the audit log's trustworthiness collapses exactly when it is most needed. This is a well-understood principle in security engineering — the enforcer must be at a lower privilege level than the entity it is enforcing against.

### Option B: Dedicated Append-Only Storage from Day One
**Description:** Use an infrastructure-level append-only storage system (e.g., Amazon S3 with Object Lock / WORM, Amazon QLDB, or a dedicated append-only log service) as the primary audit log backend from v0.1.
**Pros:**
- Strongest possible immutability guarantee. S3 WORM and QLDB provide infrastructure-level, tamper-evident storage that no application or database administrator can circumvent.
- Enterprise-ready from day one. No migration needed for Volume 10.
- Provides cryptographic verification of log integrity (e.g., QLDB's Merkle tree hash).
- Aligns with compliance frameworks (SOC 2, FedRAMP) that require infrastructure-level controls.
**Cons:**
- Introduces an external infrastructure dependency into v0.1. The v0.1 target is a single-operator CLI tool that works on a developer's laptop with no external services beyond the LLM provider.
- S3 WORM requires an AWS account, IAM configuration, and network access. QLDB requires an AWS account and has its own pricing model.
- Increases onboarding friction significantly. A developer evaluating the platform would need to set up AWS infrastructure before seeing their first audit log entry.
- The infrastructure cost is non-zero even for development/testing usage.
- Local development and offline usage (plausible for a CLI tool) become impossible or require a local append-only storage solution (e.g., a custom append-only file format), which is itself significant implementation work.
- Violates the v0.1 scope constraint of "zero infrastructure dependencies beyond the LLM provider."
**Why not chosen:** The immutability benefit is real, but the cost to v0.1 adoption is disproportionate. A single developer evaluating the CLI should not need an AWS account. The Postgres trigger approach provides strong immutability guarantees within the database layer (which is already a v0.1 dependency) and the documented migration path ensures a clean transition to infrastructure-level storage for enterprise deployment.

### Option C: PostgreSQL Triggers with Role-Based Permissions (Chosen)
**Description:** Enforce immutability at the database layer using BEFORE UPDATE/DELETE triggers that reject all modifications, combined with REVOKE of UPDATE and DELETE permissions on the `audit_log` table for the application role.
**Pros:**
- Enforcement is at a lower privilege level than the application. Even if the application is fully compromised, the database rejects modification attempts.
- Zero additional infrastructure dependencies — Postgres is already the v0.1 database (ADR-0003).
- Triggers are atomic with the transaction; a modification attempt raises an exception that the application cannot suppress.
- Role-based permissions provide a second, independent enforcement layer. Even if triggers are somehow disabled (e.g., by a superuser), the application role lacks the permissions to modify rows.
- Standard PostgreSQL feature — well-documented, well-understood, no custom code.
- The `audit_admin` role provides a controlled escape hatch for legitimate maintenance without compromising runtime immutability.
- Transparent to the application. The application only issues INSERT statements; the triggers are invisible to normal operation.
**Cons:**
- PostgreSQL-specific. If the database backend changes (e.g., to SQLite for embedded mode), the trigger mechanism must be re-implemented.
- A PostgreSQL superuser can disable triggers (`ALTER TABLE audit_log DISABLE TRIGGER ALL`) or grant themselves UPDATE/DELETE permissions. This is a fundamental limitation of database-level enforcement — it protects against application-level compromise but not against database-administrator-level compromise.
- Triggers add a small performance overhead to every INSERT (the trigger must check if the operation is UPDATE/DELETE, though the BEFORE UPDATE/DELETE triggers only fire on those operations, so INSERT overhead is zero).
- The dual-write migration path for Volume 10 adds complexity during the transition period.
- The `audit_admin` role's credentials must themselves be secured — if they are stored in the same environment as the application credentials, a compromise could escalate to audit log modification.

## Consequences

- **Positive:** The audit log is protected against application-level compromise, which is the highest-likelihood threat for v0.1 (e.g., a compromised agent attempting to cover its tracks).
- **Positive:** The dual enforcement (triggers + role permissions) provides defense-in-depth within the database layer.
- **Positive:** The mechanism is invisible to normal application operation — no code changes needed in audit-producing services.
- **Positive:** The documented migration path to S3 WORM/QLDB (Volume 10) ensures the v0.1 investment is not thrown away.
- **Negative:** The mechanism does not protect against a PostgreSQL superuser with malicious intent. This is acceptable for v0.1 (single operator, local database) but must be addressed before enterprise deployment.
- **Negative:** The `audit_admin` role introduces a secondary credential that must be managed. If this credential is leaked or stored insecurely, the immutability guarantee is compromised.
- **Negative:** Database-trigger-based enforcement is not portable across database backends. If a future version supports SQLite for embedded mode, an equivalent mechanism must be implemented (e.g., SQLite's `UPDATE` trigger support).
- **Impact on existing contracts:** Volume 6 gains a new section (Chapter 3, §Immutability Enforcement) describing the trigger and role-based mechanism. The Volume 6 schema migration files gain the trigger creation DDL. The Engineering Execution Program gains a deployment checklist item: "verify audit_log triggers are active in the deployed database."
- **When to revisit:** When Volume 10 (Enterprise & Multi-Tenant) enters Specification Review. At that point, the migration to infrastructure-level append-only storage (S3 WORM, QLDB) will be designed. Also revisit if v0.1 adds SQLite support — an equivalent trigger mechanism must be specified for that backend.
