# ADR-0016: Marketplace, Billing, and Plugin Licensing Out of Scope Until Plugin Trust Model Exists

**Status:** Final
**Date:** 2026-07-13
**Supersedes:** —
**Related RFCs:** RFC-0027 (Plugin Sandboxing & Trust Model), RFC-0036 (Marketplace Sequencing, future)
**Related Volumes:** 8 (Plugin Platform), Engineering Backlog (ENG-14)

---

## Context

The original agentx project objective includes language about a plugin marketplace — a distribution channel where third-party developers can publish and sell plugins for the platform. This is a natural long-term aspiration for any extensible platform. However, the Architecture Assessment (score 3.0/5.0) and the subsequent Improvement Plan identified that no Volume owns marketplace, billing, or plugin licensing. The Engineering Backlog correctly deferred this work item (ENG-14), but the deferral was recorded only as a backlog entry, not as a formal architectural decision.

Per Constitution Principle 1 (Architecture First), an unstated "no" is a silent violation. The project is implicitly saying "we are not building a marketplace" by omitting it from every Volume's scope, but this implicit deferral is not recorded as a decision with rationale. This matters because:

1. **Future contributors may assume the marketplace is planned and design interfaces with it in mind**, leading to premature abstraction or scope creep.
2. **External observers (potential plugin authors, enterprise customers, investors) may expect a marketplace timeline** that does not exist.
3. **The absence of a recorded decision makes it too easy to reverse the deferral without proper analysis** — someone could add marketplace scope to a Volume's PR without understanding the prerequisites.

This ADR records the deferral as a formal architectural decision, specifies the prerequisite conditions for reconsideration, and draws a clear boundary between what IS in scope (the plugin trust model and extension points) and what is NOT (the distribution, billing, and licensing layer on top).

## Decision

Marketplace, billing, and plugin licensing are explicitly out of scope for all Volumes through v1.0. This decision has three parts:

### Part 1: What Is Out of Scope
The following capabilities are NOT to be designed, specified, or implemented in any Volume through v1.0:
- **Plugin marketplace UI or API:** Any interface for browsing, searching, installing, or reviewing third-party plugins.
- **Billing and payments:** Any system for collecting payment from operators for plugins, or from plugin authors for marketplace listing.
- **Plugin licensing:** Any system for enforcing license terms on plugin usage (e.g., per-seat licensing, time-limited trials, feature gating based on license tier).
- **Plugin versioning and distribution:** Any system for hosting, versioning, or distributing plugin packages to operators (beyond the local filesystem installation that Volume 8 already defines for v0.1).
- **Plugin author identity and trust scores:** Any reputation or identity system for plugin authors.

### Part 2: What Remains In Scope
This decision does NOT prevent the following work, which is essential infrastructure for any future marketplace:
- **Plugin trust model (RFC-0027):** Sandboxing, capability restrictions, and trust boundary definitions for third-party plugins. This is the prerequisite for a marketplace and MUST be completed first.
- **Extension points (Volume 8):** The architectural interfaces that plugins implement. These should be designed for extensibility without assuming a marketplace distribution mechanism.
- **Plugin installation from local filesystem (Volume 8, v0.1):** The current scope allows installing a plugin from a local directory. This is sufficient for v0.1 and provides the test surface for the trust model.

### Part 3: Reconsideration Criteria
Marketplace, billing, and licensing may be reconsidered after BOTH of the following conditions are met:

1. **RFC-0027 (Plugin Sandboxing & Trust Model) is implemented AND verified** with at least one real third-party plugin running in the sandbox. "Verified" means: the plugin has been tested against the sandbox boundaries, the trust model's capability restrictions have been validated, and no privilege escalation paths have been found.

2. **RFC-0036 (Marketplace Sequencing)** has been authored, reviewed, and Approved. RFC-0036 will make the build-vs-wait decision: does the project build a first-party marketplace, integrate with an existing marketplace (e.g., npm, a package manager), or wait for organic demand? RFC-0036 does not exist yet and will not be authored until condition 1 is met.

These conditions ensure that the marketplace is built on a foundation of verified security, not optimistic assumptions.

## Alternatives Considered

### Option A: Build Marketplace Now with Minimal Trust
**Description:** Implement a basic marketplace (plugin listing, installation from a registry) in parallel with or shortly after the v0.1 CLI release, using a minimal trust model (e.g., "all plugins from the official registry are trusted; third-party plugins run at your own risk").
**Pros:**
- Provides a distribution channel early, enabling community plugin development sooner.
- The marketplace itself is a compelling feature for user adoption and marketing.
- A minimal marketplace can be built quickly (estimated 4-6 weeks) and iterated on.
- Real-world marketplace usage would provide valuable feedback for the trust model design.
**Cons:**
- "Minimal trust" is a euphemism for "no trust." A plugin from the official registry that is later found to contain malicious code damages the platform's reputation permanently.
- The marketplace's security posture is only as strong as its weakest plugin. A single compromised plugin in the registry is an attack vector against every operator who installs it.
- Building the marketplace before the trust model means the marketplace's architecture may need significant rework once the trust model is finalized (e.g., capability-based installation, sandbox-aware dependency resolution).
- Creates premature commitment to a marketplace architecture that may not align with the trust model's requirements.
- The "all plugins from the official registry are trusted" approach does not scale. The moment the registry accepts third-party submissions, the trust assumption breaks.
**Why not chosen:** The security risk is unacceptable. The agentx platform's primary value proposition is a trustworthy AI agent system. A marketplace with minimal trust is a liability, not an asset. Building it before the trust model is verified would be putting the cart before the horse — and in this case, the horse is a security boundary.

### Option B: Permanently Exclude Marketplace
**Description:** Amend the project's long-term vision to permanently exclude a plugin marketplace, billing, and licensing. The platform remains extensible via local filesystem plugins only, forever.
**Pros:**
- Eliminates the security risk of third-party plugin distribution entirely.
- Removes a significant scope item from the 10-year roadmap, simplifying planning.
- Avoids the complexity of billing, payments, and license enforcement — all of which are non-trivial systems.
- The platform can focus entirely on its core value (AI agent software engineering) without the distraction of marketplace operations.
**Cons:**
- Overly restrictive for a platform designed for a 10-year lifespan. A healthy plugin ecosystem is a standard expectation for extensible platforms, and the absence of a distribution mechanism limits adoption.
- Forces plugin authors to use ad-hoc distribution methods (GitHub repos, manual installation instructions) that provide a worse user experience than a marketplace.
- Removes a potential revenue stream (marketplace fees, premium plugins) that could sustain long-term development.
- Does not prevent the demand for a marketplace from arising organically. If the platform is successful, the community will build its own unofficial marketplace, which the project has even less control over.
- Inconsistent with the project's stated objective, which mentions a marketplace as a long-term aspiration.
**Why not chosen:** Permanently excluding the marketplace is as much of an unexamined decision as silently deferring it. A 10-year platform should not foreclose a major capability based on current constraints (v0.1 scope, trust model not yet built). The right approach is to defer with clear prerequisites (Option C), not to permanently exclude.

### Option C: Defer Until Trust Model Is Verified (Chosen)
**Description:** Explicitly defer marketplace, billing, and licensing until RFC-0027 (Plugin Sandboxing & Trust Model) is implemented, verified with a real third-party plugin, and RFC-0036 (Marketplace Sequencing) has made the build-vs-wait decision.
**Pros:**
- Ensures the marketplace is built on a foundation of verified security, not optimistic assumptions.
- Preserves the option to build a marketplace when the time is right, without committing to a specific timeline or architecture.
- The plugin trust model (Volume 8, RFC-0027) is essential infrastructure regardless of whether a marketplace is ever built — it protects operators from malicious plugins even when those plugins are installed from a local filesystem.
- RFC-0036 provides a natural decision point where build-vs-wait can be evaluated with real data (adoption metrics, trust model effectiveness, community demand).
- No scope creep into the v0.1 or v1.0 roadmap — the current Volumes remain focused.
- The explicit, recorded deferral satisfies Constitution Principle 1 and prevents silent reversal.
- Enterprise customers (Volume 10) who want a private plugin registry can implement their own distribution mechanism on top of the trust model, independent of any public marketplace.
**Cons:**
- The platform will not have a marketplace for v0.1 or v1.0, which may be a competitive disadvantage compared to platforms that ship a marketplace earlier.
- Plugin authors must distribute plugins through ad-hoc methods (GitHub, manual installation) in the interim. This limits the plugin ecosystem's growth.
- The two-gate reconsideration criteria (RFC-0027 verified + RFC-0036 Approved) could be perceived as overly cautious, especially if the trust model works well in practice before the verification threshold is formally met.
- There is a risk that the marketplace is never reconsidered if the project's priorities shift — the deferral could become permanent by inertia rather than by decision.
- The enterprise customer self-service distribution path (private registries) may require more Volume 8 extension-point design work than currently scoped, if those customers need capabilities not exposed by the current plugin interface.

## Consequences

- **Positive:** The platform's v0.1 and v1.0 scope remains focused on core capabilities, avoiding the distraction of marketplace infrastructure.
- **Positive:** The plugin trust model (RFC-0027) is prioritized as the critical prerequisite, ensuring security is addressed before distribution.
- **Positive:** The explicit deferral satisfies Constitution Principle 1 and prevents scope creep through silent omission.
- **Positive:** Enterprise customers (Volume 10) retain the ability to build private plugin distribution on top of the trust model, independent of any public marketplace timeline.
- **Negative:** No marketplace through v1.0. Plugin distribution is limited to local filesystem installation (Volume 8 v0.1 scope).
- **Negative:** The platform may be perceived as less mature or less extensible compared to competitors that ship a marketplace earlier.
- **Negative:** Plugin authors in the interim must use ad-hoc distribution methods, potentially leading to fragmentation and inconsistent installation experiences.
- **Impact on existing contracts:** No existing Volume contracts change. Volume 8's scope is confirmed as extension-point and trust-model only, with no marketplace layer. The Engineering Backlog item ENG-14 is formally ratified as a post-v1.0 item with the reconsideration criteria specified above. RFC-0036 is registered as a future RFC (not yet authored).
- **When to revisit:** When RFC-0027 (Plugin Sandboxing & Trust Model) is implemented and has been verified with at least one real third-party plugin. At that point, RFC-0036 (Marketplace Sequencing) will be authored to make the build-vs-wait decision. Also revisit if enterprise customers (Volume 10) explicitly request a private plugin registry before the public marketplace — this may require adjusting the reconsideration criteria to allow a private-registry-only path.
