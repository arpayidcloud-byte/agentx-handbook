# RFC-0021: Threat Model & Trust Boundaries for Tool SDK

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 7, 1

---

## Context

The Tool SDK (Volume 7) grants AI agents the ability to execute shell commands, read and write
files, and interact with the local filesystem. These capabilities are inherently powerful and
represent the highest-risk surface area in the agentx platform. An agent operating under
prompt injection, a misconfigured permission grant, or a malicious plugin could leverage these
tools to escape intended boundaries, exfiltrate data, or cause irreversible damage to the
operator's system.

Constitution Principle 7 (Security & Isolation) mandates that every Volume address its threat
model explicitly. While ADR-0004 establishes fail-closed permission checks and ADR-0005
classifies `fs.write` as destructive, neither document provides a comprehensive threat model
or defines the trust boundaries that partition the system. ADR-0011 reinforces this by
gating RFC-0021 on the existence of a lightweight STRIDE-per-trust-boundary threat model,
citing Constitution Principle 1 (Architecture First): no architectural decision may be made
by silent omission. Without a shared threat model, security review of tool implementations
becomes ad hoc and inconsistent across contributors.

This RFC fills that gap by defining a STRIDE-based threat analysis, explicit trust boundary
diagram, and threat categories that all tool implementations must acknowledge in their
security review checklist.

## Proposed Decision

Adopt a formal threat model for the Tool SDK based on the STRIDE methodology, with four
explicit trust boundaries and a mandatory threat-category checklist that every tool
implementation must complete before merge.

### Trust Boundaries

The system is partitioned into four trust zones, each separated by a trust boundary that
enforces access control:

1. **Operator-Trusted Zone:** The human operator, their workstation, and local config files.
   The operator is the root of trust. No agent output leaves this zone without explicit
   operator action (e.g., CLI approval prompt per Volume 9).

2. **Agent-Controlled Zone:** The LLM inference loop, system prompts, agent memory, and
   tool-call orchestration logic. Code in this zone is controlled by AI model output and
   is therefore untrusted. Every cross-boundary call from this zone must pass through
   `PermissionChecker.isAllowed()` (ADR-0004).

3. **Tool-Executed Zone:** The actual tool implementations (`shell.exec`, `fs.read`,
   `fs.write`, etc.). These run with the process permissions of the agentx runtime.
   Inputs from the Agent-Controlled Zone must be sanitized and validated before execution.

4. **External Network Zone:** Any service reachable over the network (LLM providers, external
   APIs, package registries). Data crossing this boundary is subject to TLS and
   provider-specific authentication per Volume 4.

### STRIDE Threat Categories

| Category          | Example Threat                                          | Mitigation                                       |
|-------------------|---------------------------------------------------------|--------------------------------------------------|
| **S**poofing     | Agent impersonates operator to self-approve actions     | Approval tokens are operator-only, never in agent context |
| **T**ampering    | Prompt injection modifies tool call parameters          | Input validation, allowlist of permitted commands |
| **R**epudiation  | Agent denies executing a destructive file write         | Append-only audit log (RFC-0024)                  |
| **I**nformation Disclosure | Agent reads `/etc/shadow` or `.env` files       | Filesystem allowlist, deny-by-default (ADR-0004)  |
| **D**enial of Service | Agent fills disk with large file writes or forks bombs | Resource limits: max file size, max exec duration, process caps |
| **E**levation of Privilege | `shell.exec` with unsanitized input runs `sudo`  | Shell command parsing, forbidden token list (`sudo`, `chmod`, etc.) |

### Mandatory Checklist

Every tool implementation PR must include a completed threat-model checklist referencing the
above categories. The checklist is a markdown table in the PR description with one row per
applicable STRIDE category and a column for the specific mitigation applied. A PR missing
this checklist is blocked by CI (enforced via a markdown-lint rule).

## Alternatives Considered

### Alternative A: Trust All Tool Calls (No Threat Model)
**Description:** Treat the agent as fully trusted and rely solely on system prompt instructions
to constrain behavior. No formal threat model, no trust boundaries, no mandatory checklist.

**Pros:**
- Zero implementation overhead — no checklist, no extra review step.
- Fastest path to shipping v0.1 features.
- No cognitive load on contributors unfamiliar with threat modeling.

**Cons:**
- Directly contradicts Constitution Principle 7 (Security & Isolation).
- Prompt-only constraints are proven insufficient (ADR-0004 context explicitly states this).
- A single prompt-injection attack grants the agent full system access.
- No structured way to communicate security expectations to new contributors.

**Why rejected:** ADR-0004 already established that prompt-only enforcement is insufficient.
Adopting a no-threat-model approach would contradict an existing Final ADR and a
constitutional principle.

### Alternative B: Full Formal Threat Model with External Audit
**Description:** Commission a professional penetration test and formal threat model (e.g.,
using Microsoft Threat Modeling Tool) before any tool ships. Produce a report, track
findings in a JIRA-style tracker, and require all findings resolved before v0.1 GA.

**Pros:**
- Most rigorous approach; catches threats that internal analysis misses.
- Provides auditable artifact for enterprise customers and SOC2 compliance (RFC-0035).
- Establishes security credibility early.

**Cons:**
- External audit costs money and weeks of calendar time — incompatible with v0.1 timeline.
- Overkill for a single-developer or small-team project at this stage.
- Threat landscape will change as new tools are added, making a one-time audit stale.
- Creates a gate that could block all v0.1 delivery.

**Why rejected:** The cost and timeline are disproportionate to v0.1 risk. The recommended
approach achieves 80% of the benefit at 20% of the cost. An external audit is appropriate
for v1.0 enterprise readiness, not v0.1.

### Alternative C: STRIDE-Based Model with Mandatory Checklist (Recommended)
**Description:** As described in Proposed Decision above. Lightweight, contributor-friendly,
enforced via CI markdown-lint, and references existing ADRs.

**Pros:**
- Systematic coverage of all major threat categories without external cost.
- CI-enforced, so compliance is automatic, not aspirational.
- Extensible — new threat categories can be added to the checklist template.
- Directly satisfies Constitution Principle 7's "per-Volume security" requirement.
- References and builds on existing ADR-0004 and ADR-0005 decisions.

**Cons:**
- Adds ~15 minutes of overhead per tool implementation PR.
- A checklist is only as good as the reviewer — superficial completions are possible.
- STRIDE is a methodology, not a guarantee; sophisticated threats may be missed.

## Consequences

- **Positive:** Every tool implementation ships with a documented security rationale.
  Security review becomes structured and repeatable. New contributors have a clear template
  to follow. Constitution Principle 7 compliance is demonstrable.
- **Negative:** Slight increase in PR turnaround time for new tool implementations.
  Contributors unfamiliar with STRIDE may need onboarding support.
- **Migration path (if applicable):** Existing tools (`shell.exec`, `fs.read`, `fs.write`)
  must be backfilled with threat-model checklists within one sprint of this RFC's adoption.
- **Affected interfaces/contracts:** `PermissionChecker.isAllowed()` (Volume 7) gains a
  new parameter for threat-category tagging in audit logs. The CI pipeline gains a new
  `rfc-threat-model-check` lint step.
