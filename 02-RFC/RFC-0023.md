# RFC-0023: Credential Runtime Retrieval Contract

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 4, 16

---

## Context

RFC-0022 defines the `SecretsBackend` interface and the pluggable storage mechanism for
secrets. However, the contract between `CredentialResolver` and `SecretsBackend` at runtime
has several behavioral properties that are not yet specified:

1. **Caching:** Should credentials be cached in memory to avoid repeated backend calls?
   If so, what is the TTL, and how is cache invalidation handled?

2. **Logging:** What may be logged about credential retrieval? Leaking a credential value
   into a log line is a critical security incident.

3. **Error Handling:** What happens when a credential is missing? Should the system fail
   loudly, return a placeholder, or retry?

4. **Rotation Invalidation:** When a credential is rotated (via `SecretsBackend.rotate()`),
   in-flight requests using the old credential may fail. How is this handled?

These questions must be answered before `CredentialResolver` can be implemented correctly.
Ambiguity in any of these areas leads to either security vulnerabilities (credential
leakage) or operational fragility (stale credentials causing cascading failures).

## Proposed Decision

Formalize the `CredentialResolver` runtime behavior contract with four guarantees:

### 1. Credential Caching (In-Memory, TTL-Based)

```typescript
interface CacheConfig {
  ttlMs: number;       // default: 300_000 (5 minutes)
  maxSize: number;     // default: 100 entries; LRU eviction
  negativeTtlMs: number; // default: 5_000 (5 seconds) for cache on "not found"
}
```

- Credentials are cached in an in-memory LRU map after first retrieval.
- Default TTL is 5 minutes, configurable via `AGENTX_CREDENTIAL_TTL_MS`.
- A "negative cache" entry is created when a secret is not found, preventing
  thundering-herd calls to the backend. Negative TTL defaults to 5 seconds.
- Cache is **per-process** — not shared across worker threads or instances. This is
  intentional: each process maintains its own cache, eliminating distributed cache
  coordination complexity at v0.1.

### 2. No-Logging Guarantee

The following invariant MUST hold for all code paths in `CredentialResolver` and
`SecretsBackend`:

> **The credential value (the secret itself) MUST NEVER appear in any log output,
> error message, stack trace, or debug artifact.**

Implementation rules:
- `CredentialResolver.resolve()` logs only the key name and a boolean indicating
  cache hit/miss: `[CredentialResolver] resolve("provider.openai.api_key") cache=miss`.
- `SecretsBackend.get()` returns the value; the caller is responsible for never
  stringifying it into a log. A `toSafeString()` utility must be used if any
  serialization is needed: `toSafeString(credential) => "***REDACTED***"`.
- Error objects from `SecretsBackend` must not include the credential value in their
  `message` property. A `SecretAccessError` class is provided with a fixed message:
  `"Secret access failed for key: {key}"`.

A contract test enforces this: it monkey-patches `console.log`, `console.error`,
and `console.warn`, calls `resolve()` with a known secret, and asserts no output
contains the secret value.

### 3. Error Handling When Secret Is Missing

When `SecretsBackend.get()` throws `SecretNotFoundError`:

- `CredentialResolver` wraps it in a `CredentialResolutionError` with a clear,
  actionable message: `"Credential not found: provider.{providerId}.api_key. Set it via 'agentx secrets set' or the AGENTX_{PROVIDER}_API_KEY env var."`.
- The error is thrown immediately — no fallback, no retry, no silent default.
- This is consistent with ADR-0004 (fail-closed): an unrecognized or missing credential
  means the provider cannot be used, and attempting to proceed would produce confusing
  errors downstream.

### 4. Cache Invalidation on Rotation

When `SecretsBackend.rotate()` is called:

- The rotated key is immediately evicted from the in-memory cache (both positive and
  negative entries).
- A `CredentialRotatedEvent` is emitted to the application event bus (Volume 5,
  Workflow Engine). Consumers that hold a long-lived reference to the old credential
  should listen for this event and re-resolve.
- In-flight requests that already retrieved the old credential are **not** interrupted.
  They may fail with an authentication error from the provider. This is acceptable:
  credential rotation is an explicit operator action, and the window of in-flight
  request failure is bounded by the provider's token validation window (typically
  seconds, not minutes).

## Alternatives Considered

### Alternative A: No Caching — Direct Backend Call Every Time
**Description:** `CredentialResolver.resolve()` always calls `SecretsBackend.get()`.
No in-memory cache.

**Pros:**
- Simplest implementation. No cache invalidation logic needed.
- Always returns the most recent value — no stale credential window.
- No memory overhead for cache storage.

**Cons:**
- Every LLM API call triggers a secrets backend lookup. For a `VaultBackend`, this
  means a network round-trip per inference request — adding 5–50ms latency.
- For the `EncryptedFileBackend`, this means a file read and decryption per request —
  adding 1–5ms and unnecessary CPU load.
- Thundering herd: if a secret is missing, repeated failed lookups hammer the backend.
- Violates the principle of not making the common case expensive for the sake of the
  rare case (rotation).

**Why rejected:** The performance cost is unacceptable for a hot path (every LLM API
call). Caching with a short TTL is standard practice and the complexity is minimal.

### Alternative B: Distributed Cache (Redis) for Credentials
**Description:** Use the existing Redis/BullMQ infrastructure (Volume 2) as a credential
cache. All worker processes share the same cache via Redis.

**Pros:**
- Single cache across all processes — rotation invalidation is instant and global.
- Redis already exists in the infrastructure; no new dependency.
- Supports cache metrics and inspection via Redis CLI.

**Cons:**
- Stores secrets in Redis, which adds another location where secrets exist in memory.
  Redis is not designed as a secrets manager — its data is serialized to disk in RDB
  snapshots unless `save ""` is configured.
- Increases blast radius: a Redis compromise now exposes all cached credentials.
- Adds a network hop to every cache lookup (vs. in-memory nanosecond access).
- The `EnvVarBackend` and `EncryptedFileBackend` are designed for local, single-process
  use — a distributed cache creates an impedance mismatch with their deployment model.

**Why rejected:** Storing secrets in Redis, even cached, expands the attack surface
contrary to Constitution Principle 7. A per-process in-memory cache is sufficient for
v0.1 and avoids the Redis-secrets anti-pattern.

### Alternative C: In-Memory TTL Cache with No-Logging Guarantee (Recommended)
**Description:** As described in Proposed Decision. Per-process LRU cache with 5-minute TTL,
no-logging invariant enforced by contract test, fail-closed error handling.

**Pros:**
- Nanosecond cache access for the hot path — zero network overhead.
- No secrets leak into external systems (Redis, disk, logs).
- TTL provides automatic staleness bound — worst case is 5 minutes of stale data.
- Negative cache prevents thundering herd on missing secrets.
- Simple to implement, test, and reason about.

**Cons:**
- Per-process cache means up to 5 minutes of stale data after rotation across processes.
  Acceptable: rotation is rare and operator-initiated.
- Cache memory is unbounded in the absence of `maxSize` (mitigated by LRU eviction).
- No global visibility into cache state (mitigated by exposing cache-hit/miss metrics
  to Volume 13 observability).

## Consequences

- **Positive:** Credential retrieval is fast, secure, and predictable. The no-logging
  guarantee is enforceable via automated tests. Error messages guide operators to
  self-service resolution. Rotation is handled gracefully with bounded staleness.
- **Negative:** Per-process cache means rotation is not instant across all workers —
  a 5-minute window of potential stale credential use exists. This is documented and
  accepted as a v0.1 trade-off.
- **Migration path (if applicable):** Existing `CredentialResolver` implementations
  (if any) must be updated to implement the caching and no-logging guarantees. The
  `SecretsBackend` interface from RFC-0022 is unchanged.
- **Affected interfaces/contracts:** `CredentialResolver` class (Volume 4, Ch. 3) gains
  `CacheConfig` parameter. New `SecretNotFoundError`, `CredentialResolutionError`,
  `SecretAccessError` exception types. New `CredentialRotatedEvent` in Volume 5 event
  bus. New contract test: `credential-no-log-leak.contract.test.ts`.
