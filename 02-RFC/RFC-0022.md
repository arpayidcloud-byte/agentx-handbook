# RFC-0022: Secrets Storage Backend & Self-Hosted Fallback

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 4, 16, 11

---

## Context

The agentx platform requires secure storage and retrieval of secrets: LLM provider API keys,
database credentials, third-party service tokens, and encryption keys. These secrets are used
at runtime by the Provider Platform (Volume 4), the Cloud Platform (Volume 11), and future
enterprise features (Volume 16). Currently, no formal design exists for how secrets are
stored or retrieved — the implicit assumption has been environment variables, which is
adequate for local development but insufficient for multi-environment, multi-tenant, or
enterprise deployment.

ADR-0007 (Mandatory Self-Hosted Fallback) requires that every managed service dependency
have a working self-hosted alternative. A secrets manager is a managed service dependency:
HashiCorp Vault and AWS Secrets Manager are the obvious enterprise choices, but both require
a self-hosted fallback that works without external infrastructure. ADR-0012 establishes the
v0.1 baseline: environment variables accessed through a `CredentialResolverConfig` abstraction
layer, with encrypted file storage explicitly deferred beyond v0.1 due to complexity-to-benefit
ratio concerns for a single-operator CLI.

Additionally, Constitution Principle 9 (No Vendor Lock-in) demands that the secrets storage
backend be pluggable — the platform must not be architecturally coupled to any single
secrets manager vendor.

This RFC designs a `SecretsBackend` abstraction, defines the v0.1 default (encrypted
file-based), and specifies the enterprise upgrade path.

## Proposed Decision

Introduce a `SecretsBackend` interface with three implementations delivered across release
versions:

### Interface Design

```typescript
interface SecretsBackend {
  /** Retrieve a secret by key. Throws SecretNotFoundError if absent. */
  get(key: string): Promise<string>;

  /** Store a secret. Overwrites if key exists. */
  set(key: string, value: string, metadata?: SecretMetadata): Promise<void>;

  /** Delete a secret. No-op if key does not exist. */
  delete(key: string): Promise<void>;

  /** List all secret keys (values not included). */
  list(): Promise<string[]>;

  /** Rotate a secret: old value is invalidated, new value stored atomically. */
  rotate(key: string, newValue: string): Promise<void>;
}

interface SecretMetadata {
  rotatedAt?: Date;
  rotatedBy?: string; // operator identity
  version?: number;
}
```

### Implementation Roadmap

**v0.1 — EnvVarBackend (default):**

- `EnvVarBackend`: Reads from `process.env` following the `AGENTX_SECRET_{PROVIDER}_{KEY}`
  naming convention established in ADR-0012. `set()` and `rotate()` throw
  `OperationNotSupportedError`. This is the zero-config default for local development.
- No encrypted file storage at this stage — consistent with ADR-0012's decision that the
  complexity-to-benefit ratio is poor for a single-operator v0.1 CLI.

**v0.5 — EncryptedFileBackend (self-hosted fallback):**

- `EncryptedFileBackend`: Stores secrets in `~/.agentx/secrets.enc` using AES-256-GCM.
  The encryption key is derived from a master secret provided via `AGENTX_MASTER_SECRET`
  environment variable (or an interactive prompt on first run). Each secret is stored as
  a JSON record with the ciphertext, IV, auth tag, and metadata. File permissions are set
  to `0600` (owner-read/write only). This provides the self-hosted fallback required by
  ADR-0007 for multi-environment deployments.

**v1.0 — VaultBackend + Cloud-Native Backends:**

- `VaultBackend`: Connects to HashiCorp Vault (self-hosted or HCP). Supports full CRUD,
  rotation, and automatic lease renewal for dynamic secrets.

**v1.0 — AWSSecretsManagerBackend + AzureKeyVaultBackend (cloud-native):**

- Cloud-native backends for enterprise deployment. Both implement the same interface.
  Configuration maps to Volume 11's managed-service table (per ADR-0007, each must have a
  self-hosted fallback documented).

### CredentialResolver Extension

Volume 4's `CredentialResolver` (Ch. 3) is extended to accept a `SecretsBackend` instance
via dependency injection, building on the `CredentialResolverConfig` abstraction introduced
in ADR-0012. The resolver no longer reads `process.env` directly; instead, it calls
`secretsBackend.get()`. This decouples credential retrieval from environment variable
assumptions while remaining backward-compatible with the ADR-0012 interface contract.

```typescript
class CredentialResolver {
  constructor(private secrets: SecretsBackend) {}

  async resolve(providerId: string): Promise<ProviderCredentials> {
    const apiKey = await this.secrets.get(`provider.${providerId}.api_key`);
    return { apiKey };
  }
}
```

### Rotation Policy

- **v0.1:** Manual rotation only. Operator calls `agentx secrets rotate <key>` which
  invokes `SecretsBackend.rotate()`. No automated scheduling.
- **v1.0:** Automated rotation for Vault-backed secrets (Vault's native lease rotation).
  For file-backed secrets, a configurable TTL triggers a warning log prompting manual
  rotation.

## Alternatives Considered

### Alternative A: Environment Variables Only (No Abstraction)
**Description:** Continue using `process.env` for all secrets. No `SecretsBackend` interface.
Every module reads environment variables directly.

**Pros:**
- Zero implementation cost. Already works for local development.
- Universal — every language and framework supports env vars.
- Simple to debug (`env | grep AGENTX`).

**Cons:**
- No rotation support. Changing a secret requires a process restart.
- All secrets are visible to any process on the same machine (via `/proc/PID/environ`).
- No multi-tenant isolation — all operators share the same env var namespace.
- Violates ADR-0007: there is no self-hosted fallback because there is no managed
  service to fall back from, but the lack of any abstraction means migrating to a
  real secrets manager later requires touching every module that reads env vars.
- Secrets are in cleartext in shell history, Docker inspect output, and CI logs.

**Why rejected:** Environment variables are acceptable as a *backend implementation*, not
as the *only mechanism*. The lack of rotation, isolation, and auditability makes this
unsuitable beyond local development.

### Alternative B: Direct HashiCorp Vault Integration (No Abstraction)
**Description:** Hard-code Vault as the secrets backend. All modules use the Vault SDK
directly. No pluggable interface.

**Pros:**
- Vault is the industry standard for secrets management — mature, well-documented,
  widely deployed.
- Supports dynamic secrets, lease management, PKI, and transit encryption.
- Enterprise customers expect and often require Vault.

**Cons:**
- Violates Constitution Principle 9 (No Vendor Lock-in). The platform would be
  architecturally coupled to Vault's API surface.
- Violates ADR-0007's self-hosted fallback requirement in spirit — Vault *is*
  self-hostable, but the coupling means no alternative can be used without a rewrite.
- Adds Vault as a hard dependency for even local development, raising the barrier to
  entry for contributors.
- Vault's API surface is large; coupling to it leaks into application code (transit
  paths, lease IDs, auth tokens).

**Why rejected:** Vendor coupling is a constitutional violation. Vault is an excellent
*implementation* of the interface, not the interface itself.

### Alternative C: Pluggable SecretsBackend with Self-Hosted Fallback (Recommended)
**Description:** As described in Proposed Decision. Interface-based design with env vars
for v0.1, encrypted file at v0.5 for self-hosted, and Vault/AWS for v1.0 enterprise.

**Pros:**
- Satisfies Constitution Principle 9 (pluggable, no vendor lock-in).
- Satisfies ADR-0007 (every managed backend has a self-hosted fallback).
- Zero-config for v0.1 contributors (EnvVarBackend is the default).
- Clear migration path: swap the DI-injected backend, no application code changes.
- Rotation is part of the interface contract, enabling future automation.
- Consistent with the Provider Abstraction pattern (RFC-0001): define the interface,
  confine vendor specifics to adapter implementations.

**Cons:**
- Interface design requires upfront investment before any implementation exists.
- The encrypted file backend must be implemented and tested at v0.5, adding scope
to that milestone.
- Multiple backends increase testing matrix (each backend needs its own unit + contract
  tests).
- Key management for the file backend (master secret) introduces a bootstrapping problem:
  the master secret itself must be stored somewhere secure.

## Consequences

- **Positive:** Secrets management is architected for the full lifecycle (v0.1 through
  enterprise). The `SecretsBackend` interface provides a clean extension point. The
  encrypted file backend at v0.5 ensures the platform works fully offline with no
  external dependencies.
- **Negative:** v0.1 scope includes the `SecretsBackend` interface and `CredentialResolver`
  refactor only; the `EncryptedFileBackend` is deferred to v0.5 per ADR-0012.
- **Migration path (if applicable):** Existing code that reads `process.env` directly
  must be migrated to use `CredentialResolver` or inject `SecretsBackend`. A one-time
  refactor PR is required before v0.1 feature freeze.
- **Affected interfaces/contracts:** `CredentialResolver` constructor signature changes
  (Volume 4, Ch. 3). New `SecretsBackend` interface in a shared package
  (`packages/secrets/src/backend.ts`). New CLI commands: `agentx secrets list`,
  `agentx secrets set <key>`, `agentx secrets rotate <key>`.
