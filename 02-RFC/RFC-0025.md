# RFC-0025: Identity & Authentication Modes

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 15, 9, 10

---

## Context

The agentx platform currently assumes a single operator (Volume 9, Ch. 1): one human who
runs the CLI, approves tool executions, and manages configuration. This "single trusted
operator" model is correct for v0.1 but does not scale to multi-user teams, enterprise
organizations, or API-based access by external services.

As the platform evolves toward v0.5 (multi-user) and v1.0 (enterprise), a formal identity
and authentication model is required. Without it, there is no way to:
- Distinguish between multiple operators in audit logs.
- Enforce per-user RBAC permissions (RFC-0012, Volume 10).
- Support enterprise SSO (OIDC/SAML) for organization-managed users.
- Issue API tokens for programmatic access (CI/CD pipelines, external integrations).

Volume 15 (Identity Platform) is the planned home for this model, but it currently exists
only as a placeholder. This RFC defines the phased authentication strategy that Volume 15
will implement.

The design must respect Constitution Principle 7 (Security & Isolation) and integrate with
Volume 10's RBAC model (RFC-0012) without creating circular dependencies.

## Proposed Decision

Define three authentication modes, each tied to a release milestone, with a unified
`Identity` abstraction that all downstream systems consume:

### Unified Identity Abstraction

```typescript
interface Identity {
  /** Unique identifier for this identity (UUID). */
  id: string;

  /** Human-readable display name. */
  displayName: string;

  /** Authentication mode that issued this identity. */
  authMode: 'local' | 'token' | 'sso';

  /** Set of role IDs assigned to this identity (resolved via RFC-0026). */
  roleIds: string[];

  /** ISO 8601 timestamp when this identity was authenticated. */
  authenticatedAt: string;

  /** ISO 8601 timestamp when this identity expires (null = session-based). */
  expiresAt: string | null;

  /** The organization this identity belongs to (null for personal/local use). */
  organizationId: string | null;
}
```

### Mode 1: Local CLI Auth (v0.1)

- No explicit login flow. The operator who starts the agentx process is the sole identity.
- `Identity.id` is derived from a machine-local fingerprint (hostname + user ID hash).
- `Identity.authMode` is `'local'`.
- `Identity.roleIds` defaults to `['owner']` (full permissions per RFC-0012).
- Session is bound to the process lifetime — no token, no expiry.

**Rationale:** v0.1 targets a single developer on a single machine. Adding a login flow
would be premature friction. The identity is implicit but still represented in the audit
log for forward compatibility.

### Mode 2: Token-Based Auth (v0.5)

- Operators authenticate via `agentx auth login` (username/password or API key).
- On successful authentication, the system issues a signed JWT access token (15-minute
  TTL) and a refresh token (7-day TTL, stored in `~/.agentx/tokens`).
- All API requests include the access token in the `Authorization: Bearer <token>` header.
- The `TokenAuthService` validates the JWT signature, checks expiry, and resolves the
  `Identity` object.
- Token revocation is supported via a revocation list stored in PostgreSQL (checked on
  every validation — this is acceptable because auth validation is not a hot path).

**Security properties:**
- Tokens are signed with RS256 (asymmetric). The private key never leaves the server.
- Refresh tokens are stored with AES-256-GCM encryption at rest.
- Failed login attempts are rate-limited (5 per minute per account) and logged.

### Mode 3: Enterprise SSO via OIDC/SAML (v1.0+)

- The platform acts as an OIDC Relying Party (RP) or SAML Service Provider (SP).
- Identity providers supported: Okta, Azure AD, Google Workspace, any OIDC-compliant IdP.
- The `SSOAuthService` handles the OIDC/SAML protocol, exchanges the authorization code
  for tokens, and maps IdP claims to the `Identity` object.
- MFA enforcement is delegated to the IdP (the platform requires that the IdP enforce
  MFA — if the IdP does not enforce MFA, the platform rejects the authentication).
- JIT (Just-In-Time) provisioning: new users from the IdP are automatically created in
  the local database on first login, with default role `viewer` (per RFC-0012).
- SCIM provisioning (optional, v1.5+): bidirectional user sync with the IdP.

**Session management:**
- Sessions are tracked in PostgreSQL with a configurable idle timeout (default: 8 hours).
- Concurrent sessions per user are limited (default: 3). The oldest session is evicted
  when the limit is exceeded.
- Global logout (back-channel logout for OIDC) terminates all sessions across all
  agentx instances for a given user.

## Alternatives Considered

### Alternative A: Single Authentication Mode from Day One
**Description:** Pick one mode (e.g., token-based auth) and use it for all releases,
  including v0.1.

**Pros:**
- No mode-switching complexity. One code path, one set of tests.
- Operators learn one authentication mechanism and never need to relearn.
- RBAC integration is simpler (only one identity resolution path).

**Cons:**
- Token-based auth requires a user database, password hashing, token signing keys,
  and a login UI — all of which are unnecessary overhead for a single-user v0.1 CLI.
- OIDC/SAML requires an external IdP, which violates the self-hosted, no-external-
  dependency ethos of v0.1.
- Violates Constitution Principle 10 (Small Stable Core): adding enterprise auth
  machinery to the core at v0.1 inflates the core beyond its minimal useful size.

**Why rejected:** The phased approach matches the platform's staged release strategy.
  Each mode is introduced when its complexity is justified by real usage patterns.

### Alternative B: Delegate All Authentication to a Reverse Proxy
**Description:** Place an authentication reverse proxy (e.g., Authelia, OAuth2 Proxy,
  Cloudflare Access) in front of the agentx API. The application trusts the
  `X-Forwarded-User` header set by the proxy.

**Pros:**
- Zero authentication code in the application — fully delegated.
- Supports any auth mechanism the proxy supports (OIDC, SAML, basic, etc.).
- Battle-tested proxy implementations handle edge cases (token refresh, session
  management, MFA).

**Cons:**
- Violates Constitution Principle 9 (No Vendor Lock-in): the application becomes
  dependent on a specific proxy's behavior and header conventions.
- The `Identity` object must still be constructed somewhere — the proxy only provides
  a header, not a structured identity with roles, org, and metadata.
- Local development requires running the proxy locally, raising the barrier to entry
  for contributors.
- No standard for what headers the proxy sets — each proxy uses different conventions,
  making the integration non-portable.

**Why rejected:** Delegation without a contract leads to implicit coupling. The proxy
  approach is viable as an **optional deployment mode** for enterprise, but it cannot
  be the sole authentication mechanism because it doesn't work for local CLI usage.

### Alternative C: Phased Authentication Modes (Recommended)
**Description:** As described in Proposed Decision. Three modes, each introduced at the
  appropriate release milestone, with a unified `Identity` abstraction.

**Pros:**
- v0.1 remains simple — no auth infrastructure needed.
- v0.5 introduces auth when multi-user support is actually needed.
- v1.0+ integrates with enterprise IdPs via standard protocols.
- The unified `Identity` interface means downstream code (RBAC, audit, logging) never
  needs to know which mode is active — it just consumes `Identity` objects.
- Each mode can be developed and tested independently.
- Consistent with Constitution Principle 10 (Small Stable Core): auth complexity grows
  only as needed.

**Cons:**
- Three code paths for authentication increase testing surface area.
- Migration from Mode 1 (local) to Mode 2 (token) requires existing local operators
  to create an account — a one-time friction event.
- SSO integration requires protocol expertise (OIDC/SAML) that may not exist on the
  core team at v1.0 — mitigated by using well-maintained libraries (e.g., `openid-client`).

## Consequences

- **Positive:** The platform has a clear, staged path from single-user CLI to enterprise
  SSO. The `Identity` abstraction provides a stable contract for all downstream systems.
  Audit logs gain a real user identity (not just a machine fingerprint). RBAC enforcement
  becomes meaningful when multiple users exist.
- **Negative:** Three auth modes triple the testing matrix. The v0.1→v0.5 migration
  requires operator action (account creation). SSO integration is a significant
  engineering effort for v1.0.
- **Migration path (if applicable):** v0.1 local identities are migrated to token-based
  identities via `agentx auth migrate` command, which creates a user account for the
  local fingerprint and assigns the `owner` role. All existing audit records retain
  their original identity reference.
- **Affected interfaces/contracts:** New `Identity` interface in `packages/auth/`.
  New `IAuthService` interface with three implementations (`LocalAuthService`,
  `TokenAuthService`, `SSOAuthService`). Volume 10's RBAC `PermissionChecker` accepts
  `Identity` instead of a raw user ID. Volume 6 audit records include `identityId`.
