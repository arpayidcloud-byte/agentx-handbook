# RFC-0028: Third-Party Agent Role Proposal Process

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 3

---

## Context

ADR-0002 establishes a fixed 4-agent roster for v0.1 (coding, review, test, security) and
requires a new RFC for any 5th agent role. This ensures clear trust boundaries and scope
control during the initial release.

However, the platform's long-term vision includes an SDK (Volume 7) and a Plugin Platform
(Volume 8) that imply external authorship of extensions. The tension is clear: the fixed
roster ensures security and quality, but the SDK implies openness. Without a formal process
for proposing new agent roles, external contributors face a barrier to entry, and the
platform risks either being too closed (no ecosystem growth) or too open (uncontrolled
agent proliferation).

This RFC defines the process by which external contributors (or internal teams) can
propose new agent roles for inclusion in the platform. The process balances openness
with the security rigor required by ADR-0002's fixed-roster principle.

## Proposed Decision

### Proposal Workflow

A new agent role is proposed through a five-stage process:

#### Stage 1: RFC Submission

The proposer creates an RFC document (using the standard RFC template, Volume 1 Ch. 5)
with the following required sections:

1. **Agent Specification:**
   - Agent name, ID, and description.
   - System prompt (or prompt template with variables).
   - Allowed tools (must be a subset of the existing Tool SDK tools — no new tools
     without a separate RFC).
   - Trust tier classification (per RFC-0027).
   - Example input/output pairs demonstrating the agent's behavior.

2. **Contract Test Suite:**
   - A `contract.test.ts` file (per ADR-0009) covering:
     - At least 5 positive test cases (expected inputs → expected outputs/behaviors).
     - At least 3 negative/edge cases (malformed input, out-of-scope request, tool
       misuse attempt).
     - A test verifying the agent stays within its declared tool allowlist.

3. **Security Review Document:**
   - STRIDE analysis (per RFC-0021 methodology) specific to this agent.
   - Trust boundary diagram showing where this agent sits relative to existing zones.
   - Identified risks and mitigations.

4. **Compatibility Statement:**
   - Confirmation that the agent does not duplicate or conflict with existing agents.
   - If the agent is a specialization of an existing role, explain why a separate
     role is needed vs. a configuration variant.

#### Stage 2: Technical Review

- A designated reviewer from the core team evaluates the RFC for:
  - Correctness of the contract test suite (do the tests actually validate the claims?).
  - Completeness of the security review (are all STRIDE categories addressed?).
  - Compatibility with existing agents and tools.
  - Adherence to Volume 1 style and notation conventions.

- The reviewer may request revisions. This stage is complete when the reviewer
  approves the RFC with no outstanding requests.

#### Stage 3: Security Review

- A second reviewer (must be a different person from Stage 2) evaluates:
  - Threat model accuracy and completeness.
  - Tool allowlist appropriateness (is the agent requesting more tools than it needs?).
  - System prompt for potential prompt-injection vulnerabilities.
  - Whether the agent could be used as a pivot to escalate privileges.

- The security reviewer may approve, request revisions, or reject with rationale.

#### Stage 4: Two-Reviewer Approval

- Both the technical reviewer (Stage 2) and the security reviewer (Stage 3) must
  explicitly approve the RFC.
- Approval is recorded as a comment on the RFC's pull request: `@agentx-review approved`.
- The RFC status is updated to `Accepted`.

#### Stage 5: Integration

- The agent is added to the Agent Registry (Volume 3, Ch. 3) as a new entry.
- The contract test suite is added to `08-Examples/agent-<slug>/contract.test.ts`.
- The agent's tools are registered in the Tool Registry (Volume 7, Ch. 2) if they
  introduce new tools (rare — most agents use existing tools).
- A changelog entry is added noting the new agent role.

### Relationship to ADR-0002

ADR-0002's fixed-roster principle is **preserved and extended**, not weakened:
- v0.1 remains exactly 4 agents. No third-party proposals are accepted before v0.5.
- Starting in v0.5, the proposal process defined in this RFC is the **only** path to
  adding a new agent role. A config-only addition (without RFC) is still prohibited.
- The proposal process's security review stage enforces the same trust-boundary
  rigor that motivated ADR-0002 in the first place.

### Resolution of the SDK Tension

The SDK (Volume 7) enables external authors to write **tools**, not **agents**. This
distinction is critical:
- **Tool SDK:** External authors can create new tools (file parsers, API clients,
  custom integrations). Tools are sandboxed per RFC-0027.
- **Agent roles:** External authors can **propose** new agent roles via this RFC
  process. The bar is higher (RFC + tests + security review + two approvals) because
  an agent role defines a trust boundary, not just a capability.

## Alternatives Considered

### Alternative A: No External Agent Roles — Ever
**Description:** Agent roles are always defined internally. The SDK is for tools only.
  External contributors cannot propose agent roles.

**Pros:**
- Simplest governance model. No review process to design or maintain.
- Complete control over trust boundaries — every agent is written and reviewed by
  the core team.
- Eliminates the "who reviews the reviewers" problem.

**Cons:**
- The platform cannot benefit from domain expertise outside the core team (e.g., a
  DevOps expert proposing an infrastructure agent, a data scientist proposing a
  data-analysis agent).
- Limits ecosystem growth. The platform's value increases with the number of
  available agent roles, but this approach caps that number at the team's capacity.
- Creates a perception of closedness that may discourage SDK adoption.

**Why rejected:** The platform's long-term value proposition includes a rich ecosystem.
  A closed agent roster limits that value. The proposal process provides sufficient
  quality control without being closed.

### Alternative B: Open Agent Registration (No Review)
**Description:** Anyone can register a new agent role via the Plugin Platform. No RFC,
  no security review. Agents are treated like any other plugin.

**Pros:**
- Lowest barrier to entry. Maximum ecosystem growth potential.
- No review bottleneck — agents can be published immediately.
- Consistent treatment of agents and plugins (both are just extensions).

**Cons:**
- Completely undermines ADR-0002's fixed-roster principle.
- No trust-boundary enforcement — a malicious agent could request full tool access
  and exfiltrate data.
- No quality control — agents could be poorly designed, producing incorrect or
  dangerous outputs.
- Makes marketplace trust impossible — no enterprise would install an unvetted agent.

**Why rejected:** This is the opposite of ADR-0002. While the platform aims for
  openness, agent roles have a fundamentally different security profile than tools.
  An agent defines a trust boundary and controls tool access — this cannot be
  delegated to an open registration model.

### Alternative C: RFC + Two-Reviewer Approval Process (Recommended)
**Description:** As described in Proposed Decision. Five-stage process with required
  RFC, contract tests, security review, and two-reviewer approval.

**Pros:**
- Balances openness with rigor. External contributions are welcomed but gated by
  quality and security requirements.
- The required contract test suite ensures behavioral correctness.
- The security review stage enforces the same trust-boundary discipline as ADR-0002.
- Two-reviewer approval prevents a single point of failure in the review process.
- The process is documented and repeatable — every proposal follows the same steps.
- Compatible with ADR-0002: v0.1 is still fixed-roster; the process activates at v0.5.

**Cons:**
- The review process takes time (estimated 2–4 weeks per proposal) — this is
  intentional, not a bug.
- Requires reviewer capacity — if the core team is small, proposals may queue up.
  Mitigated by expanding the reviewer pool as the community grows.
- The RFC requirement may be perceived as heavyweight for simple agent proposals.

## Consequences

- **Positive:** The platform has a clear, fair, and secure process for growing the
  agent ecosystem. External contributors know exactly what is expected. The quality
  and security bar is maintained without being closed to outside expertise.
- **Negative:** The review process creates a bottleneck. Simple agent proposals
  (e.g., a minor variant of an existing role) go through the same rigorous process
  as complex new roles. Reviewer capacity is a scarce resource.
- **Migration path (if applicable):** No migration needed — this is a new process
  that activates at v0.5. The v0.1 fixed roster is unaffected.
- **Affected interfaces/contracts:** Agent Registry (Volume 3, Ch. 3) gains a
  `proposed_via_rfc` field linking to the RFC number. The `08-Examples/` directory
  gains a new subdirectory structure for agent-specific contract tests.
