# RFC-0038: Task Graph Rollback Strategy for Partial Failure

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 2, 5, 6

---

## Context

The agentx workflow engine (Volume 5) executes task graphs: directed acyclic graphs
(DAGs) where nodes represent agent actions (tool calls, file writes, code generation)
and edges represent dependencies. A typical task graph for a "implement feature" workflow
might include: analyze requirements → generate code → write test file → run tests →
create PR.

Task graphs can fail partway through. For example, the code generation step succeeds
and writes files to disk, but the test execution step fails because of a missing
dependency. At this point, the task graph is in a **partially executed** state: some
files have been written, some have not, and the system must decide what to do.

Without a defined rollback strategy, the system's behavior on partial failure is
undefined. The operator is left with an inconsistent state: some files are new, some
are modified, and the relationship between them is unclear. This violates the principle
of predictable failure handling and creates operational risk.

This RFC defines the strategy for handling partial task graph failures.

## Proposed Decision

### Strategy by Release Version

**v0.1: Manual Recovery with Audit Trail**

When a task graph fails partway through:
1. The workflow engine marks the graph as `failed` and records which node failed
   and why.
2. All completed nodes are recorded in the audit log with their inputs and outputs
   (RFC-0024).
3. The workflow engine emits a `TaskGraphFailed` event with:
   - The graph ID.
   - The failed node ID and error message.
   - The list of completed node IDs and their outputs (file paths written, commands
     executed).
   - The list of pending (not yet started) node IDs.
4. The CLI displays a summary to the operator:
   ```
   Task graph failed at step 3/5: "Run tests" — error: missing dependency 'jest'.
   
   Completed steps:
     1. Analyze requirements → output: requirements.md
     2. Generate code → output: src/feature.ts (written)
     3. Write test file → output: src/feature.test.ts (written)
   
   Pending steps:
     4. Run tests (failed)
     5. Create PR (cancelled)
   
   Action required: review the files written by completed steps and decide how to proceed.
   ```
5. **No automatic rollback.** Files written by completed steps remain on disk. The
   operator is responsible for deciding whether to keep, revert, or modify them.
6. The operator can use `agentx task retry <graph-id>` to retry from the failed node
   (skipping already-completed nodes).

**Rationale for v0.1:** Automatic rollback is complex and error-prone. For file
operations, "undoing" a write requires knowing the previous file state (which may not
exist if the file was newly created). For shell commands, "undoing" is often impossible
(e.g., `npm install` cannot be cleanly reversed). Manual recovery with full audit
trail gives the operator the information they need without the risk of incorrect
automatic rollback.

**v1.0: Compensation Transactions for File Operations**

For v1.0, the workflow engine introduces **compensation transactions** — inverse
operations that are registered alongside each task node:

```typescript
interface TaskNode {
  id: string;
  action: () => Promise<TaskResult>;
  compensate?: () => Promise<void>;  // inverse operation for rollback
}

// Example: a file write node registers a compensation that deletes the file
const writeFileNode: TaskNode = {
  id: 'write-feature',
  action: () => fs.writeFile('src/feature.ts', content),
  compensate: () => fs.unlink('src/feature.ts'),  // delete the written file
};
```

When a task graph fails:
1. The workflow engine marks the graph as `failed` and identifies the failed node.
2. The engine traverses the completed nodes **in reverse order** and calls each
   node's `compensate()` function.
3. If a compensation fails (e.g., file already deleted by another process), the
   engine logs the failure and continues compensating remaining nodes.
4. After all compensations are attempted, the engine reports the final state:
   which compensations succeeded and which failed.
5. The operator can review the compensation results and manually clean up any
   remaining artifacts.

**Limitations of compensation transactions:**
- Not all operations have clean inverses. Shell commands that have side effects
  (network requests, package installs) cannot be compensated.
- Compensation is best-effort, not guaranteed. A compensation failure may leave
  the system in a partially rolled-back state.
- The operator must always be informed of the final state — compensation does not
  replace human oversight.

### Audit Trail Requirements

Regardless of the strategy (v0.1 or v1.0), the following audit events are recorded:

| Event | When | Data |
|-------|------|------|
| `task_graph.started` | Graph execution begins | Graph ID, node count |
| `task_node.completed` | Each node finishes | Node ID, output summary |
| `task_node.failed` | A node fails | Node ID, error message |
| `task_graph.failed` | Graph fails | Graph ID, failed node, completed/pending nodes |
| `task_graph.compensated` | Compensations complete (v1.0) | Graph ID, compensation results |

## Alternatives Considered

### Alternative A: No Rollback — Best Effort Execution
**Description:** When a task graph fails, the engine simply stops. No audit trail of
  completed steps, no retry mechanism, no rollback. The operator knows only that
  "something failed" and must manually investigate.

**Pros:**
- Simplest possible implementation. The engine stops on the first error.
- No compensation logic to implement or test.
- No audit overhead for tracking completed steps.

**Cons:**
- The operator has no visibility into what was completed before the failure. They
  must manually inspect the filesystem and git status to understand the state.
- No retry mechanism — the operator must manually re-run the entire workflow from
  scratch, wasting time and API costs.
- No audit trail — violates the principle that security-relevant operations (file
  writes, shell commands) must be logged.
- Unprofessional and frustrating user experience.

**Why rejected:** This is the "do nothing" option. The platform's value proposition
  includes reliability and operator control. Silent failure with no information is
  unacceptable even for v0.1.

### Alternative B: Full Automatic Rollback (Saga Pattern) from Day One
**Description:** Implement the full saga pattern with compensation transactions for
  all task nodes, including shell commands and network operations. The engine
  guarantees that either all nodes complete or all are rolled back.

**Pros:**
- Strongest consistency guarantee: the system is always in a well-defined state
  (either all-done or all-undone).
- Operator does not need to handle partial failures manually.
- Industry-standard pattern for distributed transaction management.

**Cons:**
- Implementing compensation for shell commands is often impossible. A `git push`
  cannot be cleanly undone without knowing the exact state before the push.
- The saga pattern adds significant complexity: compensation registration, ordered
  rollback, compensation failure handling, idempotency guarantees.
- For v0.1 (single operator, local machine), the cost of implementing sagas is
  disproportionate to the risk. The operator has full filesystem access and can
  manually clean up.
- Guaranteed rollback is a false promise — compensation failures mean the
  guarantee is "best effort" in practice.

**Why rejected:** The saga pattern is the right long-term target for file operations
  (v1.0), but implementing it fully for all operation types from v0.1 is premature
  and over-engineered. The phased approach (manual recovery → file compensation)
  provides the right balance.

### Alternative C: Snapshot-and-Restore (VM/Filesystem Snapshot)
**Description:** Before executing a task graph, take a snapshot of the relevant state
  (filesystem, database, git working tree). On failure, restore the snapshot.

**Pros:**
- Guaranteed rollback to the exact pre-execution state — no compensation logic needed.
- Works for any operation type (file writes, shell commands, database changes).
- Simple conceptually: snapshot before, restore after.

**Cons:**
- Filesystem snapshots require OS-level support (LVM, ZFS, btrfs) or virtualization
  (Docker commit, VM snapshot) — not universally available on developer machines.
- Snapshots are expensive: a full filesystem snapshot before every task graph
  adds seconds of latency and significant disk I/O.
- Snapshot size grows with the workspace — a large project's snapshot could be
  gigabytes.
- Snapshot restore is destructive — it reverts *everything*, including changes the
  operator wanted to keep (e.g., files edited outside the task graph during
  execution).
- Git provides a natural "snapshot" via commits — but committing before every
  task graph is a workflow change that may not be desired.

**Why rejected:** Snapshot-and-restore is too heavy for the common case. It adds
  latency, disk usage, and workflow complexity. The git-based approach (commit
  before, reset after) is a viable *optional* feature, but not the default rollback
  strategy.

### Alternative D: Manual Recovery (v0.1) → File Compensation (v1.0) (Recommended)
**Description:** As described in Proposed Decision. v0.1 uses manual recovery with
  full audit trail and retry support. v1.0 adds compensation transactions for
  file operations.

**Pros:**
- v0.1 is implementable with minimal complexity — just audit logging and retry
  logic.
- The audit trail gives operators the information they need for manual recovery.
- The retry mechanism (`agentx task retry`) avoids re-executing completed nodes.
- v1.0's compensation transactions address the most common rollback need (file
  operations) without over-engineering for impossible cases (shell commands).
- The phased approach matches the platform's staged release strategy.
- Constitution Principle 10 (Small Stable Core): compensation is not in the v0.1
  core.

**Cons:**
- v0.1 requires manual operator intervention on partial failure — not fully
  automated.
- Compensation transactions in v1.0 are limited to file operations — shell command
  side effects are still manual.
- The retry mechanism assumes idempotency of completed nodes — if a completed node
  had side effects that changed between the original execution and the retry, the
  retry may produce different results.

## Consequences

- **Positive:** Partial failure handling is defined and predictable. Operators have
  full visibility into what was completed. The retry mechanism reduces re-work.
  The v1.0 compensation path provides automated rollback for the most common case.
- **Negative:** v0.1 requires manual intervention on partial failure. Compensation
  transactions add implementation complexity for v1.0. The audit trail adds storage
  overhead for long-running task graphs.
- **Migration path (if applicable):** v0.1's manual recovery approach is the
  default. v1.0's compensation transactions are opt-in per task graph (the graph
  definition includes `rollback: 'compensate' | 'manual'`). Existing v0.1 task
  graphs default to `manual` rollback.
- **Affected interfaces/contracts:** `TaskNode` interface gains optional `compensate`
  field (v1.0). New CLI commands `agentx task retry <id>`, `agentx task status <id>`.
  New audit events in Volume 6. Workflow engine (Volume 5) gains failure handling
  and retry logic.
