# RFC-0037: Org-Level Configuration Ownership

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 9, 10

---

## Context

As the platform evolves from single-user (v0.1) to multi-user and enterprise (v0.5+),
configuration must be scoped at multiple levels:
- **Local/workspace:** Per-project settings (e.g., which agents are enabled for this
  project, project-specific tool permissions).
- **Organization:** Shared settings across all members of an organization (e.g.,
  approved plugins, SSO configuration, compliance policies).
- **Global/platform:** Settings that apply to all organizations (e.g., available
  provider adapters, system-wide defaults).

The question is: **where does org-level configuration live in the architecture?**

Volume 9 (CLI Platform) owns operator configuration (the `~/.agentx/config.yaml` file,
CLI preferences, approval settings). Volume 10 (Enterprise Platform) owns RBAC,
compliance, and organization membership. Both Volumes have a claim on org-level
configuration, and without a clear ownership decision, configuration logic will be
scattered across both Volumes, creating duplication and inconsistency.

This RFC decides the ownership and the layering model for configuration at different
scopes.

## Proposed Decision

### Ownership Decision: Volume 10 Owns Org-Level Configuration

Volume 10 (Enterprise Platform) is the canonical owner of organization-scoped
configuration. Volume 9 (CLI Platform) owns local/workspace-scoped configuration and
provides the CLI interface for *viewing* org-level config (read access), but does not
*own* or *mutate* org-level config.

### Configuration Layering Model

Configuration is resolved in a strict precedence order (highest to lowest):

```
1. CLI flags (--provider openai --model gpt-4o)
2. Environment variables (AGENTX_PROVIDER=openai)
3. Workspace config (.agentx/config.yaml in the project directory)
4. Organization config (stored in PostgreSQL, owned by Volume 10)
5. Platform defaults (hardcoded in the application)
```

Higher-precedence layers **override** lower-precedence layers. This is standard
precedence for configuration systems (similar to Git's config layering).

### Organization Configuration Schema

Organization configuration is stored in a `org_config` table (Volume 6 schema) and
managed via Volume 10's APIs:

```sql
CREATE TABLE org_config (
  organization_id  TEXT NOT NULL,
  key              TEXT NOT NULL,
  value            JSONB NOT NULL,
  set_by           TEXT NOT NULL,  -- identity ID
  set_at           TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (organization_id, key)
);
```

Example org-level configuration entries:

```json
{
  "approval.timeout_hours": 24,
  "approval.auto_reject": false,
  "plugins.allow_unverified": false,
  "plugins.approved_list": ["@agentx/plugin-jira", "@agentx/plugin-github"],
  "compliance.audit_retention_days": 90,
  "providers.enabled": ["openai", "anthropic"],
  "rbac.default_role": "viewer"
}
```

### Volume 9's Role: Local Override + CLI Interface

Volume 9 retains ownership of:
- `~/.agentx/config.yaml` (operator's personal preferences).
- `.agentx/config.yaml` in the project directory (workspace-level settings).
- CLI flags and environment variable parsing.

Volume 9 provides CLI commands for viewing org-level config (delegating to Volume 10's
API):
- `agentx config show` — displays the effective configuration (all layers merged).
- `agentx config show --org` — displays only the organization layer.
- `agentx config set <key> <value>` — sets a local/workspace config value.
- `agentx config set <key> <value> --org` — sets an org-level config value
  (requires `owner` role, delegated to Volume 10).

### Permission Model for Org Config

- **Viewing org config:** All organization members can view org-level configuration
  (`viewer` role and above).
- **Modifying org config:** Only `owner` role can modify org-level configuration.
  Changes are audit-logged (RFC-0024) and require an approval gate if the key is
  security-sensitive (e.g., `providers.enabled`, `plugins.allow_unverified`).

## Alternatives Considered

### Alternative A: Volume 9 Owns All Configuration
**Description:** Volume 9 (CLI Platform) is the single owner of all configuration
  at all scopes. Org-level config is just another config file or config table
  managed by Volume 9.

**Pros:**
- Single configuration owner — no cross-volume coordination needed.
- Volume 9 already has the config infrastructure (file parsing, env var handling,
  CLI flag parsing).
- Simpler mental model: "all config lives in Volume 9."

**Cons:**
- Volume 9 is named "CLI Platform" — owning org-level config (which is set via
  API in multi-user and SSO contexts) stretches its scope beyond CLI concerns.
- Org-level config requires RBAC enforcement (only owners can modify). This creates
  a dependency from Volume 9 to Volume 10 (RBAC) that violates the intended
  dependency direction.
- Enterprise features (SSO configuration, compliance policies, approved plugin
  lists) are conceptually part of the Enterprise Platform (Volume 10), not the CLI.
- Volume 10 would need to query Volume 9 for org config, creating a circular or
  awkward dependency.

**Why rejected:** Volume 9's scope is CLI/local concerns. Org-level config is an
  enterprise concern that belongs in Volume 10. Forcing it into Volume 9 creates
  scope creep and awkward cross-volume dependencies.

### Alternative B: New Configuration Volume (Volume 16 or similar)
**Description:** Create a dedicated Volume for configuration management at all scopes.
  Neither Volume 9 nor Volume 10 owns config; the new Volume does.

**Pros:**
- Cleanest separation: a single Volume owns all configuration concerns.
- No scope stretching of existing Volumes.
- The new Volume can define a unified configuration API that serves both CLI and
  API consumers.

**Cons:**
- Introduces a new Volume for a concern that is closely related to existing Volumes.
  Volume 9 already handles config parsing; Volume 10 already handles org management.
  A new Volume would need to coordinate with both, potentially creating more
  complexity than it resolves.
- Violates Constitution Principle 10 (Small Stable Core): adding a new Volume for
  configuration management inflates the Volume count for a concern that can be
  handled by clear ownership assignment between existing Volumes.
- The configuration layering model (local overrides org) requires the new Volume
  to understand both local and org concerns, which are architecturally different.

**Why rejected:** A new Volume is disproportionate. The configuration concern is
  well-served by assigning clear ownership: Volume 9 for local, Volume 10 for org.
  The layering model is simple enough to be documented without a dedicated Volume.

### Alternative C: Volume 10 Owns Org Config, Volume 9 Owns Local Config (Recommended)
**Description:** As described in Proposed Decision. Clear ownership split with a
  precedence-based layering model.

**Pros:**
- Each Volume owns the configuration at the scope that matches its domain:
  CLI Platform owns local config, Enterprise Platform owns org config.
- No circular dependencies: Volume 9 reads org config from Volume 10 (unidirectional).
- RBAC enforcement for org config changes is natural — Volume 10 already owns RBAC.
- The layering model (CLI > env > workspace > org > defaults) is standard and
  well-understood.
- Volume 9 provides the CLI interface for all config viewing/editing, delegating
  org config mutations to Volume 10.

**Cons:**
- Configuration is split across two Volumes, requiring a resolver that merges
  layers. This resolver must live somewhere (recommended: a shared
  `ConfigResolver` in a common package).
- Contributors must understand the ownership split to know where to add new
  configuration options.

## Consequences

- **Positive:** Configuration ownership is clear and aligned with Volume domains.
  Org-level config benefits from RBAC enforcement and audit logging. The layering
  model is standard and predictable. Volume 9 remains focused on CLI concerns.
- **Negative:** A `ConfigResolver` must be implemented to merge configuration
  layers. Contributors must learn the ownership split. Org-level config changes
  require Volume 10 API calls (slightly more complex than a local file edit).
- **Migration path (if applicable):** Existing local config (Volume 9) is
  unaffected. Org-level config is a new concept introduced at v0.5 alongside
  multi-user support. The `org_config` table is created as part of the Volume 10
  v0.5 migration.
- **Affected interfaces/contracts:** New `org_config` table in Volume 6. New
  `ConfigResolver` interface in a shared package. Volume 9 CLI commands extended
  with `--org` flag. Volume 10 gains org config CRUD API endpoints.
