# RFC-0030: Disaster Recovery & Backup Strategy

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 6, 11

---

## Context

The agentx platform stores persistent state in PostgreSQL (tasks, agents, audit logs,
role assignments, configuration) and transient state in Redis/BullMQ (job queues, rate
limiters, credential cache). Loss of either data store — through hardware failure, human
error, software bug, or malicious action — would result in operational disruption ranging
from inconvenience (lost job queue state) to critical (lost audit logs, corrupted agent
configurations).

Constitution Principle 7 (Security & Isolation) implies data protection, and Volume 6
(Data Layer) defines the data schema but not the backup and recovery procedures. Without
a documented DR strategy, recovery from a data-loss event is ad hoc, error-prone, and
potentially impossible if backups were never configured.

This RFC defines the Recovery Point Objective (RPO) and Recovery Time Objective (RTO)
for each component, the backup cadence and mechanism, the restore procedure, and the
DR drill schedule.

## Proposed Decision

### RPO/RTO Targets

| Component | RPO | RTO | Justification |
|-----------|-----|-----|---------------|
| PostgreSQL | 5 minutes | 1 hour | Transactional data — some data loss is tolerable but must be minimal. 1-hour restore is achievable from WAL replay. |
| Redis (BullMQ) | 15 minutes | 30 minutes | Job queue state is transient. Lost jobs can be re-enqueued. 30-minute restore from RDB snapshot. |
| Filesystem (workspace) | Not backed up by platform | N/A | Workspace files are the operator's responsibility (version-controlled via git). The platform does not own them. |
| Secrets (EncryptedFile) | Per-rotation | 5 minutes | Secrets are backed up by re-entering them or restoring the encrypted file. RPO equals the rotation interval. |
| Audit Log | 0 (append-only) | 1 hour | Audit logs must not lose data. Chained hashes (RFC-0024) make recovery detectable. |

### PostgreSQL Backup Strategy

**Continuous WAL Archiving:**
- PostgreSQL's Write-Ahead Log (WAL) is archived to a local directory
  (`/var/lib/agentx/backups/wal/`) on a 5-minute cycle.
- WAL archives enable point-in-time recovery (PITR) to any moment within the
  retention window.
- WAL archives are compressed with `gzip` to reduce storage.

**Daily Full Backup:**
- A full base backup is taken daily at 02:00 UTC using `pg_basebackup`.
- Stored as a compressed tarball in `/var/lib/agentx/backups/full/`.
- Retention: 30 daily backups, then weekly backups for 12 weeks.

**Self-Hosted Fallback (ADR-0007):**
- All backup storage is local filesystem — no S3 or cloud storage required for v0.1.
- Backup files are stored outside the PostgreSQL data directory to survive a data
  directory corruption.
- Backup integrity is verified by attempting to restore the backup to a temporary
  directory and running `pg_isready`.

### Redis Backup Strategy

**RDB Snapshots:**
- Redis is configured with `save 900 1` (snapshot after 15 minutes if at least 1 key
  changed).
- RDB files are stored in `/var/lib/agentx/backups/redis/`.
- Retention: 24 snapshots (6 hours of history at 15-minute intervals).

**Job Queue Recovery:**
- On Redis restore, BullMQ automatically detects incomplete jobs and marks them as
  "stalled."
- A recovery job (`agentx jobs recover`) re-enqueues stalled jobs based on their
  original configuration.
- Jobs that were in-progress at the time of failure are marked as `failed` with a
  `recovery_needed` flag, requiring operator review before re-enqueue.

### Restore Procedure

**Step-by-step restore process (documented in Volume 6, Ch. 7):**

1. Stop the agentx process: `systemctl stop agentx` (or `Ctrl+C` for local dev).
2. Stop PostgreSQL: `systemctl stop postgresql`.
3. Restore the base backup: extract the latest `pg_basebackup` tarball to the data
   directory.
4. Replay WAL archives: configure `restore_command` in `postgresql.conf` to replay
   WAL up to the desired point in time.
5. Start PostgreSQL: `systemctl start postgresql`.
6. Verify data integrity: run `agentx audit verify` (RFC-0024) to check the audit
   log hash chain.
7. Restore Redis: copy the latest RDB snapshot to the Redis data directory and
   start Redis.
8. Run `agentx jobs recover` to re-enqueue stalled jobs.
9. Start the agentx process: `systemctl start agentx`.
10. Verify operational readiness: run `agentx health check` and confirm all services
    report healthy.

### DR Drill Schedule

- **Frequency:** Quarterly (every 3 months).
- **Scope:** Full restore from the most recent backup to a clean environment.
- **Success criteria:**
  - PostgreSQL restores to the correct point in time.
  - Audit log hash chain is intact (`agentx audit verify` returns OK).
  - BullMQ recovers stalled jobs correctly.
  - All services start and pass health checks within the RTO window.
- **Documentation:** Each drill produces a runbook entry with timestamps, issues
  encountered, and lessons learned. Failures trigger an RFC for process improvement.

### Multi-Region Considerations (v2.0+)

- PostgreSQL: streaming replication to a standby in a second region. Automatic
  failover via Patroni.
- Redis: Redis Sentinel for automatic failover.
- Backups: cross-region replication of WAL archives and base backups.
- RTO target: 5 minutes (automatic failover vs. manual restore).

## Alternatives Considered

### Alternative A: No Formal DR Strategy — Operator Responsibility
**Description:** The platform does not implement backup or restore tooling. Operators
  are responsible for their own PostgreSQL and Redis backup configuration.

**Pros:**
- Zero implementation effort. The platform ships without DR complexity.
- Operators can choose whatever backup tool they prefer (pgBackRest, Barman, etc.).
- No false sense of security — the platform doesn't claim to handle DR.

**Cons:**
- Most operators will not configure backups until they experience data loss — by
  which point it is too late.
- The platform's audit log (RFC-0024) has an RPO of 0, but without platform-managed
  backups, this guarantee is empty.
- Violates Constitution Principle 7: security-relevant data (audit logs, secrets,
  role assignments) must be protected.
- Enterprise customers require documented DR procedures as part of vendor evaluation.

**Why rejected:** "Operator responsibility" is a valid deployment philosophy for
  infrastructure primitives, but agentx is an application platform that owns
  security-critical data. The platform must provide a baseline DR strategy.

### Alternative B: Cloud-Native Managed Backup (AWS RDS, ElastiCache)
**Description:** Use managed database services with automatic backups. AWS RDS handles
  PostgreSQL backups; ElastiCache handles Redis snapshots. No custom backup code.

**Pros:**
- Managed services handle backup scheduling, retention, and cross-region replication.
- Point-in-time recovery is a checkbox in the AWS console.
- Eliminates backup-related operational burden from the platform team.
- Enterprise-ready with minimal effort.

**Cons:**
- Violates ADR-0007: there is no self-hosted fallback. If the operator runs agentx
  on bare metal or a non-AWS cloud, they lose all backup functionality.
- Introduces a cloud vendor dependency (AWS) that contradicts Constitution Principle 9.
- Managed service costs scale with data volume — potentially expensive for large
  audit log tables.
- Backup data is stored in AWS's infrastructure — an enterprise compliance concern
  for data residency requirements.

**Why rejected:** ADR-0007 is explicit: every managed service must have a self-hosted
  fallback. Starting with cloud-managed backups and retrofitting a self-hosted path
  is harder than starting self-hosted and adding cloud integration later.

### Alternative C: Self-Hosted Backup with Documented Procedures (Recommended)
**Description:** As described in Proposed Decision. WAL archiving, daily base backups,
  RDB snapshots, quarterly DR drills, multi-region roadmap for v2.0.

**Pros:**
- Fully self-hosted — no external service dependency (ADR-0007 compliant).
- Portable across any PostgreSQL and Redis deployment (Constitution Principle 9).
- DR drills ensure the procedures actually work, not just look good on paper.
- Clear RPO/RTO targets provide measurable SLAs.
- The quarterly drill cadence catches configuration drift and procedure decay.

**Cons:**
- Requires operational discipline: backup schedules must be maintained, disk space
  monitored, drills executed.
- WAL archiving consumes disk space (~1GB/day for moderate traffic).
- The restore procedure is manual — no automated one-click recovery for v0.1.
- DR drills are time-consuming (~2–4 hours per drill including documentation).

## Consequences

- **Positive:** The platform has a concrete, tested DR capability. RPO/RTO targets
  are defined and measurable. Quarterly drills ensure the procedures remain valid.
  The self-hosted approach satisfies ADR-0007 and Constitution Principle 9.
- **Negative:** Operational overhead for backup maintenance and DR drills. Disk
  space consumption for WAL archives and RDB snapshots. Manual restore procedure
  is slower than automated failover (acceptable for v0.1).
- **Migration path (if applicable):** Existing deployments must configure the backup
  directories and cron jobs. A one-time setup script (`agentx setup backups`) creates
  the directory structure, configures WAL archiving in `postgresql.conf`, and adds
  the cron entries.
- **Affected interfaces/contracts:** New CLI commands: `agentx backup now`,
  `agentx backup status`, `agentx restore <timestamp>`, `agentx jobs recover`.
  New `agentx setup backups` configuration command. New `agentx health check`
  command includes backup freshness verification.
