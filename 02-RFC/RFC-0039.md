# RFC-0039: Dashboard Scope Decision

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 10

---

## Context

The agentx platform needs a web-based dashboard for operators and administrators to:
- Monitor task execution status, agent activity, and system health.
- View observability data (metrics, traces, logs from Volume 13).
- Manage organization settings, users, and RBAC (Volume 10).
- Review approval requests (Volume 9, RFC-0034).
- Browse the plugin marketplace (Volume 8, RFC-0036, v2.0).

The question is: **where does the dashboard live in the architecture?** Three options
exist:

1. A new Volume (Volume 17: Dashboard) with its own specification.
2. A sub-section of Volume 10 (Enterprise Platform), since the dashboard is primarily
   an enterprise/administrative concern.
3. A separate application (`apps/dashboard/`) in the monorepo without a dedicated
   Volume — governed by existing Volumes' APIs.

This decision has architectural significance because a new Volume is a first-class
architectural entity with a specification, contract tests, and an approval gate. Adding
a Volume for the dashboard would set a precedent: is every frontend application a
Volume? If so, the Volume count will proliferate.

## Proposed Decision

### The Dashboard Is a Separate Application Governed by Existing Volume APIs

The dashboard is implemented as `apps/dashboard/` in the monorepo — a Next.js
application that consumes the platform's existing APIs. It is **not** a new Volume and
**not** a sub-section of Volume 10.

### Rationale

1. **The dashboard is a consumer, not a domain.** Volumes define domains (Agent
   Platform, Tool SDK, Workflow Engine). The dashboard is a UI layer that
   *consumes* these domains — it displays data from Volume 13 (observability),
   manages entities from Volume 10 (RBAC), and shows task status from Volume 5
   (workflow engine). It does not define a new domain.

2. **A dashboard Volume would be a "UI Volume,"** which is a category that does not
   exist and would dilute the Volume concept. Volumes are about architecture, not
   presentation.

3. **Volume 10 is about enterprise features (RBAC, SSO, compliance),** not about
   UI. Adding a dashboard sub-section to Volume 10 would conflate "what the
   enterprise platform provides" with "how it's displayed."

4. **As a separate app, the dashboard** follows the same pattern as any other
   consumer of the platform's APIs. It authenticates via the same mechanism
   (RFC-0025), respects the same RBAC permissions (RFC-0012), and uses the same
   observability data (Volume 13). No special architectural treatment is needed.

### Governance Model

The dashboard application is governed by:

| Concern | Governing Volume | Mechanism |
|---------|-----------------|-----------|
| Authentication | Volume 15 (Identity) | Dashboard uses TokenAuthService (RFC-0025 Mode 2) |
| Authorization | Volume 10 (RBAC) | Dashboard calls PermissionChecker via API; UI hides features user can't access |
| Data | Volumes 5, 6, 13 | Dashboard consumes REST/GraphQL APIs |
| Approval gates | Volume 9 | Dashboard renders approval requests from the notification channel (RFC-0034) |
| Observability of dashboard itself | Volume 13 | Dashboard emits its own metrics, traces, logs |
| Quality | Volume 14 | Dashboard has its own contract tests and E2E tests |

### Technical Stack

- **Framework:** Next.js 14+ (App Router, Server Components).
- **Styling:** Tailwind CSS (consistent with the platform's frontend stack).
- **State management:** React Query for server state, Zustand for client state.
- **API consumption:** OpenAPI-generated TypeScript client from the platform's
  REST API specification.
- **Deployment:** Served by the agentx server (NestJS serves the Next.js static
  export) or as a standalone deployment behind a reverse proxy.

### Feature Phasing

| Feature | Version |
|---------|---------|
| System health overview (service status, uptime) | v0.5 |
| Task graph visualization and status | v0.5 |
| Agent activity log (recent actions per agent) | v0.5 |
| RBAC management (user/role/permission CRUD) | v1.0 |
| Approval request review and response | v1.0 |
| Observability dashboards (metrics, traces) | v1.0 |
| Plugin marketplace browsing and installation | v2.0 |
| Compliance and audit log viewer | v2.0 |

## Alternatives Considered

### Alternative A: New Volume 17 (Dashboard)
**Description:** Create Volume 17: Dashboard as a first-class architectural Volume
  with its own specification, contract tests, and approval gate.

**Pros:**
- The dashboard gets the same architectural rigor as any other domain.
- A Volume document provides a specification for the dashboard's features and
  behavior.
- Consistent treatment — if the dashboard is important enough to build, it's
  important enough to specify.

**Cons:**
- Sets a precedent that every UI application is a Volume. Future UI apps (mobile
  app, CLI TUI, VS Code extension) would also need Volumes, inflating the Volume
  count.
- The dashboard does not define a new domain — it consumes existing domains. A
  Volume for a consumer is architecturally incorrect.
- The dashboard's features are entirely derived from existing Volume APIs. A
  Volume specification would be redundant — it would restate what Volume 10, 13,
  and 5 already specify.
- Violates Constitution Principle 10 (Small Stable Core): adding a Volume for a
  UI layer expands the core beyond its domain-architecture purpose.

**Why rejected:** The dashboard is a UI consumer, not a domain. Treating it as a
  Volume dilutes the Volume concept and sets a problematic precedent.

### Alternative B: Sub-Section of Volume 10 (Enterprise Platform)
**Description:** The dashboard is specified as a chapter or sub-section within
  Volume 10, since it primarily serves enterprise/administrative use cases.

**Pros:**
- No new Volume — the dashboard is "owned" by the existing enterprise Volume.
- The dashboard features (RBAC management, org settings, compliance) align with
  Volume 10's scope.
- Simpler governance — one fewer Volume to manage.

**Cons:**
- Volume 10's scope expands significantly — it now covers RBAC, SSO, compliance,
  *and* a full frontend application. This makes the Volume document unwieldy.
- Not all dashboard features are enterprise-specific. Task graph visualization
  (Volume 5) and observability dashboards (Volume 13) are useful for single-user
  v0.5 as well.
- Creating a precedent that Volumes can include UI specifications. Other Volumes
  might start including UI sub-sections, further blurring the Volume concept.

**Why rejected:** Volume 10's scope is already substantial. Adding a full dashboard
  specification overloads it. The dashboard is a cross-cutting UI that consumes
  multiple Volumes, not an extension of Volume 10.

### Alternative C: Separate Application with Cross-Volume Governance (Recommended)
**Description:** As described in Proposed Decision. `apps/dashboard/` in the
  monorepo, consuming existing Volume APIs, governed by the cross-cutting
  architectural standards.

**Pros:**
- The dashboard is treated as a consumer, which is architecturally correct.
- No new Volume, no Volume scope expansion.
- The dashboard can be developed, deployed, and versioned independently of the
  platform Volumes.
- Feature phasing is natural: the dashboard gains features as the underlying
  Volume APIs become available.
- The governance model (which Volume governs which concern) is explicit and
  documented.
- Follows Constitution Principle 10: the core Volumes are stable; the dashboard
  is a peripheral application.

**Cons:**
- Without a Volume document, the dashboard's specification lives in its own README
  and design documents, which may be less visible than a Volume in the handbook.
  Mitigated by referencing the dashboard from Volume 1's architecture overview.
- The dashboard's quality gates (contract tests, E2E tests) must be defined
  outside the Volume approval process. Mitigated by applying the same quality
  standards via the CI pipeline.

## Consequences

- **Positive:** The Volume concept remains focused on domain architecture. The
  dashboard is developed as a standard application consuming well-defined APIs.
  No precedent is set for UI-specific Volumes. Feature phasing is aligned with
  underlying API availability.
- **Negative:** The dashboard lacks a formal Volume-level specification. Its
  governance relies on referencing multiple Volumes, which requires contributors
  to understand the cross-cutting model.
- **Migration path (if applicable):** No migration — this is a greenfield decision.
  The dashboard is implemented as `apps/dashboard/` from the start.
- **Affected interfaces/contracts:** No changes to existing interfaces. The
  dashboard consumes existing REST/GraphQL APIs. New APIs are added to serve
  dashboard needs as features are phased in.
