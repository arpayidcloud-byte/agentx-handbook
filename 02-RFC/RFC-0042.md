# RFC-0042: Developer Experience Workstream Scope

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** Cross-cutting

---

## Context

"Developer Experience" (DX) encompasses everything that affects a developer's
productivity and satisfaction when using or contributing to the agentx platform:
local development setup, error messages, documentation, debugging tools, IDE
integration, onboarding, and more.

As the platform grows, DX concerns accumulate across multiple workstreams:
- The CLI Platform (Volume 9) affects DX through its command-line interface design.
- The Provider Platform (Volume 4) affects DX through its error messages and
  configuration.
- The Plugin Platform (Volume 8) affects DX through its SDK and manifest format.
- The Quality Assurance Volume (Volume 14) affects DX through CI speed and
  error reporting.
- The Governance documents (Volume 0, Volume 1) affect DX through contributing
  guidelines and documentation quality.

The question is: **should Developer Experience be a standalone workstream (with its
own roadmap, team allocation, and success metrics) or a cross-cutting practice
embedded in every workstream?**

Making DX a standalone workstream risks creating a centralized DX team that becomes
a bottleneck and is disconnected from domain-specific context. Making DX purely
cross-cutting risks it being deprioritized in every workstream because "feature
work is more important."

This RFC decides the organizational and architectural scope of Developer Experience.

## Proposed Decision

### Developer Experience Is a Cross-Cutting Practice, Not a Standalone Workstream

DX is **not** a standalone workstream with its own roadmap, dedicated team, or
separate backlog. Instead, DX is a **cross-cutting practice** that every workstream
is responsible for. Each workstream's definition of done includes a DX checklist.

### DX Checklist (Per Workstream)

Every workstream (every Volume, every feature PR, every release) must satisfy the
following DX checklist before merging:

1. **Local Development Setup < 5 Minutes:**
   - A new contributor can clone the repo, run `npm install && npm run dev`, and
     have a working local environment in under 5 minutes.
   - No manual steps (e.g., "install this global package," "set up this database
     manually") unless documented and automated in the setup script.
   - `npm run dev` starts all required services (API, CLI, database, Redis) via
     docker-compose or process manager.

2. **Clear Error Messages:**
   - Every public method's error messages are actionable: they describe what went
     wrong, why, and what the operator should do to fix it.
   - Error messages must not leak internal implementation details (stack traces
     are logged, not returned to the user).
   - Error codes are machine-readable (e.g., `PROVIDER_TIMEOUT`, `SECRET_NOT_FOUND`)
     to enable programmatic error handling.
   - Example:
     ```
     Error: Secret not found: provider.anthropic.api_key.
     Fix: Run 'agentx secrets set provider.anthropic.api_key <value>' or set the
     ANTHROPIC_API_KEY environment variable.
     Code: SECRET_NOT_FOUND
     ```

3. **Debug Logging:**
   - Every public method logs its entry, exit, and duration at `debug` level.
   - Structured log format (JSON) with consistent fields: `timestamp`, `level`,
     `module`, `method`, `requestId`, `durationMs`.
   - Debug logging is off by default and enabled via `AGENTX_LOG_LEVEL=debug`.
   - No secrets in debug logs (per RFC-0023's no-logging guarantee).

4. **Documentation:**
   - Every public interface has a JSDoc comment with a one-line description,
     parameter descriptions, return type, and an example.
   - Every CLI command has a `--help` output that is sufficient for use without
     reading the full documentation.
   - Every configuration option is documented in the Volume's configuration chapter
     with its type, default value, and description.

5. **TypeScript Strict Mode:**
   - All packages compile with `strict: true` in `tsconfig.json`.
   - No `any` types in public interfaces (internal `any` is acceptable with a
     `// TODO: narrow this type` comment).
   - All public APIs have complete type definitions — no "will be typed later."

### DX Advocate Role

While DX is not a standalone workstream, each release has a **DX Advocate** — a
rotating role (one person per release cycle) responsible for:
- Reviewing PRs for DX checklist compliance.
- Running the "new contributor" test: setting up the environment from scratch
  on a clean machine and timing the setup.
- Maintaining the DX metrics dashboard (see below).
- Escalating systemic DX issues to the Architecture Review Board.

The DX Advocate is a part-time role (estimated: 2–4 hours per week) assigned to
a different team member each release cycle. This rotation ensures that DX
awareness is distributed, not concentrated.

### DX Metrics

The following metrics are tracked and reviewed at each release retrospective:

| Metric | Target | Measurement |
|--------|--------|-------------|
| `dev_setup_time` | < 5 minutes | Time from `git clone` to `npm run dev` success on a clean machine |
| `error_message_actionability` | > 90% | Percentage of public error messages that include a suggested fix |
| `ts_strict_violations` | 0 | Number of `any` types in public interfaces |
| `doc_coverage` | > 80% | Percentage of public interfaces with JSDoc comments |
| `new_contributor_success_rate` | > 80% | Percentage of new contributors who complete their first PR within 2 weeks |

Metrics are collected by the DX Advocate and published in the release retrospective
document. Trends are tracked over time — a declining metric triggers an RFC or ADR
to address the root cause.

### What This Means for Each Workstream

| Workstream | DX Responsibility |
|-----------|-------------------|
| Core Runtime (Volume 2) | Fast startup, clear bootstrap errors, hot-reload support |
| Agent Platform (Volume 3) | Clear agent error messages, useful debug logs for prompt/execution |
| Provider Platform (Volume 4) | Actionable provider error messages, helpful configuration validation |
| Tool SDK (Volume 7) | Clear permission denied messages, tool execution diagnostics |
| Plugin Platform (Volume 8) | Plugin development guide, manifest validation with helpful errors |
| CLI Platform (Volume 9) | Consistent command interface, helpful `--help`, shell completion |
| Enterprise Platform (Volume 10) | Clear RBAC denial messages, useful SSO configuration errors |
| Observability (Volume 13) | Useful debug logging, clear metric/trace interpretation |
| Quality Assurance (Volume 14) | Fast CI, clear test failure messages, actionable lint errors |

## Alternatives Considered

### Alternative A: Standalone DX Workstream
**Description:** Create a dedicated DX workstream with its own roadmap, team
  allocation, and success metrics. A DX team is responsible for improving
  developer experience across the platform.

**Pros:**
- DX has a clear owner and dedicated resources — it is not deprioritized.
- The DX team can develop deep expertise in DX best practices and tooling.
- Consistent DX vision across the platform — a single team ensures coherence.
- DX improvements can be prioritized independently of feature work.

**Cons:**
- The DX team becomes a bottleneck — every workstream must wait for the DX team
  to review and approve their DX work.
- The DX team lacks domain-specific context. An error message for the Provider
  Platform requires understanding of LLM API semantics that a generalist DX team
  may not have.
- Creates an "us vs. them" dynamic: feature teams feel micromanaged by the DX
  team, and the DX team feels ignored.
- Violates Constitution Principle 10 (Small Stable Core): a dedicated DX team
  is an organizational overhead that the platform's size does not justify.
- The DX team's roadmap competes with feature roadmaps for engineering capacity,
  creating organizational conflict.

**Why rejected:** A standalone DX workstream is appropriate for large organizations
  (50+ engineers) but is disproportionate for the agentx platform's current and
  near-term scale. The cross-cutting approach distributes DX responsibility
  without creating a bottleneck.

### Alternative B: No Formal DX Practice — Let It Emerge Organically
**Description:** Do not define DX standards, checklists, or metrics. Let each
  workstream handle DX as they see fit. DX improves organically as pain points
  are discovered and fixed.

**Pros:**
- No overhead. No checklists, no metrics, no DX Advocate role.
- Each workstream optimizes for its own DX needs without a one-size-fits-all
  standard.
- Avoids "DX theater" — improvements driven by real pain, not checkboxes.

**Cons:**
- DX consistently loses to feature work in prioritization. Without a checklist,
  "clear error messages" is always "we'll fix it later."
- Inconsistent DX across workstreams: one workstream has excellent error messages,
  another has cryptic ones. The platform's overall DX is determined by its
  weakest link.
- No metrics mean no accountability. DX regressions go undetected.
- New contributors have a fragmented experience — some parts of the platform are
  well-documented, others are not.

**Why rejected:** Organic DX improvement has been demonstrated to fail in practice.
  Without explicit standards and accountability, DX degrades over time as feature
  pressure mounts. The checklist approach provides a minimum bar without
  over-constraining workstreams.

### Alternative C: Cross-Cutting Practice with DX Checklist (Recommended)
**Description:** As described in Proposed Decision. DX is everyone's responsibility,
  enforced via a per-workstream checklist, with a rotating DX Advocate and tracked
  metrics.

**Pros:**
- DX is embedded in every workstream — no bottleneck, no context gap.
- The checklist provides a clear, objective minimum bar.
- The rotating DX Advocate distributes DX awareness without creating a permanent
  role.
- Metrics provide accountability and trend visibility.
- Consistent with Constitution Principle 10: no new team or workstream, just a
  practice.
- The checklist is lightweight (5 items) and achievable without significant
  overhead.

**Cons:**
- The checklist adds items to the definition of done for every PR — slight
  increase in review time.
- The DX Advocate role is a new responsibility that team members must learn and
  perform.
- Metrics require ongoing collection and reporting effort.
- Without a dedicated team, there is a risk that the checklist becomes a "tick
  the box" exercise rather than a genuine quality improvement.

## Consequences

- **Positive:** Every workstream owns its DX. The checklist provides a clear
  minimum bar. The rotating DX Advocate ensures awareness is distributed. Metrics
  provide accountability and enable continuous improvement. No new team or
  workstream is created — the organizational overhead is minimal.
- **Negative:** The checklist adds items to every PR's definition of done. The
  DX Advocate role requires 2–4 hours per week from a team member. Metrics
  collection is an ongoing effort. There is a risk of "checkbox compliance"
  rather than genuine DX improvement.
- **Migration path (if applicable):** No migration — this is an organizational
  practice. The checklist and DX Advocate role are introduced at the next
  sprint planning. Existing PRs are not retroactively required to meet the
  checklist.
- **Affected interfaces/contracts:** No code changes. This RFC affects process
  and organization, not interfaces. The DX checklist is added to CONTRIBUTING.md
  and the PR template.
