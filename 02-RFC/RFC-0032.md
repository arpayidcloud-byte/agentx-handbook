# RFC-0032: Agent System Prompt Versioning & Regression Testing

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 3, 14

---

## Context

Agent system prompts are the primary mechanism for controlling agent behavior. Each agent
(coding, review, test, security per ADR-0002) has a system prompt that defines its role,
constraints, output format, and tool usage guidelines. These prompts are stored as
markdown files in the repository (Volume 6, Prompts directory).

A system prompt change can have subtle, far-reaching effects. A seemingly minor wording
tweak can:
- Cause the agent to use a tool it previously avoided (security implication).
- Change the output format, breaking downstream parsers.
- Alter the agent's tone or verbosity, affecting user experience.
- Introduce or remove constraints that affect correctness (e.g., "always run tests"
  vs. "run tests when appropriate").

Currently, system prompt changes are not versioned, not regression-tested, and not
tracked against provider model changes. A prompt that works well with GPT-4o may produce
different behavior with Claude 3.5 Sonnet — and there is no automated way to detect this
drift.

This RFC defines a versioning scheme, storage format, and regression testing strategy
for agent system prompts.

## Proposed Decision

### Prompt Versioning (SemVer)

System prompts follow Semantic Versioning, independent of the platform version:

- **MAJOR:** Fundamental behavior change (e.g., agent role redefinition, new output
  format, added constraints). Consumers must review and potentially update their
  integration code.
- **MINOR:** Additive change (e.g., additional examples, new edge-case handling,
  supplementary instructions). Existing behavior is preserved.
- **PATCH:** Clarification or typo fix (e.g., rewording for clarity, fixing a
  grammatical error). No behavioral change intended.

Version is tracked in two places:
1. **In the prompt file header:**
   ```markdown
   <!--
   Agent: coding
   Version: 1.2.0
   Last Modified: 2026-07-13
   Modified By: @jane
   Change Description: Added instruction to prefer named exports over default exports.
   -->
   ```
2. **In a prompt registry database table:**
   ```sql
   CREATE TABLE prompt_version (
     agent_id       TEXT NOT NULL,
     version        TEXT NOT NULL,  -- SemVer string
     content_hash   TEXT NOT NULL,  -- SHA-256 of prompt content
     content        TEXT NOT NULL,  -- Full prompt text
     change_desc    TEXT,
     created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
     PRIMARY KEY (agent_id, version)
   );
   ```

### Prompt Registry

A `PromptRegistry` service manages prompt versions:

```typescript
interface PromptRegistry {
  /** Get the current (latest) prompt for an agent. */
  getCurrent(agentId: string): Promise<PromptVersion>;

  /** Get a specific version of an agent's prompt. */
  getVersion(agentId: string, version: string): Promise<PromptVersion>;

  /** Register a new prompt version. Validates SemVer ordering. */
  register(agentId: string, version: string, content: string, changeDesc: string): Promise<void>;

  /** List all versions for an agent. */
  listVersions(agentId: string): Promise<PromptVersion[]>;
}
```

The registry enforces SemVer ordering: a new version must be greater than the current
latest version for that agent. Attempting to register an out-of-order version throws
`InvalidVersionError`.

### Regression Testing

**Test Fixtures:**

For each agent, a test fixture file defines input/output pairs that must remain stable
across prompt versions:

```typescript
// 08-Examples/agent-coding/prompt-regression.test.ts
describe('Coding Agent Prompt Regression', () => {
  const fixtures = loadPromptFixtures('agent-coding');

  for (const fixture of fixtures) {
    it(`fixture: ${fixture.name}`, async () => {
      const prompt = await promptRegistry.getCurrent('coding');
      const result = await evaluatePrompt(prompt, fixture.input, fixture.model);
      expect(result.output).toMatchPromptFixture(fixture.expectedOutput);
    });
  }
});
```

Each fixture contains:
- `name`: Human-readable description.
- `input`: A representative task description.
- `expectedOutput`: The expected agent behavior (not exact string match — a structural
  and semantic comparison that allows for LLM non-determinism).
- `model`: The provider model used for evaluation.
- `assertions`: Specific checks (e.g., "must call fs.read tool", "must not call
  shell.exec", "output must include test file path").

**Evaluation Strategy:**

Prompt regression tests are **not** run on every commit — LLM API calls are slow and
expensive. Instead:

1. **On prompt change PR:** The CI pipeline detects modifications to prompt files and
   triggers the regression test suite for the affected agent. This is a manual trigger
   (a reviewer comments `@agentx-bot run prompt-regression`) or an opt-in GitHub
   Actions workflow dispatch.

2. **On provider model update:** When a provider releases a new model version (e.g.,
   GPT-4o → GPT-4o-2026-07), the regression suite is run against the new model and
   compared against the baseline (recorded against the previous model). Significant
   behavioral drift is flagged for review.

3. **Weekly scheduled run:** The full regression suite runs weekly against all
   supported provider models. Results are compared to the previous week's baseline.

**Non-Determinism Handling:**

LLM outputs are non-deterministic. The regression framework handles this by:
- Running each fixture **3 times** and requiring at least **2/3** to pass (majority
  vote).
- Using `temperature: 0` for regression evaluations to minimize randomness.
- Comparing structural properties (tool calls made, output format, key phrases present)
  rather than exact string matches.

## Alternatives Considered

### Alternative A: No Versioning — Prompts Are Just Files
**Description:** Prompts remain unversioned markdown files. Changes are tracked via
  git history. No registry, no regression tests.

**Pros:**
- Zero implementation cost. Prompts are edited like any other text file.
- Git history provides a change log (though not SemVer-structured).
- No test infrastructure to build or maintain.

**Cons:**
- No way to programmatically track which prompt version is in use.
- No regression testing — a prompt change that breaks agent behavior is discovered
  only in production (or manual testing).
- No structured change tracking — git history is noisy and not queryable.
- Provider model changes can silently break agent behavior with no detection.

**Why rejected:** The prompt is the most critical control surface for agent behavior.
  Treating it as "just a file" with no versioning or testing is equivalent to shipping
  code without version control or tests.

### Alternative B: Embed Prompts in Code (String Literals)
**Description:** Prompts are TypeScript string literals in the agent implementation
  files. Versioned as part of the code.

**Pros:**
- Prompts are co-located with the code that uses them — easy to find and modify.
- No separate storage or registry needed.
- Refactoring tools (IDE rename, find usages) work naturally.

**Cons:**
- Prompts are long (often 50–200 lines) — embedding them in TypeScript files makes
  the code unreadable.
- String literal escaping is error-prone (backticks, template literals, special chars).
- No separate versioning — prompt changes are tied to code changes, making it
  impossible to update a prompt without deploying new code.
- No prompt-only regression testing — testing a prompt change requires deploying the
  entire application.
- The existing convention (markdown files in `06-Prompts/`) would need to be
  uprooted and migrated.

**Why rejected:** The existing file-based prompt storage is a better separation of
  concerns. Code owns logic; files own prompts. Embedding prompts in code conflates
  these concerns and makes prompt iteration slower.

### Alternative C: SemVer-Tagged Files with Regression Testing (Recommended)
**Description:** As described in Proposed Decision. File-based storage with headers,
  database-backed registry, and fixture-based regression testing.

**Pros:**
- Prompts remain human-editable markdown files — low barrier for prompt engineers.
- SemVer provides clear, machine-readable versioning that is industry-standard.
- The registry enables programmatic access to current and historical prompt versions.
- Regression testing catches prompt changes that alter agent behavior.
- The majority-vote approach handles LLM non-determinism pragmatically.
- The weekly scheduled run provides ongoing drift detection.

**Cons:**
- Maintaining the registry requires a new database table and service.
- Regression tests require LLM API calls — cost and latency (mitigated by on-demand
  and weekly scheduling, not per-commit).
- Fixture maintenance is ongoing — as agent behavior evolves, fixtures must be updated
  to match the new expected behavior (this is a feature, not a bug — fixture updates
  are intentional and reviewed).
- Non-determinism means regression tests are not perfectly reliable (3 runs, 2/3
  majority mitigates but does not eliminate this).

## Consequences

- **Positive:** Prompt changes are tracked, versioned, and regression-tested. Provider
  model updates are detected before they cause production issues. The prompt registry
  provides a single source of truth for which prompt version each agent is using.
- **Negative:** The registry adds schema and service complexity. Regression tests have
  an ongoing API cost (estimated $5–20/week for weekly full-suite runs). Fixture
  maintenance is a new ongoing responsibility for prompt engineers.
- **Migration path (if applicable):** Existing prompt files are backfilled with version
  headers (v1.0.0) and registered in the prompt_version table. Initial fixture set
  is created by running each agent against a representative task and recording the
  output as the baseline.
- **Affected interfaces/contracts:** New `PromptRegistry` service in Volume 3.
  New `prompt_version` table in Volume 6. New regression test framework in Volume 14.
  Prompt file headers standardized across all agent prompts.
