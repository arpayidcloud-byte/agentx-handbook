# RFC-0033: Observability Instrumentation Requirement for Volume Approval

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 13, 14, 1

---

## Context

Volume 13 (Observability) defines the platform's approach to metrics, traces, and logs.
ADR-0009 establishes a precedent: a Volume cannot be marked Approved unless it has a
corresponding contract-test template covering its functional requirements.

This RFC extends the ADR-0009 pattern to observability: a Volume cannot reach Approved
status unless it specifies the **instrumentation requirements** that its implementation
must satisfy. Without this gate, Volumes can be Approved with vague or absent
observability specifications, leading to an unmonitored platform where failures are
invisible until users report them.

The specific concern is that observability is often treated as an afterthought —
something "added later" after the functional implementation is complete. By making
instrumentation requirements a prerequisite for Volume approval, observability becomes
a first-class architectural concern, not a retrofit.

## Proposed Decision

### Minimum Instrumentation Specification

Every Volume seeking Approved status must include an "Observability Requirements"
section (or a separate observability sub-chapter) that specifies at minimum:

1. **Metrics (at least one histogram per public method):**
   - Each public method in the Volume's API surface must have a corresponding
     histogram metric measuring execution duration.
   - Metric name: `<volume-slug>.<method-name>.duration_ms`.
   - Labels: `status` (success/error), `provider` (if provider-specific).
   - Example: `agent-platform.execute.duration_ms{status="success",provider="openai"}`.

2. **Counters (at least one counter per published event):**
   - Each event published to the event bus (Volume 5) must have a corresponding
     counter metric.
   - Metric name: `<volume-slug>.<event-name>.total`.
   - Labels: match the event's key attributes.
   - Example: `workflow-engine.task_completed.total{agent="coding",status="success"}`.

3. **Trace Spans (at least one span per cross-module call):**
   - Each call from this Volume to another Volume must create a trace span.
   - Span name: `<volume-slug>.<method-name>`.
   - Span attributes: include input parameters (sanitized, no secrets per
     RFC-0023) and output status.
   - Example: A call from Volume 3 (Agent Platform) to Volume 4 (Provider
     Platform) creates a span `agent-platform.call_provider` with attributes
     `provider_id`, `model_id`, `token_count`.

4. **Log Patterns (structured logging for all error paths):**
   - Every error path in the Volume must emit a structured log entry.
   - Log level: `error` for unexpected errors, `warn` for expected degradation.
   - Required fields: `error.message`, `error.stack`, `error.code`, `volume`,
     `method`, `requestId` (for correlation with traces).
   - Example:
     ```json
     {
       "level": "error",
       "message": "Provider request failed after 3 retries",
       "error": { "code": "PROVIDER_TIMEOUT", "message": "...", "stack": "..." },
       "volume": "provider-platform",
       "method": "complete",
       "requestId": "abc-123",
       "timestamp": "2026-07-13T10:30:00Z"
     }
     ```

### Approval Gate Mechanism

The Volume approval checklist (Volume 1, Ch. 6) gains a new mandatory item:

> **[ ] Observability Requirements:** The Volume specifies all required metrics,
> counters, trace spans, and log patterns in an "Observability Requirements" section.

A Volume PR seeking Approved status is blocked in CI if the observability section
is missing or does not cover all public methods and events. The check is a
markdown-lint rule similar to RFC-0031's content bar enforcement:

```yaml
observability-lint:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - name: Check observability requirements
      run: npx obs-lint-handbook --volume ${{ github.event.pull_request.title }}
```

The linter parses the Volume document, extracts the list of public methods and
events from interface blocks and event bus references, and verifies that each has
a corresponding metric/counter/span specification in the observability section.

### Relationship to ADR-0009

This RFC mirrors ADR-0009's pattern:

| Aspect | ADR-0009 (Contract Tests) | This RFC (Observability) |
|--------|--------------------------|--------------------------|
| Gate | Contract-test template required | Instrumentation spec required |
| Location | `08-Examples/<slug>/contract.test.ts` | Volume document, "Observability Requirements" section |
| Enforcement | CI check for file existence | CI check for section completeness |
| Minimum content | Covers functional requirements + edge cases | Covers metrics + counters + spans + logs |

Both gates must be satisfied for a Volume to reach Approved status. They are
independent checks — a Volume can pass one and fail the other.

## Alternatives Considered

### Alternative A: No Observability Gate — Trust Implementation Teams
**Description:** Observability is left to the discretion of the implementation team.
  No formal requirement for Volume approval.

**Pros:**
- No additional documentation burden for Volume authors.
- Implementation teams can choose the right level of instrumentation for their
  specific use case.
- Avoids "metric theater" — metrics added just to satisfy a checklist rather than
  for operational value.

**Cons:**
- In practice, observability is consistently deprioritized without a gate. The
  "added later" promise is rarely fulfilled.
- Unmonitored Volumes are a reliability risk — failures are invisible until they
  cause user-facing incidents.
- Inconsistent instrumentation across Volumes makes cross-Volume correlation
  impossible.
- ADR-0009 established the precedent that "required for approval" is an effective
  enforcement mechanism — not using it for observability is inconsistent.

**Why rejected:** Trust without verification has been demonstrated to fail for
  quality (the reason ADR-0009 exists). The same logic applies to observability.

### Alternative B: Centralized Observability Team Adds Instrumentation After Implementation
**Description:** A dedicated observability team reviews each Volume's implementation
  after it is Approved and adds instrumentation as needed.

**Pros:**
- Volume authors focus on functionality; observability experts handle instrumentation.
- Potentially higher quality instrumentation (experts vs. generalists).
- No gate delays Volume approval.

**Cons:**
- Adds a serial dependency: Volume implementation → observability review →
  instrumentation → deployment. This slows the overall delivery pipeline.
- The observability team becomes a bottleneck (a single team reviewing all Volumes).
- Instrumentation added after the fact often misses context that the original
  author had — the "why" of certain error paths is lost.
- Requires a dedicated team, which violates Constitution Principle 10 (Small
  Stable Core) — the core team should not need a separate observability team.

**Why rejected:** The centralized team approach does not scale and adds organizational
  complexity that is disproportionate for the platform's size. The gate-based
  approach distributes the responsibility to Volume authors, where the domain
  knowledge lives.

### Alternative C: Volume Approval Gate with Minimum Instrumentation Spec (Recommended)
**Description:** As described in Proposed Decision. Observability requirements section
  required for Volume approval, enforced via CI lint, with specific minimums for
  metrics, counters, spans, and logs.

**Pros:**
- Consistent with ADR-0009's proven pattern — same enforcement mechanism, same
  approval gate.
- Distributes responsibility to Volume authors, who have the most domain knowledge.
- The minimum specification (one metric per method, one counter per event) is
  achievable without excessive effort.
- CI enforcement ensures the gate is objective and consistent.
- The specification is forward-compatible — additional instrumentation can be added
  beyond the minimum.

**Cons:**
- Adds documentation burden to Volume authors (estimated: 30–60 minutes per Volume
  to write the observability section).
- The CI lint tool must be built and maintained (estimated: 1 day of effort).
- The minimum specification may be insufficient for complex Volumes — but it is a
  *minimum*, not a *maximum*. Volumes can specify more.

## Consequences

- **Positive:** Every Approved Volume has a documented observability plan. The
  platform's operational visibility improves incrementally as each Volume reaches
  Approved status. The CI gate ensures no regression.
- **Negative:** Volume authors must invest time in specifying instrumentation before
  implementation begins (shifting left, which is good for quality but adds upfront
  effort). The CI lint tool requires maintenance.
- **Migration path (if applicable):** Already-Approved Volumes (if any) are not
  retroactively required to add observability sections. However, any substantive
  revision of an Approved Volume must include the observability section before the
  revision can be merged.
- **Affected interfaces/contracts:** Volume approval checklist (Volume 1, Ch. 6)
  gains a new mandatory item. CI pipeline gains `obs-lint-handbook` step. Each
  Volume's document gains an "Observability Requirements" section.
