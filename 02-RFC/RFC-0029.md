# RFC-0029: API/Interface Deprecation Policy

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 1

---

## Context

As the agentx platform evolves, interfaces, APIs, and configuration formats will need to
change. Some changes are additive (new fields, new methods) and are non-breaking. Others
are breaking (removed fields, changed method signatures, renamed configuration keys) and
require consumers to update their code.

Without a formal deprecation policy, breaking changes are either:
- Never made (leading to API bloat and accumulated technical debt), or
- Made without warning (breaking consumers unexpectedly, violating trust).

Constitution Principle 1 (Architecture First) requires that all architectural decisions
be documented and followed. A deprecation policy is an architectural decision about how
the platform manages change. Volume 1, Chapters 5 and 6 define the RFC and document
lifecycle, but do not address the lifecycle of the platform's own APIs.

This RFC establishes the deprecation windows, notification mechanisms, and consumer
migration requirements that govern all public interfaces in the agentx platform.

## Proposed Decision

### Deprecation Windows

**Non-Breaking Changes (additive):**
- No deprecation window needed. New fields are added, old fields remain.
- Consumers can adopt new features at their own pace.
- Documented in the CHANGELOG under "Added" section.

**Breaking Changes:**
- **2 minor versions** of deprecation warning before removal.
- Example: a feature deprecated in v1.2 is removed in v2.0 (the next major version).
- During the deprecation window: the old API continues to work, emits a runtime warning,
  and the TypeScript type is marked `@deprecated`.
- After the deprecation window: the old API is removed. The TypeScript type is deleted.
  Importing or calling the removed API produces a compile-time error.

### Versioning Semantics

The platform follows SemVer (Semantic Versioning) strictly:

| Change Type | Version Bump | Example |
|-------------|-------------|---------|
| Bug fix (no API change) | Patch (x.x.Z) | Fix edge case in tool permission check |
| Non-breaking feature | Minor (x.Y.0) | Add new optional parameter to method |
| Breaking change | Major (X.0.0) | Remove or rename a public method |

A "breaking change" is any change that causes existing, correct consumer code to fail
to compile or behave differently. This includes:
- Removed or renamed exports (functions, classes, interfaces, types).
- Changed function signatures (added required parameters, changed return types).
- Changed configuration key names or value formats.
- Changed default behavior that consumers relied on.
- Removed support for a previously-supported provider or tool.

### Notification Mechanisms

Deprecation notifications use **three channels** in order of immediacy:

1. **TypeScript `@deprecated` JSDoc tag:** Every deprecated export is annotated:
   ```typescript
   /**
    * @deprecated Use `resolveCredentials()` instead. Will be removed in v2.0.0.
    * @see https://docs.agentx.dev/migration/v2#credential-resolver
    */
   export function getCredentials(providerId: string): Promise<Credentials> { ... }
   ```
   Consumers see this in their IDE (VSCode, WebStorm) and at compile time via
   `tsc --noEmit`.

2. **Runtime warning log:** When a deprecated API is called at runtime, a structured
   warning is emitted:
   ```
   [DEPRECATION] getCredentials() is deprecated and will be removed in v2.0.0.
   Use resolveCredentials() instead. See: https://docs.agentx.dev/migration/v2#credential-resolver
   ```
   The warning includes the deprecated API name, the replacement, the removal version,
   and a migration guide URL. Warnings are emitted at most once per process per API
   (deduplicated via a `Set<string>`).

3. **CHANGELOG entry:** Every deprecation is documented in the CHANGELOG under a
   "Deprecated" section:
   ```markdown
   ### Deprecated
   - `getCredentials()` — use `resolveCredentials()` instead. Removal: v2.0.0. (#1234)
   ```

### Consumer Migration Guide Requirement

Every breaking change (major version) must be accompanied by a migration guide published
at `docs.agentx.dev/migration/vX`. The guide must include:
- A summary of all breaking changes with before/after code examples.
- An automated codemod script (using `ts-morph` or `jscodeshift`) that can
  mechanically apply the migration where possible.
- Manual migration steps for changes that cannot be automated.
- A timeline: when the deprecation was announced, when removal occurs, and how long
  consumers have to migrate.

## Alternatives Considered

### Alternative A: No Deprecation — Breaking Changes Only via Major Versions with No Warning
**Description:** Breaking changes happen only in major versions. No deprecation window,
  no runtime warnings. Consumers discover breaking changes when they upgrade and their
  code fails to compile.

**Pros:**
- Simplest approach. No deprecation infrastructure to build or maintain.
- No cognitive overhead of maintaining both old and new APIs simultaneously.
- CHANGELOG-only communication is sufficient for a small, technical user base.

**Cons:**
- Consumers experience breaking changes as a surprise. Trust is eroded.
- No runtime signal means consumers may not discover deprecations until they upgrade
  to the major version and face a wall of compile errors.
- Violates the principle of giving consumers adequate time to adapt.
- Particularly harmful for plugins (Volume 8) and agent roles (Volume 3) that are
  maintained by external contributors — they may not watch every CHANGELOG.

**Why rejected:** Surprise breaking changes are a well-known anti-pattern in platform
  engineering. The cost of maintaining a deprecation window is small compared to the
  cost of consumer frustration and ecosystem fragmentation.

### Alternative B: Infinite Deprecation — Never Remove Deprecated APIs
**Description:** Deprecated APIs are marked but never removed. They remain in the
  codebase forever, accumulating over time.

**Pros:**
- Maximum backward compatibility. No consumer ever breaks.
- No migration guide needed — old code just works.
- Simple mental model: "everything we ever shipped still works."

**Cons:**
- Codebase bloat: deprecated APIs accumulate, increasing bundle size, maintenance
  burden, and cognitive load for new contributors.
- Type pollution: deprecated types appear in autocomplete suggestions, confusing
  developers about which API to use.
- Testing burden: deprecated APIs must still be tested (they're still shipped code),
  even though they're not recommended.
- Eventually, deprecated APIs conflict with new features (same name, incompatible
  semantics), forcing a removal anyway.

**Why rejected:** Infinite deprecation is technically possible but practically harmful.
  It trades short-term convenience for long-term maintainability debt. The 2-minor-
  version window is a proven balance (used by Node.js, React, and other major platforms).

### Alternative C: 2-Minor-Version Deprecation with Multi-Channel Notification (Recommended)
**Description:** As described in Proposed Decision. Structured deprecation windows,
  TypeScript + runtime + CHANGELOG notifications, and migration guides with codemods.

**Pros:**
- Consumers have 2–6 months (typical minor version cadence) to migrate, depending
  on release frequency.
- Three notification channels ensure no consumer misses the deprecation (IDE, runtime,
  release notes).
- The codemod requirement reduces migration effort for mechanical changes.
- Consistent with industry best practices (Node.js, React, Angular).
- The policy is objective and enforceable — no debate about "how long is long enough."

**Cons:**
- Maintaining deprecated APIs during the window increases codebase size temporarily.
- Codemod scripts require maintenance and testing.
- The 2-minor-version window may be too long for urgent security fixes that require
  breaking changes (mitigated by an exception process described below).

### Exception: Security-Fix Breaking Changes

When a breaking change is required to fix a security vulnerability (e.g., a tool
permission model change that requires a new method signature), the deprecation window
may be shortened to **1 minor version** or, in extreme cases, **0 minor versions**
(immediate removal in a patch). This exception must be:
- Approved by a security reviewer.
- Documented in the CHANGELOG with a `[SECURITY]` prefix.
- Accompanied by an expedited migration guide.

## Consequences

- **Positive:** The platform has a predictable, fair, and industry-standard approach
  to API evolution. Consumers can plan upgrades with confidence. The multi-channel
  notification ensures deprecations are visible. Codemods reduce migration friction.
- **Negative:** Deprecated code must be maintained during the window. Codemod scripts
  add to the release engineering burden. The security exception process introduces a
  subjective judgment call (is the security fix urgent enough to shorten the window?).
- **Migration path (if applicable):** No migration needed — this is a policy that
  applies to all future API changes. Existing deprecated patterns (if any) are
  documented with their removal timeline.
- **Affected interfaces/contracts:** This policy applies to all public exports in
  `packages/*/src/index.ts` (library interfaces), all REST/gRPC endpoints (API
  interfaces), all configuration keys (config interfaces), and all CLI commands
  (operator interfaces).
