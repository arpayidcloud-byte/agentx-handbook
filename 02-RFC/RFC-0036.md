# RFC-0036: Marketplace Sequencing Decision

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 8, 14

---

## Context

Volume 8 (Plugin Platform) envisions a marketplace where operators can discover, install,
and manage third-party plugins (custom tools, agent extensions, workflow steps). A
marketplace provides significant ecosystem value: it enables community contributions,
reduces the platform team's burden of building every integration, and creates a network
effect that drives adoption.

However, a marketplace without a robust trust model is a security liability. An operator
who installs a malicious plugin from an unvetted marketplace could compromise their entire
development environment. RFC-0027 defines the plugin sandboxing and trust model, and
ADR-0016 establishes the principle that security-critical features must be validated before
exposure to untrusted input. Together these inform the sequencing question: **should the marketplace be built now, or should
the trust model be completed first?**

The tension is between speed-to-ecosystem (build marketplace early to attract plugin
authors) and security-first (complete the trust model before exposing operators to
third-party code). A wrong sequencing decision either delays ecosystem growth or exposes
operators to unnecessary risk.

## Proposed Decision

### Recommended Sequence: Trust Model MVP → Plugin Platform → Marketplace UI (v2.0)

**Phase 1: Trust Model MVP (v0.5, same release as Plugin Platform)**
- Implement the `PluginSandboxConfig` and permission manifest from RFC-0027.
- Implement process-level isolation (child_process with restricted require).
- Implement plugin signing and verification (platform key pair).
- Ship with **core-tier plugins only** (built-in tools, no third-party).
- The trust model is in place and tested, but there is no UI for discovering or
  installing third-party plugins.

**Phase 2: Plugin Installation via CLI (v0.5, same release)**
- `agentx plugin install <package>` downloads and installs a plugin from an npm
  registry or git repository.
- The plugin's `plugin.json` manifest is validated. Permission manifest is checked
  against the trust tier.
- Verified-tier plugins require a valid signature. Unverified-tier plugins show a
  warning and require `--allow-unverified` flag.
- No marketplace UI — plugin discovery is via documentation, word of mouth, or
  direct URL.

**Phase 3: Marketplace UI (v2.0)**
- A web-based marketplace (served by the agentx server or a separate app) where
  operators can browse, search, and install plugins.
- Plugin listing requires completion of the review process (RFC-0028 for agent
  roles, a lighter review for tool plugins).
- Ratings, download counts, and security badges provide additional trust signals.
- The marketplace is gated behind the trust model: no plugin is listed without
  passing the appropriate review.

### Why Not Build the Marketplace Now

1. **Trust model is incomplete without real plugins.** The sandbox and permission
   manifest cannot be validated against real-world plugin behavior until real
   plugins exist. Building the marketplace first means the trust model is untested
   when it matters most.

2. **Marketplace requires plugin review capacity.** The review process (RFC-0028)
   requires human reviewers. Until the plugin ecosystem is large enough to justify
   dedicated review capacity, a marketplace UI creates an expectation of rapid
   listing that cannot be met.

3. **v0.5 CLI installation is sufficient for early adopters.** Early plugin authors
   are developers who are comfortable with `agentx plugin install <git-url>`. A
   graphical marketplace is not needed for this audience.

4. **Marketplace UI is a significant frontend effort.** Building a browsable,
   searchable, authenticated web UI is 4–8 weeks of work that would distract from
   the core platform's v0.5 priorities (multi-user auth, RBAC, plugin sandboxing).

## Alternatives Considered

### Alternative A: Build Marketplace Now (v0.5) with Minimal Trust
**Description:** Ship a marketplace UI alongside the plugin platform. Plugins are
  listed with minimal review (automated manifest validation only, no human review).
  Trust signals are limited to a "verified" badge for signed plugins.

**Pros:**
- Maximum ecosystem growth potential — plugin authors have a visible distribution
  channel from the earliest possible moment.
- Early feedback on the marketplace UI from real users.
- Competitive advantage: being first to market with a plugin ecosystem attracts
  both authors and users.

**Cons:**
- Minimal trust model means operators are the QA team. A malicious plugin that
  passes manifest validation but contains subtle exploits would be installed by
  early adopters.
- The marketplace's reputation is fragile — a single high-profile security incident
  (malicious plugin stealing credentials) would permanently damage trust.
- Review capacity does not exist — the "verified" badge is meaningless without
  actual review.
- Violates Constitution Principle 7: the platform would be exposing operators to
  unvetted third-party code before the security model is proven.

**Why rejected:** The risk of a marketplace security incident outweighs the benefit
  of early ecosystem growth. Trust, once lost, is nearly impossible to regain.

### Alternative B: Wait for Full Trust Model Before Any Plugin Support (v2.0)
**Description:** No third-party plugin support until the full trust model is
  implemented: sandboxing, signing, review process, and marketplace UI all ship
  together in v2.0.

**Pros:**
- Safest approach — operators are never exposed to third-party code until every
  security control is in place.
- The trust model has maximum time to mature (v0.5 through v1.5) before being
  exposed to real plugins.
- No partial or inconsistent security posture — the platform is either "plugin-
  safe" or "plugin-free," never in between.

**Cons:**
- Delays ecosystem growth by 2+ years (v0.5 to v2.0). Plugin authors have no
  platform to target.
- The trust model cannot be validated against real plugin behavior without real
  plugins — a chicken-and-egg problem. The sandbox may have vulnerabilities that
  are only discovered under real-world use.
- The platform loses competitive positioning to alternatives that support plugins
  earlier.
- Internal teams cannot use custom plugins for their own workflows, limiting
  the platform's utility even for the core team.

**Why rejected:** Waiting until v2.0 for any plugin support is overly conservative.
  The trust model MVP (sandboxing + manifest + signing) provides sufficient security
  for CLI-based plugin installation by technical early adopters. The full trust
  model (including marketplace review) is needed for the UI, but not for CLI
  installation.

### Alternative C: Trust Model MVP → CLI Install → Marketplace UI (Recommended)
**Description:** As described in Proposed Decision. Phase the feature delivery:
  trust model first, CLI installation second, marketplace UI third (v2.0).

**Pros:**
- The trust model is proven against real plugins before the marketplace amplifies
  distribution.
- CLI-based installation serves early adopters without the UI development cost.
- The marketplace launches with a mature, tested trust model and an existing
  plugin ecosystem — a strong foundation.
- Each phase delivers standalone value: trust model enables internal plugins,
  CLI install enables early community, marketplace enables broad adoption.
- Consistent with Constitution Principle 10 (Small Stable Core): marketplace UI
  is not part of the core; it's a separate application.

**Cons:**
- The marketplace UI is delayed to v2.0 — a long wait for non-technical operators
  who want a graphical plugin browser.
- CLI-based installation may not attract non-developer plugin authors (they want a
  web form, not a git URL).
- The "verified" badge on CLI-installed plugins has less visibility than in a
  marketplace UI.

## Consequences

- **Positive:** The trust model is validated by real plugin usage before the
  marketplace amplifies distribution risk. Each phase delivers standalone value.
  The marketplace launches with an existing plugin ecosystem and proven security.
- **Negative:** Marketplace UI is delayed to v2.0. Early plugin authors must use
  CLI-based distribution. The gap between CLI install (v0.5) and marketplace (v2.0)
  may confuse users about the "right way" to install plugins.
- **Migration path (if applicable):** No migration — this is a sequencing decision,
  not a technical change. Plugins installed via CLI in v0.5 are automatically
  available in the marketplace UI when it launches in v2.0.
- **Affected interfaces/contracts:** Volume 8 roadmap updated to reflect the
  three-phase sequence. Plugin installation CLI commands shipped in v0.5.
  Marketplace API (for the v2.0 UI) designed but not implemented until v2.0.
