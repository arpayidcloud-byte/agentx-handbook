# RFC-0026: Identity-to-RBAC Handoff Contract

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 15, 10

---

## Context

RFC-0025 defines the `Identity` abstraction and three authentication modes. RFC-0012
(Volume 10, Ch. 2) defines the RBAC model with three roles: Owner, Developer, Viewer.
These two systems must be connected: an authenticated `Identity` must be mapped to a set
of roles and permissions before any authorization check can occur.

The question is: **where and how does this mapping happen?** The mapping is a cross-volume
concern — it consumes output from Volume 15 (Identity) and feeds input to Volume 10
(RBAC). Without a formal contract, the implementation risks circular dependencies, tight
coupling, or inconsistent behavior across authentication modes.

This RFC defines the handoff contract: the interface that bridges authentication
(Volume 15) and authorization (Volume 10), the role assignment workflow, role inheritance
rules, and the permission resolution order.

## Proposed Decision

### Handoff Interface: `IRoleResolver`

A new interface in a shared package (`packages/auth/src/role-resolver.ts`) that is
implemented by Volume 15 and consumed by Volume 10:

```typescript
interface RoleAssignment {
  roleId: string;       // e.g., 'owner', 'developer', 'viewer'
  scope: 'global' | 'organization' | 'project';
  scopeId?: string;     // org ID or project ID if scope is not 'global'
  grantedAt: string;    // ISO 8601
  grantedBy: string;    // identity ID of the grantor
}

interface IRoleResolver {
  /**
   * Resolve all role assignments for a given identity.
   * Called by PermissionChecker on every authorization check.
   */
  resolveRoles(identityId: string): Promise<RoleAssignment[]>;
}
```

### Role Assignment Workflow

1. **Local mode (v0.1):** `IRoleResolver` returns a hardcoded assignment
   `[{ roleId: 'owner', scope: 'global' }]` for the single local identity.
   No database involved.

2. **Token mode (v0.5):** Role assignments are stored in a `role_assignment` table
   with `(identityId, roleId, scope, scopeId)` as a unique composite key. The
   `IRoleResolver` implementation queries this table. Assignments are managed via:
   - `agentx rbac assign <identity> <role> [--scope org:<orgId>]`
   - `agentx rbac revoke <identity> <role> [--scope org:<orgId>]`

3. **SSO mode (v1.0+):** Role assignments are still stored locally (the platform
   retains its own RBAC model), but initial assignments can be seeded from IdP group
   claims. A mapping configuration (Volume 10, Ch. 4) maps IdP groups to agentx roles:
   ```yaml
   idpGroupMappings:
     "admin-group": "owner"
     "developer-group": "developer"
     "viewer-group": "viewer"
   ```
   IdP group mappings are applied on JIT provisioning and can be refreshed via
   `agentx rbac sync-from-idp`.

### Role Inheritance

Roles do **not** inherit from each other in v0.1–v0.5. Each role's permissions are
explicitly defined (RFC-0012). An identity with the `developer` role does not
automatically get `viewer` permissions — instead, the `developer` role's permission
set is a superset of `viewer` permissions, defined explicitly.

Rationale: implicit inheritance creates subtle bugs when a lower role's permissions
are changed and the higher role unexpectedly gains or loses capabilities. Explicit
permission sets are more maintainable and auditable.

**v1.0+ exception:** Organization-scoped `owner` role implicitly includes all
organization-scoped permissions of `developer` and `viewer` within the same org.
This is documented as a deliberate exception to the no-inheritance rule, justified
by the semantic meaning of "owner" in an organizational context.

### Permission Resolution Order

When `PermissionChecker.isAllowed(identity, permission, resource)` is called:

1. Resolve all `RoleAssignment[]` for `identity.id` via `IRoleResolver`.
2. For each assignment, look up the role's permission set (RFC-0012).
3. If **any** assignment grants the requested permission on the requested resource,
   return `true` (disjunctive — most permissive assignment wins).
4. If **no** assignment grants the permission, return `false`.
5. If the identity has **no assignments at all**, return `false` (fail-closed per
   ADR-0004).

This order is deliberately simple: no priority, no weighting, no deny-override.
Either a role grants a permission or it doesn't. Deny-lists (if needed in v1.0+)
are a separate feature that adds a second pass after the allow-check.

## Alternatives Considered

### Alternative A: Embed Role Resolution in Identity
**Description:** The `Identity` object (RFC-0025) includes resolved permissions directly.
  `PermissionChecker` checks `identity.permissions.includes(permission)`.

**Pros:**
- Simplest consumer API — no second service call needed.
- Permission resolution happens once at authentication time, not on every check.
- Easy to log and audit (permissions are part of the identity).

**Cons:**
- Permissions become stale if roles are changed after authentication. The identity
  would need to be refreshed, or the permission set would be wrong until re-auth.
- Violates separation of concerns: identity (who you are) is conflated with
  authorization (what you can do). RFC-0025 deliberately keeps `Identity` focused on
  authentication attributes.
- The `Identity` object grows unboundedly as the permission set grows — problematic
  for JWT encoding (size limits) and in-memory caching.
- Role changes require invalidating all cached identities — a heavy operation.

**Why rejected:** Stale permissions are a security risk. The `IRoleResolver` approach
  keeps identity and authorization as separate, independently refreshable concerns.

### Alternative B: Central Policy Engine (e.g., OPA/Cedar)
**Description:** Introduce a dedicated policy engine (Open Policy Agent or AWS Cedar)
  that evaluates all authorization decisions. Application code sends
  `(identity, action, resource)` to the policy engine and receives allow/deny.

**Pros:**
- Most flexible approach — policies can be changed without code deployment.
- Supports complex logic: attribute-based access control (ABAC), conditional policies,
  time-based access.
- Industry-standard approach for enterprise authorization.
- Separation of policy from application code is a best practice.

**Cons:**
- Adds a new runtime dependency (OPA server or Cedar engine) — increases operational
  complexity.
- Policy language (Rego for OPA, Cedar for AWS) is a new skill requirement for
  contributors and operators.
- Overkill for three roles with simple permission sets. The complexity budget is
  disproportionate to the problem.
- Performance overhead of policy evaluation on every authorization check (though OPA
  is fast, it's still an extra network hop or embedded evaluation step).
- Violates Constitution Principle 10 (Small Stable Core) at the current stage.

**Why rejected:** A policy engine is the right architecture for v2.0+ when the RBAC
  model evolves to ABAC. For v0.1–v1.0, three roles with explicit permission sets
  are sufficiently expressive. The `IRoleResolver` interface is compatible with a
  future migration to OPA (the resolver could delegate to OPA internally).

### Alternative C: IRoleResolver with Explicit Assignments (Recommended)
**Description:** As described in Proposed Decision. A dedicated interface, database-backed
  role assignments, no inheritance, disjunctive resolution order.

**Pros:**
- Clean separation between authentication (Volume 15) and authorization (Volume 10).
- Role assignments are queryable and auditable — stored in PostgreSQL.
- No inheritance complexity — permission sets are explicit per role.
- The `IRoleResolver` interface is a stable extension point: SSO mapping, ABAC
  migration, or OPA delegation can be implemented behind the same interface.
- Compatible with existing RFC-0012 RBAC model — no changes to role definitions.
- Fail-closed by default (no assignments = no permissions), consistent with ADR-0004.

**Cons:**
- A database query on every authorization check adds latency. Mitigated by caching
  role assignments in-memory with a short TTL (similar to RFC-0023's credential cache).
- The `role_assignment` table is a new schema addition to Volume 6.
- No inheritance means the `developer` role must explicitly list all permissions,
  including those shared with `viewer` — a minor maintenance burden.

## Consequences

- **Positive:** The authentication-to-authorization path is explicit, testable, and
  auditable. Role assignments are a first-class entity with their own CRUD operations
  and audit trail. The design supports the three authentication modes without mode-
  specific authorization code.
- **Negative:** One additional database query per authorization check (mitigated by
  in-memory caching). The `role_assignment` table adds schema complexity. No inheritance
  means permission sets must be maintained explicitly.
- **Migration path (if applicable):** v0.1 uses a hardcoded `LocalRoleResolver`. v0.5
  migrates to a `DatabaseRoleResolver` via `agentx auth migrate`, which creates the
  `role_assignment` table and seeds the local identity with the `owner` role.
- **Affected interfaces/contracts:** New `IRoleResolver` interface in `packages/auth/`.
  New `role_assignment` table in Volume 6 schema. New CLI commands `agentx rbac assign`,
  `agentx rbac revoke`, `agentx rbac list`. Volume 10's `PermissionChecker` updated to
  accept `IRoleResolver` via dependency injection.
