# RFC-0031: RFC/ADR Minimum Content Bar

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 00-Governance

---

## Context

The Architecture Assessment Report scored the project 3.0/5.0, citing "RFCs are only
15–20 lines" as a key gap. The existing RFCs (RFC-0001 through RFC-0020) are
concise — they state the problem, proposal, alternatives (often a single sentence),
and decision in under 20 lines total. While brevity is valuable, it has led to:

1. **Institutional memory loss:** Future contributors reading an RFC cannot understand
   *why* a decision was made because the alternatives analysis is too thin. The
   "why rejected" reasoning is compressed into a half-sentence that omits the
   trade-offs considered.

2. **Repeated debates:** The same architectural question resurfaces because the
   original RFC did not adequately document the alternatives and their drawbacks.
   Without a written record, new contributors re-litigate settled decisions.

3. **Inconsistent quality:** Some RFCs have alternatives sections; others do not.
   There is no enforcement mechanism — the RFC template (Volume 1, Ch. 5) lists
   "Alternatives" as a section but does not specify minimum content.

This RFC establishes a minimum content bar for all RFCs and ADRs going forward, and
defines a CI enforcement mechanism to prevent regression. This is consistent with the
governance quality standards described in ADR-0013.

## Proposed Decision

### RFC Minimum Content Requirements

Every RFC must include the following sections with minimum content:

1. **Context:** At least 2 paragraphs explaining the problem or opportunity that
   motivates the RFC. Must reference relevant Constitution Principles, existing
   ADRs/RFCs, and affected Volumes.

2. **Proposed Decision:** A detailed description of the recommended approach.
   For interface changes, include a TypeScript interface block. For process changes,
   include a step-by-step description. Minimum: 1 paragraph plus supporting detail
   (code, table, or diagram).

3. **Alternatives Considered:** At least **2 alternatives** (not counting the
   recommended approach). Each alternative must include:
   - **Description:** 1–3 sentences explaining the approach.
   - **Pros:** At least 2 concrete benefits.
   - **Cons:** At least 2 concrete drawbacks.
   - **Why rejected:** A clear, specific reason the alternative was not chosen.
     "Not chosen" is not sufficient — the reason must reference a constitutional
     principle, a technical constraint, or a trade-off analysis.

4. **Consequences:** At least 3 bullet points covering:
   - Positive consequences (benefits).
   - Negative consequences (costs, risks).
   - Migration path or affected interfaces (if applicable).

5. **Affected Volumes & Chapters:** A table mapping specific chapters to the
   changes required.

**Minimum line count:** An RFC must be at least **80 lines** of substantive content
(excluding blank lines, the header block, and section headers). This is a floor, not
a target — complex RFCs will naturally be longer.

### ADR Minimum Content Requirements

ADRs are intentionally shorter than RFCs but must still meet a minimum bar:

1. **Context:** At least 1 paragraph explaining the situation and forces at play.

2. **Decision:** A clear statement of what was decided. Must be implementable by
   a developer who reads only this ADR.

3. **Alternatives Considered:** At least **1 alternative** briefly described with a
   reason it was not chosen. This may be a single paragraph but must exist.

4. **Consequences:** At least 2 bullet points.

**Minimum line count:** An ADR must be at least **20 lines** of substantive content.

### CI Enforcement

A CI lint check validates RFC and ADR files on every pull request:

```yaml
# .github/workflows/doc-lint.yml
rfc-lint:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - name: Lint RFC/ADR documents
      run: npx markdown-lint-handbook --min-rfc-lines 80 --min-adr-lines 20
```

The linter checks:
- **Required section headers:** `## Context`, `## Proposed Decision` (or
  `## Decision` for ADRs), `## Alternatives Considered`, `## Consequences`.
- **Minimum line count:** Substantive content lines (excluding headers, blank
  lines, and the metadata block) meet the minimum.
- **Alternatives count:** At least 2 distinct `### Alternative` headings for RFCs,
  at least 1 for ADRs.
- **Affected Volumes:** RFCs must include an `## Affected Volumes` section.

A failing lint check blocks the PR. An exception process exists for housekeeping
RFCs (e.g., typo fixes, formatting corrections) — the PR description must include
`[RFC-LINT-EXEMPT]` with a rationale, and a maintainer must approve the exemption.

### Grandfathering

Existing RFCs (RFC-0001 through RFC-0020) are **grandfathered** — they are not
required to be retroactively expanded. However, if an existing RFC is substantively
amended (status change, supersession, or content update), it must be expanded to
meet the new minimum content bar at that time.

## Alternatives Considered

### Alternative A: No Minimum Bar — Trust Authors
**Description:** Continue with the current approach. Trust RFC/ADR authors to include
  sufficient content. No CI enforcement.

**Pros:**
- No additional process or tooling. Authors write as much or as little as they
  feel is appropriate.
- Avoids "writing for the linter" — content driven by substance, not rules.
- Fastest path for simple decisions that don't warrant lengthy analysis.

**Cons:**
- The current state (15–20 line RFCs) demonstrates that trust alone is insufficient.
  Without enforcement, the natural tendency under time pressure is to write less.
- Institutional memory loss continues — the original motivation for this RFC.
- No objective quality metric — "sufficient content" is subjective without a minimum.
- Reviewers have no objective criterion to request more detail from authors.

**Why rejected:** The assessment explicitly identified short RFCs as a gap. Trust
  without verification has produced the current inadequate state. A minimum bar
  provides an objective, enforceable quality floor.

### Alternative B: Peer Review Gate Only (No CI)
**Description:** Require that a reviewer explicitly check a "content completeness"
  box before approving an RFC PR. No automated linting.

**Pros:**
- Human judgment can distinguish between "concise but complete" and "incomplete."
- No tooling to build or maintain.
- Flexible — the reviewer can accept shorter content for genuinely simple decisions.

**Cons:**
- Relies on reviewer diligence, which varies. A tired reviewer may approve a thin
  RFC to avoid conflict.
- Subjective — different reviewers have different standards for "complete."
- Adds cognitive load to the reviewer (they must mentally check all required sections
  and alternatives).
- Does not scale — as the number of RFCs grows, the review burden grows linearly.

**Why rejected:** CI enforcement is more reliable and consistent than human review
  for structural requirements. The reviewer's judgment is still valuable for
  *quality* of content, but the *presence* of required sections is better checked
  by a machine.

### Alternative C: CI-Enforced Minimum Content Bar (Recommended)
**Description:** As described in Proposed Decision. Required sections, minimum line
  counts, CI lint check, grandfathering of existing RFCs, exception process for
  housekeeping.

**Pros:**
- Objective, automated, and consistent — every PR is checked against the same rules.
- The minimum line count prevents the "15-line RFC" pattern without mandating
  verbosity.
- Required section headers ensure a consistent structure across all RFCs.
- Grandfathering avoids the cost of retroactively expanding 20 existing RFCs.
- The exception process allows flexibility for genuinely simple housekeeping changes.
- Sets a clear expectation for new contributors: "your RFC must have these sections
  and this much content."

**Cons:**
- Authors may pad content to meet the line count, adding low-value text. Mitigated
  by reviewer judgment on *quality* (the linter checks quantity, the reviewer checks
  quality).
- The lint tool must be built and maintained (estimated: 1 day of effort).
- Grandfathering means the first 20 RFCs remain thin — but this is acceptable because
  the new RFCs (starting from RFC-0021) set the standard going forward.

## Consequences

- **Positive:** All future RFCs and ADRs meet a documented quality bar. Institutional
  memory is preserved in the alternatives analysis. New contributors have a clear
  template and expectation. The CI check prevents regression.
- **Negative:** Slightly higher barrier to entry for RFC authors — a simple decision
  now requires 80+ lines. The exception process adds a minor workflow overhead for
  housekeeping changes.
- **Migration path (if applicable):** No migration for existing RFCs. The lint tool
  is added to CI in the next sprint. The RFC template (Volume 1, Ch. 5) is updated
  to reflect the new requirements.
- **Affected interfaces/contracts:** RFC template updated. CI pipeline gains a new
  lint step. CONTRIBUTING.md (Volume 0) updated with RFC/ADR authoring guidelines.
