# RFC-0040: Documentation Platform Tooling

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** Governance services

---

## Context

The agentx handbook (`00-Governance/`, `01-Volumes/`, `02-RFC/`, `03-ADR/`,
`04-Schemas/`, `05-Templates/`, `08-Examples/`) is the authoritative source of
architectural decisions and specifications. As the handbook grows past 20 RFCs,
14 Volumes, and 10 ADRs, maintaining its internal consistency becomes a challenge:

1. **Cross-references break.** An RFC references Volume 7, Ch. 2, but a later
   refactor renumbers the chapters. The reference is now stale.
2. **Schemas drift.** A TypeScript interface in `04-Schemas/` is updated, but the
   corresponding Volume chapter still describes the old interface.
3. **Templates are not followed.** A new RFC is written without the required
   sections (the problem RFC-0031 was created to solve).
4. **Format inconsistencies.** Some documents use Mermaid diagrams, others use
   ASCII art. Some use TypeScript interfaces, others use JSON schemas.

These issues undermine the handbook's credibility as the single source of truth.
Without automated tooling, consistency can only be maintained by manual review —
which is unreliable at scale.

This RFC defines the tooling for linting, validating, and maintaining the handbook.

## Proposed Decision

### Tool 1: Schema Validation for 04-Schemas

The `04-Schemas/` directory contains TypeScript interface definitions that represent
canonical data models. A CI step validates that:
- Every `.ts` file in `04-Schemas/` compiles without errors (`tsc --noEmit`).
- Every interface referenced in a Volume chapter exists in `04-Schemas/` (cross-
  reference check).
- Every interface in `04-Schemas/` is referenced by at least one Volume chapter
  (orphan check — unused schemas should be removed or documented as "planned").

```bash
npx schema-lint-handbook
# Checks:
# ✓ All 04-Schemas/*.ts compile
# ✓ All Volume-referenced interfaces exist in 04-Schemas
# ✓ All 04-Schemas interfaces are referenced by at least one Volume
# ✗ Interface 'PluginSandboxConfig' referenced in RFC-0027 but not found in 04-Schemas
```

### Tool 2: Cross-Reference Checker

A linter that validates internal links and references across all handbook documents:

**RFC cross-references:**
- Every RFC must reference at least one Volume in its `Affected Volumes` section.
- Every RFC that references another RFC (via "Depends on" or "References") must
  specify a valid RFC number (0001–0042 and growing).
- Every RFC that references an ADR must specify a valid ADR number.

**ADR cross-references:**
- Every ADR must reference the RFC it implements (in the `Implements:` metadata).
- Every ADR must reference at least one Volume.

**Volume cross-references:**
- Every Volume chapter that references another Volume's chapter must specify a valid
  chapter number.
- Volume chapter numbers are validated against the Volume's table of contents.

**Orphan detection:**
- Every RFC must be referenced by at least one Volume chapter or another RFC/ADR.
  (Exception: the most recent RFC may be unreferenced until the next Volume update.)

```bash
npx xref-lint-handbook
# Checks:
# ✓ All RFCs have valid Volume references
# ✓ All ADRs reference a valid implementing RFC
# ✓ All internal links resolve
# ✗ RFC-0021 references "Volume 7, Ch. 5" but Volume 7 has only 4 chapters
```

### Tool 3: Template Compliance Checker

Validates that governance documents follow their templates:

**RFC compliance:**
- Checks for required sections: Context, Proposed Decision, Alternatives Considered,
  Consequences, Affected Volumes (per RFC-0031).
- Checks minimum line count (80 lines substantive content per RFC-0031).
- Checks for at least 2 alternatives with Description, Pros, Cons, Why Rejected
  (per RFC-0031).

**ADR compliance:**
- Checks for required sections: Context, Decision, Consequences (per RFC-0031).
- Checks minimum line count (20 lines substantive content per RFC-0031).
- Checks that the ADR references an implementing RFC.

**Volume compliance:**
- Checks for required sections: Overview, Functional Requirements, Interface
  Definitions (at minimum).
- Checks for an "Observability Requirements" section if the Volume is seeking
  Approved status (per RFC-0033).

```bash
npx template-lint-handbook
# Checks:
# ✓ RFC-0021: all required sections present
# ✓ RFC-0021: 142 substantive lines (≥ 80)
# ✓ RFC-0021: 3 alternatives with full detail
# ✗ ADR-0003: 15 substantive lines (< 20 minimum)
```

### CI Integration

All three tools run on every PR that touches handbook files:

```yaml
# .github/workflows/handbook-lint.yml
handbook-lint:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - name: Install handbook lint tools
      run: npm install -g @agentx/handbook-lint
    - name: Schema validation
      run: schema-lint-handbook
    - name: Cross-reference check
      run: xref-lint-handbook
    - name: Template compliance
      run: template-lint-handbook
```

The CI job fails if any tool reports errors. Warnings (e.g., orphan detection for the
most recent RFC) do not fail the build but are logged for review.

### Implementation Approach

The handbook lint tools are implemented as a single npm package
`@agentx/handbook-lint` with three CLI commands:
- `schema-lint-handbook`
- `xref-lint-handbook`
- `template-lint-handbook`

Implementation uses:
- `markdown-it` for parsing markdown files.
- `typescript` compiler API for schema compilation and interface extraction.
- `glob` for file discovery.
- `chalk` for colored output.

Estimated implementation effort: 2–3 days.

## Alternatives Considered

### Alternative A: No Automated Tooling — Manual Review Only
**Description:** Consistency is maintained by reviewers checking PRs that touch
  handbook files. No automated linting.

**Pros:**
- Zero implementation effort. No tooling to build or maintain.
- Human reviewers can catch nuanced issues that automated tools miss (e.g., an
  RFC that references the right Volume but the wrong chapter content).

**Cons:**
- Manual review does not scale. With 20+ RFCs, 14 Volumes, and 10 ADRs, a single
  PR that changes a chapter number requires checking every cross-reference manually.
- Reviewer fatigue: checking cross-references is tedious and error-prone. Reviewers
  will start skipping checks.
- No orphan detection — unreferenced or unused documents accumulate silently.
- No template compliance enforcement — the RFC-0031 minimum content bar has no
  automated enforcement (the very problem RFC-0031 was created to solve).

**Why rejected:** Manual review is necessary but not sufficient. Automated tooling
  catches the mechanical errors (broken links, missing sections, orphaned files)
  so reviewers can focus on content quality.

### Alternative B: External Documentation Platform (Confluence, Notion, ReadTheDocs)
**Description:** Migrate the handbook to an external documentation platform that
  provides built-in link checking, search, and versioning.

**Pros:**
- External platforms provide mature features: full-text search, link checking,
  version history, access control, analytics.
- Better reading experience for consumers (responsive UI, table of contents
  navigation, dark mode).
- No custom tooling to build — leverage the platform's capabilities.

**Cons:**
- Violates Constitution Principle 9 (No Vendor Lock-in): the handbook is coupled
  to the documentation platform's API, storage format, and pricing.
- The handbook is currently version-controlled alongside the codebase (same repo).
  Migrating to an external platform separates documentation from code, making it
  harder to keep them in sync.
- External platforms may not support the specific document types (RFCs, ADRs,
  Volume specifications) with the same structure as the current markdown-based
  approach.
- ADR-0007's self-hosted fallback requirement would need a self-hosted documentation
  platform (e.g., Wiki.js, Outline), adding operational complexity.

**Why rejected:** The current markdown-in-git approach is version-controlled,
  diffable, and portable. External platforms add vendor lock-in and operational
  complexity. The tooling proposed in this RFC provides the needed consistency
  checks without leaving the git-based workflow.

### Alternative C: Custom npm Package with Three Linters (Recommended)
**Description:** As described in Proposed Decision. `@agentx/handbook-lint` with
  schema validation, cross-reference checking, and template compliance.

**Pros:**
- Fully automated — runs in CI on every PR.
- Custom-built for the handbook's specific structure (RFCs, ADRs, Volumes, Schemas).
- Incremental implementation — each linter can be built independently.
- No external dependencies beyond the existing TypeScript/markdown ecosystem.
- CI integration is straightforward — a single npm install and three CLI commands.
- Consistent with the platform's "build our own tools" philosophy (cf. contract
  test framework from ADR-0009).

**Cons:**
- 2–3 days of implementation effort for the initial version.
- The linters must be maintained as the handbook structure evolves (new document
  types, new template requirements).
- False positives are possible — the linter may flag a valid reference that uses
  a non-standard format.

## Consequences

- **Positive:** The handbook's internal consistency is automatically maintained.
  Broken cross-references, orphaned documents, and template violations are caught
  in CI before merge. Reviewers can focus on content quality instead of mechanical
  checks.
- **Negative:** 2–3 days of implementation effort. The linters require ongoing
  maintenance as the handbook evolves. False positives may require suppression
  mechanisms (e.g., `<!-- xref-ignore -->` comments).
- **Migration path (if applicable):** The linters are run against the existing
  handbook on first implementation. Existing violations (e.g., short ADRs that
  don't meet the new 20-line minimum) are either fixed or exempted via the
  grandfathering clause in RFC-0031.
- **Affected interfaces/contracts:** New npm package `@agentx/handbook-lint`. CI
  pipeline gains `handbook-lint` workflow. CONTRIBUTING.md updated with lint
  instructions for handbook contributors.
