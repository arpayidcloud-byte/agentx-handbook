# RFC-0041: Release Pipeline Execution Mechanism

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** Release engineering

---

## Context

The agentx platform is a monorepo with multiple packages (`packages/core`,
`packages/provider-sdk`, `packages/auth`, etc.) and multiple applications
(`apps/api`, `apps/cli`, `apps/dashboard`). Releasing the platform involves
building, testing, versioning, and publishing multiple coordinated artifacts.

Currently, no formal release pipeline exists. Releases are manual: a developer
bumps version numbers, runs tests locally, builds artifacts, and publishes to npm.
This manual process is error-prone (forgotten version bumps, missed test failures,
inconsistent changelogs) and does not scale as the number of packages grows.

RFC-0029 defines the API deprecation policy (what changes when), but does not
address the mechanical process of how releases are executed. This RFC defines the
automated release pipeline that enforces the deprecation policy and ensures
consistent, reliable releases.

## Proposed Decision

### Commit Convention: Conventional Commits

All commits to the `main` branch must follow the Conventional Commits specification:

```
type(scope): description

feat(core): add task graph retry mechanism
fix(provider-sdk): handle provider timeout correctly
docs(rfc): add RFC-0038 for task graph rollback
chore(deps): update TypeScript to 5.4
BREAKING CHANGE: rename CredentialResolver.resolve to resolveCredentials
```

The commit type determines the version bump:

| Commit Type | Version Bump | CHANGELOG Section |
|-------------|-------------|-------------------|
| `feat` | Minor | Features |
| `fix` | Patch | Bug Fixes |
| `docs` | None (unless it changes public API) | Documentation |
| `chore` | None | Miscellaneous |
| `refactor` | Patch (if behavior changes) | Code Refactoring |
| `perf` | Patch | Performance |
| `test` | None | Tests |
| `ci` | None | Continuous Integration |
| `BREAKING CHANGE` (any type) | Major | Breaking Changes |

### CHANGELOG Generation

Automated using `conventional-changelog`:

1. On every release, the tool reads commits since the last release tag.
2. Commits are grouped by type (feat, fix, BREAKING CHANGE) and scope (package name).
3. A CHANGELOG entry is generated for each package and the monorepo root.
4. The CHANGELOG includes references to RFCs/ADRs affected by the changes (extracted
   from commit message footers: `Refs: RFC-0029`).

Example generated CHANGELOG entry:

```markdown
## [0.5.0] - 2026-07-13

### Features
- **auth:** Add TokenAuthService with JWT-based authentication (Refs: RFC-0025)
- **rbac:** Add IRoleResolver interface for identity-to-RBAC handoff (Refs: RFC-0026)

### Breaking Changes
- **auth:** CredentialResolver constructor now requires SecretsBackend injection (Refs: RFC-0022)

### Bug Fixes
- **core:** Fix task graph retry skipping wrong nodes (Refs: RFC-0038)
```

### Automated SemVer Bump

A release script (`scripts/release.ts`) orchestrates the version bump:

1. Read all commits since the last release tag.
2. Determine the highest-impact commit type (BREAKING > feat > fix > other).
3. Bump the monorepo root version and all affected package versions.
4. Run `npm run build --if-present` across all packages to verify compilation.
5. Run the full test suite (`npm run test --workspaces`).
6. If any step fails, abort and report.
7. Generate CHANGELOG entries.
8. Commit the version bumps and CHANGELOG with message `chore(release): v0.5.0`.
9. Create a git tag `v0.5.0`.
10. Publish packages to npm (if configured).
11. Create a GitHub Release with the CHANGELOG as the release notes.

### Canary Deployment Strategy

For the `apps/api` and `apps/dashboard` applications:

1. The release pipeline deploys the new version to a canary environment (10% of
   traffic).
2. Automated health checks run for 15 minutes: error rate, latency p99, and
   key business metrics (task completion rate, agent success rate).
3. If health checks pass, the deployment proceeds to 50%, then 100%.
4. If health checks fail at any stage, the deployment is automatically rolled
   back to the previous version.

For the `apps/cli` application (distributed via npm):

1. No canary — CLI versions are published directly to npm.
2. A `latest` npm tag points to the stable release.
3. A `next` npm tag points to the release candidate for early adopters.
4. Users opt into `next` via `npm install @agentx/cli@next`.

### Rollback Automation

If a release causes issues post-deployment:

1. `scripts/rollback.ts <version>` reverts all packages to the specified version.
2. For API/dashboard: the deployment is rolled back to the previous Docker image
   or Kubernetes deployment.
3. For CLI: a deprecation warning is emitted by the new version pointing users to
   downgrade: `[WARNING] v0.5.1 has known issues. Please downgrade to v0.5.0:
   npm install @agentx/cli@0.5.0`.
4. The rollback is documented in the CHANGELOG and audit log.

### Release Approval Gate

Before a release is published, the following gates must pass:

| Gate | Check | Enforcement |
|------|-------|-------------|
| Test suite | All tests pass | Automated |
| Deprecation check | No deprecated APIs removed before their deprecation window (RFC-0029) | Automated |
| CHANGELOG completeness | Every commit since last release appears in the CHANGELOG | Automated |
| Handbook lint | All handbook files pass `@agentx/handbook-lint` (RFC-0040) | Automated |
| Manual approval | A maintainer approves the release in CI (GitHub Environment approval) | Manual |

## Alternatives Considered

### Alternative A: Fully Manual Release Process
**Description:** A developer runs version bump commands, builds, tests, and publishes
  manually. No automation.

**Pros:**
- No CI/CD pipeline to build or maintain.
- Full human control over every step — no surprises.
- Simple for a small team with infrequent releases.

**Cons:**
- Error-prone: forgotten version bumps, missed test failures, inconsistent
  changelogs.
- Does not scale: as the monorepo grows, manual releases become untenable.
- No canary deployment or automated rollback.
- Deprecation policy (RFC-0029) cannot be enforced mechanically.
- Releases are slow (hours of manual work) and block other development.

**Why rejected:** Manual releases are acceptable for v0.1 (single package, single
  developer) but fail as the platform grows. Automating from the start avoids the
  painful transition from manual to automated releases later.

### Alternative B: Release Please (Google's Automated Release Tool)
**Description:** Use Google's [Release Please](https://github.com/googleapis/release-please)
  tool for automated versioning, CHANGELOG generation, and release PRs.

**Pros:**
- Mature, well-maintained tool used by many Google open-source projects.
- Handles monorepo release coordination out of the box.
- Creates release PRs that can be reviewed before merging.
- Integrates with GitHub Actions natively.

**Cons:**
- Vendor lock-in to Google's toolchain and conventions. The release PR format,
  CHANGELOG style, and versioning logic are defined by Release Please.
- Custom hooks (e.g., canary deployment, deprecation checks) require workarounds
  or plugins.
- The tool's opinionated approach may conflict with the platform's specific
  requirements (e.g., RFC-aware CHANGELOGs, multi-application deployment).
- Less control over the release pipeline — the tool abstracts away details that
  may matter for the platform's unique requirements.

**Why rejected:** While Release Please is an excellent tool, the platform's release
  requirements (canary deployment, deprecation enforcement, handbook-aware
  CHANGELOGs) are specific enough that a custom pipeline provides more control
  with comparable effort.

### Alternative C: Custom Pipeline with Conventional Commits (Recommended)
**Description:** As described in Proposed Decision. Custom release script,
  conventional-changelog for CHANGELOG generation, automated SemVer bump,
  canary deployment, and rollback automation.

**Pros:**
- Full control over every step of the release process.
- Conventional Commits is an open standard — no vendor lock-in.
- The pipeline enforces RFC-0029's deprecation policy mechanically.
- Canary deployment and automated rollback provide production safety.
- The release approval gate combines automated checks with manual oversight.
- The pipeline is testable — the release script itself has unit tests.

**Cons:**
- Custom pipeline requires implementation and maintenance (estimated: 3–5 days).
- The pipeline must be kept in sync with the monorepo structure as packages are
  added or reorganized.
- Canary deployment requires infrastructure (load balancer, health checks) that
  may not exist in all deployment environments.

## Consequences

- **Positive:** Releases are consistent, automated, and enforce the deprecation
  policy. Canary deployment reduces the blast radius of faulty releases. The
  CHANGELOG is always complete and accurate. Rollback is a single command.
- **Negative:** 3–5 days of implementation effort for the release pipeline. The
  pipeline requires maintenance as the monorepo evolves. Canary deployment
  requires infrastructure that may not be available for all deployment targets.
- **Migration path (if applicable):** The pipeline is built once and used for all
  future releases. No migration from an existing pipeline.
- **Affected interfaces/contracts:** New `scripts/release.ts` and `scripts/rollback.ts`.
  New `conventional-changelog` configuration. New GitHub Actions workflow for
  releases. New `.release-rc.yml` configuration for release settings.
