# RFC-0034: Notification Channel for Approval Gates Beyond CLI Prompt

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 9

---

## Context

The agentx platform has approval gates: certain operations (destructive file writes,
shell commands, configuration changes) require explicit operator approval before
execution (ADR-0004, ADR-0005). In v0.1, the only approval mechanism is a CLI prompt:
the operator must be actively watching the terminal and type "yes" to approve.

This CLI-only model works for a single developer running agentx interactively. It
breaks down in several scenarios:

1. **Async workflows:** An agent runs a long task (e.g., a multi-file refactoring)
   overnight. A destructive operation occurs at 3 AM when no one is watching. The
   CLI prompt hangs indefinitely, and the task stalls.

2. **Team environments:** Multiple operators share a project. An approval is needed,
   but the operator who started the task is not available. Another team member should
   be able to approve.

3. **CI/CD pipelines:** An agent runs as part of a CI pipeline. There is no
   interactive terminal for approval. The pipeline needs a way to request and receive
   approval.

4. **Mobile/remote operators:** An operator wants to approve actions from their phone
   while away from their desk.

This RFC defines a phased expansion of the approval notification and acknowledgment
mechanism beyond the CLI prompt.

## Proposed Decision

### Phase 1: CLI Prompt (v0.1 — Current)

No changes. The CLI prompt remains the sole approval mechanism. This phase documents
the existing behavior for reference:

- Approval requests are written to stdout with a clear description of the pending
  action.
- The operator types `yes`/`y` to approve or `no`/`n` to reject.
- No timeout — the prompt waits indefinitely.
- No audit trail beyond the in-terminal log.

### Phase 2: Webhook Notification (v0.5)

Approval requests are published to an event bus (Volume 5) and delivered via webhook
to a configurable URL:

```typescript
interface ApprovalRequest {
  id: string;
  action: string;           // e.g., "fs.write", "shell.exec"
  description: string;      // human-readable description
  riskLevel: 'low' | 'medium' | 'high';  // based on ADR-0005 classification
  agentId: string;
  taskId: string;
  requestedAt: string;      // ISO 8601
  expiresAt: string;        // ISO 8601
  approveUrl: string;       // URL to approve (POST with approval token)
  rejectUrl: string;        // URL to reject
}
```

The webhook payload is sent to the operator's configured notification URL. The
operator approves or rejects by making a POST request to the provided URL. The
platform validates the approval token and resumes or aborts the pending operation.

**Configuration:**
```yaml
approval:
  notification:
    type: webhook
    url: https://hooks.slack.com/services/xxx
    timeoutHours: 24
    autoRejectOnTimeout: false
```

### Phase 3: Multi-Channel Notification (v1.0+)

Additional notification channels are supported via a pluggable `NotificationChannel`
interface:

```typescript
interface NotificationChannel {
  id: string;
  send(request: ApprovalRequest): Promise<void>;
}
```

Shipped channels:
- **Webhook:** Generic HTTP POST (Phase 2).
- **Email:** SMTP-based notification with approve/reject links.
- **Slack/Discord:** Chat platform integrations via webhook.

**Operator preference configuration:**
```yaml
approval:
  notification:
    channels:
      - type: webhook
        url: https://hooks.slack.com/services/xxx
      - type: email
        address: ops@company.com
    timeoutHours: 24
    autoRejectOnTimeout: false
```

When multiple channels are configured, all channels receive the notification.
Approval from any single channel is sufficient (first-acknowledgment-wins).

### Acknowledgment Timeout & Auto-Reject

- **Default timeout:** 24 hours. Configurable per-organization via Volume 10
  configuration.
- **Behavior on timeout:** By default, the request **remains pending** (safe default —
  no data loss from timeout). The operator is re-notified every 24 hours.
- **Auto-reject option:** If `autoRejectOnTimeout: true`, the request is automatically
  rejected on timeout, and the pending operation is aborted. This is useful for CI/CD
  pipelines where indefinite hanging is worse than aborting.
- **Escalation (v1.5+):** If no one acknowledges within the timeout, the request is
  escalated to a configured escalation channel (e.g., a Slack channel with @here
  mention).

### Audit Trail

All approval requests and responses are recorded in the audit log (Volume 6, RFC-0024):
- Request: `approval.requested` event with full `ApprovalRequest` payload.
- Approval: `approval.approved` event with `approverId`, `channel`, `responseTimeMs`.
- Rejection: `approval.rejected` event with `rejectorId`, `channel`, `reason`.
- Timeout: `approval.timed_out` event.

## Alternatives Considered

### Alternative A: CLI Prompt Only (Forever)
**Description:** Never expand beyond the CLI prompt. All approvals require an active
  terminal session.

**Pros:**
- Simplest implementation. No webhook, no email, no notification infrastructure.
- Guarantees that the operator is actively engaged (they're looking at the terminal).
- No async complexity — the prompt is a synchronous blocking call.

**Cons:**
- Blocks async and long-running workflows. An overnight task that hits an approval
  gate at 3 AM stalls until morning.
- No team support — only the operator at the terminal can approve.
- No CI/CD integration — pipelines cannot include approval-gated agent tasks.
- The operator must remain at their desk for the entire duration of an agent session
  that might involve approvals.

**Why rejected:** The CLI-only model is a v0.1 constraint, not a long-term design.
  The platform must support async, team, and CI/CD workflows to be viable beyond
  a single developer's local machine.

### Alternative B: Pre-Approved Action Policies (No Runtime Approval)
**Description:** Operators define policies that pre-approve certain actions. For
  example, "all fs.write actions to the /src directory are pre-approved." Runtime
  prompts are never shown; the policy engine decides.

**Pros:**
- Eliminates the need for real-time operator attention entirely.
- Fast — no waiting for approval, actions proceed immediately.
- Deterministic — the same action always gets the same approval decision.

**Cons:**
- Pre-approval policies are a form of blanket permission. A broad policy
  ("approve all fs.write") negates the security benefit of approval gates.
- Policy specification is complex — operators must anticipate all actions and
  classify them correctly. Mis-classification has the same risks as ADR-0004's
  "prompt-only" approach.
- No operator awareness — when pre-approved actions execute, the operator has
  no visibility into what happened.
- Increases the attack surface: prompt injection that triggers a pre-approved
  action bypasses the operator entirely.

**Why rejected:** Pre-approval policies are complementary to runtime approval, not
  a replacement. They may be added as an optimization in v1.5+ (e.g., "pre-approve
  fs.write to test files during test agent execution"), but they cannot replace the
  requirement for operator awareness.

### Alternative C: Phased Multi-Channel Notification (Recommended)
**Description:** As described in Proposed Decision. CLI prompt for v0.1, webhook for
  v0.5, multi-channel (webhook + email + chat) for v1.0, with configurable timeout
  and auto-reject.

**Pros:**
- Each phase adds capability when it is needed, maintaining the "small stable core"
  principle for each release.
- The webhook approach (v0.5) is the simplest possible async mechanism — a single
  HTTP POST with no library dependencies.
- Multi-channel (v1.0) covers the most common operator workflows (desktop, mobile,
  chat, email).
- The timeout and auto-reject options give operators control over the trade-off
  between safety (wait forever) and progress (auto-reject and move on).
- Audit trail integration ensures all approval decisions are recorded.

**Cons:**
- Three phases of implementation across three releases.
- Webhook delivery is not guaranteed (network failures, downtime of the receiving
  service). Mitigated by retry logic (3 retries with exponential backoff).
- The pluggable channel interface adds a small abstraction layer.
- Email notifications require SMTP configuration, which may be complex for some
  operators.

## Consequences

- **Positive:** The platform supports async, team, and CI/CD workflows. Operators
  can approve actions from any device. The audit trail provides full visibility into
  who approved what and when.
- **Negative:** Implementation spans three releases. Webhook reliability requires
  retry logic. Multi-channel support adds configuration complexity.
- **Migration path (if applicable):** v0.1 CLI prompts continue to work unchanged.
  v0.5 adds webhook configuration alongside CLI. The CLI prompt remains available
  in all phases as a fallback.
- **Affected interfaces/contracts:** New `NotificationChannel` interface in Volume 9.
  New `ApprovalService` that manages request lifecycle. New configuration schema
  for notification channels. New audit log events for approval requests/responses.
