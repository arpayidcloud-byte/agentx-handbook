# RFC-0035: Compliance Framework Mapping (SOC2 Baseline)

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 10

---

## Context

Enterprise customers evaluating the agentx platform will require evidence of compliance
with industry-standard security and privacy frameworks. SOC2 (System and Organization
Controls 2) is the most commonly requested baseline, covering Trust Service Criteria
(TSC): Security, Availability, Processing Integrity, Confidentiality, and Privacy.

While full SOC2 certification is a v1.0+ milestone (it requires auditor engagement,
evidence collection over a period of months, and organizational controls beyond
software), the platform's architecture must be designed with SOC2 alignment from the
start. Retrofitting compliance controls into an existing architecture is far more
expensive than designing them in from the beginning.

This RFC maps the agentx platform's existing and planned capabilities to SOC2 TSC
categories, identifies gaps, and defines a gating strategy: which compliance
activities are appropriate for v0.1, which are deferred.

**Important:** This RFC is a planning document, not a certification. No SOC2
audit occurs before v1.0.

## Proposed Decision

### Data Classification

All data in the agentx platform is classified into four tiers:

| Tier | Definition | Examples | Encryption at Rest | Encryption in Transit |
|------|-----------|----------|-------------------|----------------------|
| **Public** | Intended for public access | Open-source code, documentation, marketplace listings | Not required | TLS (standard) |
| **Internal** | For authorized users only | Agent configurations, workflow definitions, plugin manifests | AES-256 (via SecretsBackend) | TLS |
| **Confidential** | Sensitive business data | Audit logs, RBAC configurations, organization settings | AES-256-GCM | TLS + mutual auth (v1.0) |
| **Restricted** | Highly sensitive, regulated | Secrets (API keys, tokens), credentials, encryption keys | AES-256-GCM with HSM (v1.0) | TLS + mTLS |

### Gating Strategy

| Activity | Target Release | Prerequisite |
|----------|---------------|--------------|
| Data classification document | v0.1 | This RFC |
| Credential encryption (file backend) | v0.1 | RFC-0022 |
| Audit log immutability | v0.1 | RFC-0024 |
| Trust boundary documentation | v0.1 | RFC-0021 |
| Multi-user authentication | v0.5 | RFC-0025 |
| SSO integration | v1.0 | RFC-0025 |
| HSM-backed key management | v1.0 | RFC-0022 |
| WORM audit storage | v1.0 | RFC-0024 |
| Formal SOC2 Type I audit | v1.0 | All v1.0 controls implemented |
| SOC2 Type II audit (ongoing) | v1.5+ | 6+ months of Type I evidence |

## Alternatives Considered

### Alternative A: No Compliance Planning Until Enterprise Demand
**Description:** Defer all compliance-related work until an enterprise customer
  explicitly requests SOC2 evidence. At that point, implement the required controls.

**Pros:**
- No compliance-related work in v0.1 or v0.5 — maximum focus on functional features.
- Avoids premature investment in controls that may not be needed if enterprise
  adoption doesn't materialize.
- Compliance requirements can be tailored to the specific customer's needs.

**Cons:**
- Retrofitting compliance controls into an existing architecture is 3–10x more
  expensive than designing them in from the start.
- Some controls (e.g., audit log immutability, credential encryption) require
  architectural decisions that are difficult to change later.
- Enterprise customers evaluate platforms based on security architecture, not just
  audit reports. A platform with no compliance planning signals immaturity.
- Without a data classification framework, developers may inadvertently store
  restricted data in inappropriate locations.

**Why rejected:** Compliance-conscious architecture is a form of "Architecture First"
  (Constitution Principle 1). The controls described in this RFC (encryption, audit
  logging, RBAC) are valuable even without a formal audit — they make the platform
  more secure for all users.

### Alternative B: Full SOC2 Certification for v0.1
**Description:** Pursue SOC2 Type I certification as part of the v0.1 release.
  Implement all required controls, engage an auditor, and obtain certification
  before GA.

**Pros:**
- Maximum credibility — the platform ships with a compliance certification.
- Forces the team to implement all security controls early.
- Removes compliance as a blocker for enterprise sales from day one.

**Cons:**
- SOC2 Type I requires at least 3–6 months of evidence collection. This delays
  v0.1 by months.
- Auditor engagement costs $20,000–$50,000+ — a significant expense for a pre-
  revenue platform.
- Some controls (SSO, HSM, WORM storage) are architecturally premature for v0.1's
  single-user, self-hosted deployment model.
- The compliance overhead (evidence collection, control testing, auditor meetings)
  distracts from feature delivery.

**Why rejected:** Full certification for v0.1 is disproportionate. The platform
  should build compliance-aligned architecture (which this RFC does) without the
  certification overhead until enterprise demand justifies it.

### Alternative C: Compliance-Aligned Architecture with Phased Certification (Recommended)
**Description:** As described in Proposed Decision. Map controls to TSC, implement
  v0.1-appropriate controls, defer certification to v1.0+.

**Pros:**
- The architecture is compliance-aligned from the start — no costly retrofits.
- v0.1 implements the controls that are appropriate for its scope (encryption,
  audit logging, RBAC).
- Certification is deferred until the platform has multi-user support (a
  prerequisite for meaningful SOC2 evaluation).
- The control mapping provides a clear roadmap for enterprise readiness.
- Data classification guides all future development decisions.

**Cons:**
- The control mapping document requires maintenance as the platform evolves.
- Some v0.1 controls (audit log immutability) add implementation effort that
  wouldn't be needed without compliance planning.
- The gap analysis may create pressure to implement v1.0 controls earlier than
  planned.

## Consequences

- **Positive:** The platform has a documented compliance roadmap. Architecture
  decisions are informed by SOC2 requirements. Enterprise evaluation teams have a
  concrete control mapping to review. Data classification provides a framework for
  all future data handling decisions.
- **Negative:** The compliance mapping requires maintenance. Some v0.1 features
  (audit immutability, credential encryption) add scope. The gap analysis may
  create anxiety about the distance to full certification.
- **Migration path (if applicable):** No migration — this is a planning document.
  As each control is implemented, the mapping is updated from "Planned" to
  "Implemented" with a reference to the implementing RFC/ADR.
- **Affected interfaces/contracts:** Volume 10 gains a new chapter on compliance
  mapping. Data classification is referenced across all Volumes that handle data
  (Volumes 3, 4, 5, 6, 7, 8).
