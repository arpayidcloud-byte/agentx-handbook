# RFC-0024: Audit Log Immutability Mechanism

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 6, 10

---

## Context

Volume 6 (Data Layer) defines the audit log schema for tracking all security-relevant
events: tool executions, permission checks, credential access, configuration changes, and
operator approvals. These records are essential for post-incident investigation, compliance
audits (RFC-0035 SOC2 mapping), and operational debugging.

However, the current Volume 6 specification does not address **immutability** — the
guarantee that once an audit record is written, it cannot be altered or deleted. Without
immutability, a compromised operator account (or a bug) could tamper with audit records to
cover unauthorized activity. This defeats the purpose of audit logging and renders
compliance claims unverifiable.

Constitution Principle 7 (Security & Isolation) requires that security-relevant data be
protected against tampering. The audit log is the most security-relevant data in the
system: it is the authoritative record of "what happened and who authorized it."

ADR-0014 establishes the v0.1 enforcement mechanism: PostgreSQL triggers and role-based
permissions that prevent UPDATE and DELETE at the database layer. This RFC builds on that
foundation by adding chained hashes for tamper evidence and defining the enterprise migration
path to S3 WORM storage.

This RFC designs a tamper-evident audit log mechanism suitable for v0.1 (Postgres-backed)
with a migration path to enterprise-grade append-only storage.

## Proposed Decision

Implement a two-layer tamper-evidence mechanism: chained hashes at the record level, and
database-level append-only enforcement via PostgreSQL triggers.

### Layer 1: Chained Hashes (Tamper Evidence)

Every audit log record includes a `record_hash` field computed as:

```
SHA-256(previous_record_hash + serialized_record_content + record_timestamp)
```

For the first record, `previous_record_hash` is the string `"GENESIS"`.

```sql
ALTER TABLE audit_log ADD COLUMN record_hash TEXT NOT NULL;
ALTER TABLE audit_log ADD COLUMN previous_hash TEXT NOT NULL;
```

The hash chain provides tamper **evidence** (not tamper **prevention**): any modification
or deletion of a record breaks the chain, and a verification function can detect this
in O(n) time.

A CLI command `agentx audit verify` walks the chain and reports:
- `OK` — all hashes valid.
- `TAMPERED at record_id={id}` — hash mismatch detected.

This verification runs automatically as part of the quarterly DR drill (RFC-0030).

### Layer 2: PostgreSQL Append-Only Trigger (per ADR-0014)

ADR-0014 mandates database-level immutability enforcement. The trigger and role-based
permission design defined there is adopted wholesale:

A database trigger prevents UPDATE and DELETE operations on the `audit_log` table:

```sql
CREATE OR REPLACE FUNCTION audit_log_append_only()
RETURNS TRIGGER AS $$
BEGIN
    RAISE EXCEPTION 'audit_log table is append-only: % operations are not permitted', TG_OP;
    RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER audit_log_append_only_trigger
    BEFORE UPDATE OR DELETE ON audit_log
    FOR EACH ROW EXECUTE FUNCTION audit_log_append_only();
```

This trigger is the **enforcement** layer. Even a database administrator with direct SQL
access must `DROP TRIGGER` before modifying records — an action that itself is detectable
(by the trigger's absence) and should be logged by PostgreSQL's own audit extension
(`pgaudit`).

### Enterprise Migration Path (v1.0+)

For enterprise deployments requiring stronger guarantees:

1. **Append-only S3/GCS bucket:** New audit records are written to an append-only cloud
   storage bucket with Object Lock (WORM — Write Once Read Many) enabled. PostgreSQL
   retains a recent window (e.g., 90 days) for querying; older records are archived
   to S3 with a 7-year retention policy.

2. **Separate audit database:** Audit records are written to a separate PostgreSQL
   instance with no UPDATE/DELETE grants for the application role. The application role
   has INSERT-only access. This limits the blast radius of application-level compromise.

3. **Cryptographic signing:** Each record (or batch) is signed with an HSM-backed key.
   Verification requires the public key, providing non-repudiation even if the storage
   layer is fully compromised.

The v0.1 design (trigger + chained hashes) is intentionally portable and requires no
external services beyond PostgreSQL itself.

## Alternatives Considered

### Alternative A: Application-Level Enforcement Only
**Description:** The application code never calls `UPDATE` or `DELETE` on the audit_log
table. No database trigger, no chained hashes. Trust the code.

**Pros:**
- Zero database overhead — no trigger, no extra columns, no hash computation.
- Simplest implementation.
- No schema changes required.

**Cons:**
- A bug, a future refactoring, or a direct database access (e.g., `psql` session by an
  operator) can silently modify or delete records.
- No tamper evidence — even if records are modified, there is no way to detect it.
- Insufficient for any compliance framework (SOC2, ISO 27001) that requires audit log
  integrity controls.
- Violates Constitution Principle 7: "security-relevant data protected against tampering."

**Why rejected:** Application-level enforcement is necessary but not sufficient. It is
the software equivalent of a sign saying "please do not enter" — it relies on everyone
reading the sign and choosing to comply. A database trigger is the equivalent of a locked
door.

### Alternative B: Write-Once S3 Storage from Day One
**Description:** Write all audit records directly to an S3 bucket with Object Lock
enabled. No PostgreSQL audit table at all. Querying is done via S3 Select or Athena.

**Pros:**
- Strongest possible immutability guarantee — S3 Object Lock is enforced by the storage
  layer, not the application.
- Naturally supports long retention periods (up to indefinite).
- Separates audit storage from operational database, reducing load on Postgres.
- Already standard practice in enterprise environments.

**Cons:**
- Adds S3 as a hard dependency for audit logging — violates ADR-0007's self-hosted
  fallback requirement (MinIO is the fallback, but it must be configured and tested).
- Query performance for recent records is worse than PostgreSQL (network latency, no
  indexes, no joins).
- Introduces a second storage system to operate, monitor, and back up.
- Overkill for v0.1, which targets a single operator on a single machine.
- Complicates local development (requires MinIO or S3-compatible local storage).

**Why rejected:** The operational complexity and ADR-0007 friction make this premature for
v0.1. It remains the correct target for v1.0+ enterprise deployments, and the migration
path is explicitly documented in the recommended approach.

### Alternative C: PostgreSQL Trigger + Chained Hashes (Recommended)
**Description:** As described in Proposed Decision. Database trigger for enforcement,
chained hashes for tamper evidence, with documented enterprise migration path.

**Pros:**
- Enforced at the database layer — cannot be bypassed by application code.
- Tamper evidence via chained hashes provides detectability even if the trigger is
  somehow circumvented (e.g., trigger dropped by a DBA).
- Zero external dependencies — uses only PostgreSQL features available in every
  supported version (PostgreSQL 14+).
- Portable to any PostgreSQL-compatible database (per Constitution Principle 9).
- Chained hashes add only ~64 bytes per record — negligible storage overhead.
- `agentx audit verify` provides a simple, auditable integrity check.

**Cons:**
- Trigger adds ~0.1ms per INSERT (hash computation + trigger invocation) — negligible.
- A database superuser can DROP the trigger — this is an accepted limitation documented
  as a "trust boundary" issue (the DBA is in the operator-trusted zone per RFC-0021).
- Chained hash verification is O(n) — acceptable for the expected record volume
  (thousands, not millions, at v0.1 scale).

## Consequences

- **Positive:** Audit log integrity is enforceable and verifiable. The platform gains a
  concrete, demonstrable security control. The enterprise migration path is clear and
  documented. Compliance teams have an artifact to point to in audits.
- **Negative:** Schema migration required (two new columns + trigger). The `record_hash`
  computation adds a small amount of CPU and storage overhead per record. The trigger
  may surprise developers who are used to freely updating records during debugging.
- **Migration path (if applicable):** Existing audit_log records must be backfilled with
  `record_hash` and `previous_hash` values. A one-time migration script computes the
  chain retrospectively. If any existing records were modified (unlikely in v0.1), the
  chain will break at that point — the migration script logs a warning and continues.
- **Affected interfaces/contracts:** `audit_log` table schema (Volume 6, Ch. 3). New
  `AuditService.verifyChain()` method. New CLI command `agentx audit verify`. New
  migration file: `add_audit_log_hash_chain.sql`.
