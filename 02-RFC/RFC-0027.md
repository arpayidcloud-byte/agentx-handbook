# RFC-0027: Plugin Sandboxing & Trust Model

**Status:** Accepted
**Author:** Architecture Review Board
**Date:** 2026-07-13
**Supersedes:** —
**Affected Volumes:** 8

---

## Context

Volume 8 (Plugin Platform) enables third-party extensions to the agentx runtime: custom
tools, agent roles, workflow steps, and provider adapters. Plugins run inside the agentx
process and have access to the same APIs, filesystem, and network as the core platform.

This access is dangerous. A malicious or buggy plugin could:
- Read secrets from the `SecretsBackend` (RFC-0022).
- Execute arbitrary shell commands via the Tool SDK (Volume 7).
- Modify or delete files outside the project workspace.
- Exfiltrate data by making network requests to attacker-controlled servers.
- Crash the agentx process by consuming excessive memory or CPU.

Constitution Principle 7 (Security & Isolation) requires that plugins be isolated from
the core platform. RFC-0021's threat model methodology (STRIDE analysis, trust boundaries)
applies equally to plugins, but the specific isolation mechanism must be chosen.

This RFC defines the plugin trust model, the sandboxing approach, and the plugin
permission manifest that controls what a plugin can access.

## Proposed Decision

### Trust Tiers

Plugins are classified into three trust tiers, each with a different isolation level:

| Tier | Source | Isolation | Example |
|------|--------|-----------|---------|
| **Core** | Platform repository | None (part of the core process) | Built-in tools (shell, fs, git) |
| **Verified** | Signed, reviewed by platform team | Process-level (child_process) | Official marketplace plugins |
| **Unverified** | Unsigned, any source | Process-level + restricted permissions | Community plugins, local dev plugins |

### Sandboxing: Process-Level Isolation (child_process)

The recommended isolation mechanism is **process-level isolation** using Node.js
`child_process.fork()` with restricted capabilities:

```typescript
interface PluginSandboxConfig {
  /** Whether the plugin can access the filesystem (and which paths). */
  filesystem: {
    allowed: boolean;
    allowedPaths?: string[];   // e.g., ['/home/user/project']
    allowNetwork?: boolean;    // default: false
  };

  /** Whether the plugin can spawn child processes. */
  childProcess: boolean;       // default: false

  /** Memory limit in MB. */
  memoryLimitMb: number;       // default: 128

  /** CPU time limit in seconds (per invocation). */
  cpuTimeLimitSec: number;     // default: 30

  /** Network access: allowed domains (empty = no network). */
  networkAllowlist?: string[];
}
```

The plugin runs in a forked child process with:
- `--max-old-space-size=128` (memory limit).
- A custom `require` hook that intercepts and blocks imports of sensitive modules
  (`child_process`, `fs`, `net`, `dns`) unless explicitly allowed in the manifest.
- Communication with the host process via `process.send()` / `process.on('message')`
  — the plugin receives a message port, not the full IPC channel.

### Plugin Permission Manifest

Every plugin package must include a `plugin.json` manifest:

```json
{
  "name": "@agentx/plugin-jira",
  "version": "1.2.0",
  "tier": "verified",
  "permissions": {
    "filesystem": { "allowed": true, "allowedPaths": ["${PROJECT_DIR}"] },
    "network": { "allowed": true, "allowlist": ["api.atlassian.com"] },
    "childProcess": false,
    "secrets": { "allowed": true, "keys": ["jira.api_token"] }
  },
  "signature": "sha256:abc123..."
}
```

The platform loads this manifest before executing the plugin. If the requested
permissions exceed the tier's maximum, the plugin is rejected at load time (not at
runtime — fail early per ADR-0004).

### Plugin Signing & Verification

Verified-tier plugins are signed with an asymmetric key pair:
- The platform team holds the private signing key.
- The public verification key is embedded in the agentx binary.
- Signature covers the entire plugin package (all files, not just the manifest).
- Verification occurs at install time (`agentx plugin install`). A tampered plugin
  is rejected with a `SignatureVerificationError`.

### Review Process for Marketplace Inclusion

To be listed in the official marketplace (v2.0+):
1. Plugin author submits the plugin package + source code.
2. Platform team conducts a security review (2–5 business days).
3. Reviewer checks: manifest accuracy, no hidden permissions, no obfuscated code,
   no dependency on known-vulnerable packages.
4. Two reviewers must approve (the same two-reviewer rule as RFC-0028).
5. On approval, the plugin is signed with the platform key and published.

## Alternatives Considered

### Alternative A: No Sandboxing (Trust All Plugins)
**Description:** Plugins run in the same process with full access to all platform APIs.
  No permission manifest, no isolation, no signing.

**Pros:**
- Simplest implementation. No IPC overhead, no custom require hook, no manifest
  validation.
- Plugins can access any platform API directly — maximum flexibility.
- No review process needed — anyone can publish, anyone can install.

**Cons:**
- Directly contradicts Constitution Principle 7 (Security & Isolation).
- A single malicious plugin compromises the entire platform: secrets, filesystem,
  network, process.
- No basis for user trust — installing a plugin is equivalent to running arbitrary
  code with full privileges.
- Makes marketplace viability impossible — no enterprise would allow unvetted plugins.

**Why rejected:** This is the "trust everything" anti-pattern. It is only viable for
  a personal project with no multi-user or enterprise aspirations. The agentx
  platform explicitly targets enterprise use (Volume 10), making this unacceptable.

### Alternative B: WebAssembly (WASM) Sandbox
**Description:** Plugins are compiled to WebAssembly and run in a WASM runtime with
  capability-based security. The WASM sandbox provides memory isolation, no direct
  filesystem/network access unless explicitly granted via WASI.

**Pros:**
- Strongest isolation guarantee — WASM memory is fully sandboxed by the runtime.
- Capability-based security is a well-studied model (WASI).
- Language-agnostic — plugins can be written in Rust, Go, C++, AssemblyScript.
- No risk of a plugin accessing Node.js internals (it's a different runtime).

**Cons:**
- Requires plugins to be compiled to WASM — TypeScript/JavaScript plugins (the
  platform's primary language) cannot run natively. They would need a JS-to-WASM
  compiler, which is immature and adds build complexity.
- WASI support for network and filesystem is still evolving — many operations are
  not yet standardized.
- Debugging WASM plugins is significantly harder than debugging Node.js.
- The platform's TypeScript API surface cannot be consumed from WASM without
  compilation bindings — massive DX friction.
- Performance overhead of bridging between Node.js and WASM on every call.

**Why rejected:** WASM sandboxing is architecturally elegant but practically premature
  for a TypeScript-first platform. It remains a v3.0+ consideration when the plugin
  ecosystem is mature enough to justify the build and debug complexity.

### Alternative C: Container-Level Isolation (Docker)
**Description:** Each plugin runs in its own Docker container. The platform communicates
  with the plugin via HTTP or gRPC over a Docker network.

**Pros:**
- Very strong isolation — the plugin has its own kernel namespace, filesystem, and
  network stack.
- Standard infrastructure — any DevOps team already knows Docker.
- Resource limits are well-supported (cgroups, memory limits, CPU quotas).

**Cons:**
- Requires Docker installed and running — adds a hard infrastructure dependency
  that violates the v0.1 "works on a laptop with Node.js" bar.
- Startup latency: spinning up a container takes 1–5 seconds vs. 50ms for a
  child_process fork. Unacceptable for plugins invoked on every agent turn.
- Resource overhead: each container has its own Node.js runtime (~50MB RAM minimum).
  Running 10 plugins means ~500MB just for plugin runtimes.
- Network complexity: inter-plugin and plugin-to-platform communication requires
  service discovery or hardcoded ports.
- CI/CD complexity: testing containerized plugins requires Docker-in-Docker or
  Docker socket mounting.

**Why rejected:** Container isolation is appropriate for long-running, heavy plugins
  (e.g., a full database proxy) but is disproportionate overhead for the common case
  of a lightweight tool plugin. Process-level isolation achieves sufficient isolation
  for v0.1–v1.0.

### Alternative D: Process-Level Isolation with Permission Manifest (Recommended)
**Description:** As described in Proposed Decision. child_process fork, custom require
  hook, permission manifest, plugin signing, marketplace review process.

**Pros:**
- Good isolation without external dependencies (no Docker, no WASM runtime).
- TypeScript plugins work natively — same language, same toolchain, same debug
  experience.
- Permission manifest provides fine-grained, declarative access control.
- Plugin signing provides supply-chain integrity for verified plugins.
- Marketplace review process establishes trust without requiring technical
  enforcement for every possible attack vector.
- Low overhead: fork takes ~50ms, IPC adds ~1ms per call.

**Cons:**
- Process-level isolation is not as strong as WASM or container isolation. A
  determined attacker with a Node.js zero-day could potentially escape the sandbox.
- The custom `require` hook is a maintenance burden — new Node.js APIs must be
  evaluated for inclusion in the blocklist.
- Signing key management is a responsibility: key compromise means an attacker
  can sign malicious plugins as verified.

## Consequences

- **Positive:** Plugins are isolated from the core platform with a clear, auditable
  permission model. The trust tier system allows the platform to evolve from
  permissive (v0.1, core-tier only) to restrictive (v2.0, verified marketplace) as
  the ecosystem matures.
- **Negative:** Plugin authors must learn the permission manifest format and test
  their plugins within the sandbox. The custom require hook adds maintenance burden.
  The signing key is a single point of failure (mitigated by key rotation procedures).
- **Migration path (if applicable):** v0.1 ships with core-tier plugins only (no
  third-party plugin support). v0.5 introduces the plugin loading infrastructure and
  verified-tier plugins. v2.0 introduces unverified-tier support and the marketplace.
- **Affected interfaces/contracts:** New `PluginSandboxConfig` interface. New
  `plugin.json` manifest schema. New `PluginService` in Volume 8 that manages plugin
  lifecycle (load, sandbox, execute, unload). Modified CI pipeline to validate
  manifest compliance.
